# External-partner listings are skipped with a debug log, not enriched

When a **Parser**'s `enrich` discovers that the source's detail page is an outbound redirect to a third-party job board rather than an embeddable on-domain body (no iframe, no `raw-url` payload, just an `<a>` pointing off-domain), the parser yields a new `ExternalRedirect(stub, outbound_url)` payload to the orchestrator instead of a `Position` or a `ParserError`. The orchestrator marks the stub in the **Deduplication Store** with `status: "external_redirect"`, bumps a dedicated `RunSummary.external_redirects` counter, and does not WARN. The parser additionally appends one line per detection to a per-component debug log file at `<logs_dir>/<component_id>.log` via `parser_log.record`, so the population of outbound targets can be analyzed later as input to a follow-up parser.

## Why

- **External listings are a distinct outcome, not a failure.** Today `enrich` raises `ParserError` for any unparseable wrapper, the orchestrator marks the stub `enrich_failed`, and the run logs a WARNING. For the genuinely-broken case that semantic is correct. For external-partner listings — well-formed pages that just don't carry an embeddable body — the WARNING is noise and the `enrich_failed` bucket conflates "the source changed shape and we need to fix the parser" with "this is a wrapper around a third-party post." Once the buckets are conflated, the metric ("how many external listings did we drop this month?") is only recoverable by log-scraping.
- **The information we want to keep is the outbound target.** External listings are worthless in v1 because the wrapper page has no description text — only title, employer, and location, which the **Position Stub** already carries from `discover`. There is no useful `Position` to build. But the *outbound URL* is the input to the long-term plan: once we know which third-party hosts dominate (`go.opportuno.de` → `jobware.de`, `talent.com`, etc.), one or more of those becomes a follow-up parser. The debug log is the staging ground for that decision.
- **One file per component is enough for the analysis workflow.** The end-state query is `cat logs/jobs_beim_staat_html.log | extract outbound | extract host | sort | uniq -c`. A per-component file is a natural namespace, keeps unrelated quirks separated, and avoids one component's log noise drowning another's. The helper takes a `component_id` and a freeform message; the file name is derived deterministically. The helper is intentionally generic — parsers are the first caller, but any subsystem can adopt it without renaming.
- **The path lives next to the config, in the synced folder.** The synced folder (`~/application-pipeline/data/synched/` on the Pi) already houses **Config**, **Layout**, the **Results File**, **Failure Reports**, and `.seen.json`. Putting `logs/` there means the debug stream travels to the laptop via the same Syncthing channel as the rest of the user-visible artifacts. The laptop is the analysis machine; that's where this data needs to be.
- **The helper is module-level state set once at startup, not a singleton class or constructor-injected dependency.** Parsers are constructed by the orchestrator as zero-arg context managers (`cls()`); the **Parser Protocol** has no path parameter and shouldn't grow one for a debug helper. `__main__` calls `parser_log.configure(Path("synched") / "logs")` after `basicConfig`; caller modules call `parser_log.record(component_id, event_type, **fields)` and remain unaware of the path. Smoke tests and direct module invocations (`python -m application_pipeline.parsers.jobs_beim_staat_html`) skip the `configure` call, so `record` is a silent no-op — debug logs never leak into the test workspace.
- **Detection is shape-driven, not allowlist-driven.** After `_extract_job_id` returns `None`, the parser scans for the first `<a href>` whose host is not `jobs-beim-staat.de`. If found, the listing is external; record the href, emit `ExternalRedirect`. If no such link exists, the page genuinely is unexpected — raise `ParserError` as today. An allowlist of known redirect hosts (`go.opportuno.de`, …) would pre-filter exactly the population we want to discover; the debug log only earns its keep if every novel target reaches it.

## Considered alternatives

- **Keep raising `ParserError` and add a structured log line before raising.** Rejected as the long-term shape. Zero new surface and the stub is already sticky-marked `enrich_failed` (so we don't re-fetch next run), but the run-summary metric stays conflated, the `WARNING` is misleading for an expected-and-recurring case, and "log line in the main stderr" doesn't aggregate cleanly for the later-deployment analysis.
- **Return a wrapper-only `Position` with `raw_description=""`.** Rejected. The wrapper page carries title, employer, and location — fields the **Position Stub** already has from `discover`. The **Domain Pre-Filter** and **Relevance Classifier** would then run on a `Position` with no body text and silently drop it. The cost is paid (LLM call queues, classifier prompt) for a deterministic-zero-value outcome. It also pollutes the `kept`/`off_domain` ratios in metrics with a third population that isn't really either.
- **Follow the outbound redirect and scrape the third-party page.** Rejected for v1. Every partner aggregator (jobware, talent.com, opportuno, …) has its own HTML shape; each is a parser-sized commitment. The point of this ADR is the opposite — *first* characterize which partners actually appear, *then* invest in the most common one. Following the redirect today would either be a generic best-effort scrape (low signal) or a multi-parser landgrab (high cost). Out of scope.
- **Skip in `discover` instead of `enrich`.** Rejected on shape. The signal that distinguishes external from on-domain listings lives on the wrapper page, not on the card returned by `jobsearchRestApi`. Skipping in `discover` would require a probe fetch per stub, doubling HTTP cost and collapsing the two-phase `discover` / `enrich` split for this parser. The `enrich`-side detection costs one fetch per stub — which we pay anyway — and short-circuits before the second fetch (the iframe content fetch).
- **A `ParserError` subclass `ExternalListingError` with an `outbound_url` attribute.** Rejected. Mechanically routes through the existing error channel; the orchestrator could `isinstance`-check it before the generic `ParserError` branch. Saves one new payload type but embeds external listings in the error type hierarchy. Every reader who encounters `raise ExternalListingError` will reach for the "is this a bug?" model, exactly the conflation we want to retire.
- **Change `enrich`'s return type to `Position | None`.** Rejected. `None`-as-"skip-by-design" is a third in-band signal across the **Parser Protocol** that every parser, test, and orchestrator branch would have to handle — to encode one parser's quirk. The new `ExternalRedirect` payload is local to the parser → orchestrator queue and doesn't touch the Protocol's return type.
- **Persist the outbound URL on the dedup store record.** Rejected. Putting per-parser-quirk fields onto `.seen.json` couples the dedup schema to one parser's failure mode. The debug log file is the cheaper venue: it's already segregated per component, it's not on a hot path, and it can be deleted/rotated without touching dedup state.
- **One shared `external_redirects.log` for all parsers.** Rejected. The helper is generic ("any component can append a debug line to its own file") because debug streams from unrelated callers shouldn't share a file — a future jobs-beim-staat schema-change burst would drown a stellen.hamburg signal. Per-component files namespace cleanly.
- **Parser-only naming (`parser_debug_log`, `parser_id`).** Rejected. The implementation has zero parser-specific behavior — it takes an id and writes a file. Naming it after the first caller would force a rename the first time a non-parser subsystem wanted its own debug stream. `debug_log` / `component_id` is the same code with a name that survives.
- **`DebugLog` singleton class with explicit `set`/`get`.** Rejected as equivalent-but-heavier. The helper has no logic worth mocking — its sole job is to append a line to a file. A module-level `configure(path)` plus `append(component_id, message)` is two functions and zero classes; a singleton with `set`/`get` is three things with no behavior difference. Tests that care can call `configure(tmp_path)`; tests that don't care let `append` no-op silently.
- **Constructor-inject the logs path into every parser.** Rejected. The **Parser Protocol**'s effective contract today is "zero-arg context manager"; adding `logs_dir: Path | None = None` to every `__init__` is paper-cut boilerplate in every parser, every smoke test, and every test fixture, just so parser modules can pass the path into a helper that could have read it from module-level state in one place. The hidden global is a single-writer hidden global, configured exactly once.

## Consequences

- **`src/application_pipeline/parser_log.py`** owns both the logs-directory bootstrap and the per-component append behaviour. It exposes `configure(logs_dir: Path) -> None` (called once by `__main__` at startup; creates the directory if missing) and `record(component_id: str, event_type: str, **fields) -> None` (called by any subsystem; resolves to `<logs_dir>/<component_id>.log`, opens in append mode, prepends an ISO-8601 UTC timestamp, writes `<ts> <event_type> <k=v ...>\n`). Before `configure` is called, all functions are silent no-ops. State is one `Path | None` at module scope; writes are not locked (POSIX `O_APPEND` is atomic for line-sized writes and each `component_id` owns its own file).
- **`__main__` calls `parser_log.configure(Path("synched") / "logs")`** after `basicConfig`, before `run(config_path)`. The directory is materialized at startup; failure to create it raises and surfaces via the existing `_FATAL` failure-report flow.
- **A new payload type `ExternalRedirect(stub: PositionStub, outbound_url: str)`** is added to the parser → orchestrator queue alphabet (alongside `PositionStub`, `Position`, `ParserError`, `_PARSER_DONE`, `_ParserDead`). Lives in `src/application_pipeline/parsers/types.py` next to `Position`/`PositionStub`. Frozen dataclass.
- **`JobsBeimStaatParser.enrich` is the first caller.** After `_extract_job_id` returns `None`, scan the wrapper soup for the first `<a href>` whose URL host is not `jobs-beim-staat.de`. If found: call `parser_log.record("jobs_beim_staat_html", "external_redirect", stub_url=stub.url, outbound=href)`, return `ExternalRedirect(stub, href)` (i.e. the parser thread puts it on the outbound queue and skips the second fetch). If not found: raise `ParserError` as today. The check costs zero new HTTP fetches.
- **The orchestrator grows a fourth payload branch** (`isinstance(payload, ExternalRedirect)`): pops the matching pending-enrich entry, logs at INFO `external_redirect parser_id=… stub_url=… outbound=…`, calls `dedup_store.mark_seen(stub, "external_redirect")`, and increments a new local counter feeding `RunSummary.external_redirects`. No WARNING.
- **`RunSummary` gains an `external_redirects: int = 0` field**, reported alongside `enrich_failed` in the `done` line printed by `__main__`.
- **`.seen.json`'s `status` field gains a fourth value `"external_redirect"`** alongside `"kept"`, `"off_domain"`, `"enrich_failed"`. The **Deduplication Store** treats it as another sticky-skip status: subsequent runs short-circuit on the URL without re-fetching the wrapper. Once written to `.seen.json` this name is durable; renaming later requires migration.
- **CONTEXT.md updates:** **Deduplication** entry's `status` enumeration grows by one value; a new entry **External Redirect** is added near **Position Stub** describing the parser-emitted payload and the policy ("listings whose wrapper page is an outbound redirect to a third-party board are not enriched in v1; the outbound URL is recorded to the per-component debug log for later analysis").
- **The smoke test `test_discover_hamburg_returns_stubs_and_enrich_populates_description`** stops being a hard failure when the first stub happens to be external — `enrich` now returns an `ExternalRedirect` rather than raising. The test either iterates until it finds a non-external stub or asserts the union of outcomes is well-typed.
- **No change to existing parsers (`bundesagentur_api`, `stellen_hamburg`, `stellen_hamburg_api`).** The `parser_log` surface is available to them but unused at this point; they continue to raise `ParserError` on enrich failures as today.
