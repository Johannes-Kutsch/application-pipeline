# Parser owns body production; gates bundle interposes between discover and enrich

Per-source **Parsers** regain a body-production responsibility, opt-in per parser, exposed as a new mandatory `enrich(stub) -> EnrichResult` method on the Parser Protocol. `EnrichResult` carries an updated `PositionStub` (any optional fields the native path learned), the body text, and a per-stub `mode: Literal["native", "fallback"]` label. Parsers without a native path call a shared fallback helper (today's `httpx.get(stub.url)` + `strip_to_text(html, body_selector)`, lifted out of `LLMEnricher` into `parsers/body_fetch.py`).

The **LLM Enricher** collapses to: receive `(stub, body)`, run the LLM call (classify), run the post-LLM **Freshness Gate** arm on the LLM-inferred Header date, write the Card to `extracts.json`. It no longer fetches, follows redirects, strips, or owns `body_selector`. `body_selector` becomes parser-private state consumed only by the fallback helper.

A new **Gates Bundle** groups every non-LLM gate ŌĆö Dedup (Run-scoped + Store), Domain Pre-Filter, Freshness Gate, Content Gate ŌĆö behind a single orchestrator call site. The bundle runs **twice**: post-`discover()` and post-`enrich()`. Each gate inside the bundle is stateless and idempotent, with a "ready" predicate that no-ops when its required input isn't present (e.g. Content Gate no-ops pre-enrich because there's no body yet; Freshness Gate no-ops when both `posted_date` and `deadline` are `None`). Dedup and Pre-Filter run twice (cheap, idempotent) ŌĆö no per-stub "already-passed" tracking.

Pipeline order becomes: `discover ŌåÆ gates bundle ŌåÆ enrich ŌåÆ gates bundle ŌåÆ LLM call (classify) ŌåÆ post-LLM Freshness ŌåÆ CardStore`.

`enrich()` raises a typed exception when both the native path and the fallback fail unrecoverably (e.g. all HTTP layers exhausted). The orchestrator catches it and calls `mark_enrich_failed(stub)` ŌĆö terminal-skip. This aligns the implementation with CONTEXT.md's documented `enrich_failed` semantics (a pre-existing inconsistency under ADR-0041, where `LLMEnricher._fetch` returned `None` on any `httpx.HTTPError` without marking).

## Why

- **Body quality.** Bundesagentur's `/jobdetails` API returns a clean structured `stellenbeschreibung` field. Today's `LLMEnricher` ignores it, re-fetches the public HTML page, and strips it ŌĆö losing fidelity and paying an extra HTTP round trip per stub. Returning body ownership to parsers lets each source use its best available signal (API body, detail-page selector, library fallback) without baking source-specific knowledge into a shared stage.
- **Cost on steady-state runs.** Under ADR-0041, the `LLMEnricher` fetches the body for every stub that survives the upstream gates. With the bundle interposing dedup + prefilter + freshness *before* enrich, only fresh, in-domain, non-blacklisted stubs ever pay the body fetch. Typical daily run: 90k yielded stubs ├Ś ~95% dedup-hit rate in steady state = ~85k avoided body fetches per day.
- **Honest accounting on the status row.** The per-parser status row's `M enriched` counter has been dead-coded since ADR-0041 (`metrics.enriched()` exists but is never called; tests hard-code `0 enriched`). Resurrecting it with per-stub `mode` lets the operator see how often a "native" parser is silently falling back ŌĆö early warning when a source's API changes shape. Counter is shown only on parsers declaring `has_native_enrich = True`; counter wording is `M/N enriched` where M = `mode == "native"` count this run, N = stubs that reached enrich (issue #540).
- **Uniform gate site.** All non-LLM gates were already conceptually one bucket ŌĆö they all read stub-shaped input and drop with a transcript row ŌĆö but were threaded through the orchestrator at different call sites. Bundling them removes the temptation to "skip just this one gate this time" and makes the pipeline order explicit at a single place.
- **Single gate-bundle call site, not three.** A "run gate X here, run gate Y there" arrangement makes it hard to reason about ordering invariants when a new gate is added. The bundle runs everyone, every time, and each gate's ready-predicate keeps it honest about when it has work to do.

## Considered alternatives

- **Keep ADR-0041 as-is; just drop the dead `enriched` counter from the status row.** Rejected: leaves Bundesagentur's clean API body unused and the steady-state-run wastage unaddressed. The status-row dichotomy in #540 was the surface symptom, not the underlying problem.
- **Lazy callback ŌĆö `LLMEnricher` calls `parser.enrich(stub)` when present, falls back to its own fetch when not.** Rejected during grilling: the user found "LLM Enricher reaches back into the parser for body" architecturally convoluted. Cleaner to give the parser sole body-production responsibility and let the LLM Enricher know nothing about parsers.
- **Eager ŌĆö `discover()` yields stubs with bodies inline.** Rejected: every yielded stub pays the body fetch regardless of whether dedup/prefilter/freshness would have dropped it. On a daily run dominated by URL-hits, that's tens of thousands of wasted API calls per day. Splitting `discover()` from `enrich(stub)` lets the orchestrator interpose dedup before body fetch.
- **Optional `enrich()` method, pipeline calls fallback when absent.** Rejected: forces `hasattr` branching in the orchestrator. Mandatory method with parser-internal fallback delegation is uniform and keeps `body_selector` off the public Parser Protocol.
- **Per-parser `has_native_enrich` flag controls counter increment.** Rejected: a parser with a native path that 404s 30% of the time would inflate its counter on fallback paths. Per-stub `mode` is honest; the per-parser flag only governs *whether the column is shown*, not whether the counter increments.
- **Subset of gates in the bundle (e.g. exclude Content Gate because it lives near the LLM call).** Rejected: the user's principle is "all non-LLM gates in one bundle." Content Gate is a body-validity check, not an LLM-call guard. Including it leaves `LLMEnricher` with just the LLM call site as its sole responsibility.
- **Track per-stub gate-passage state to skip Dedup/Pre-Filter on the second bundle run.** Rejected: state on the stub adds complexity for ~zero CPU savings (Dedup is a dict lookup, Pre-Filter is a substring scan). Stateless idempotent no-ops on repeat are simpler.
- **Preserve today's "soft skip" on HTTP fetch failures (return `None`, don't mark `.seen.json`).** Rejected: misaligned with CONTEXT.md's documented `mark_enrich_failed` rule and bloats `.seen.json` retry pressure on permanently broken URLs. Aligning now under this refactor is the right scope boundary.

## Consequences

- **`Parser` Protocol gains** `enrich(stub: PositionStub) -> EnrichResult` (mandatory) and `has_native_enrich: bool` (module-level constant, default `False` ŌĆö gates status-row column visibility only). `body_selector: str | None` retires from the Protocol surface and becomes parser-private.
- **`EnrichResult` dataclass (new)** carries `stub: PositionStub` (possibly with optional fields back-filled by native enrich), `body: str`, `mode: Literal["native", "fallback"]`. The orchestrator threads the returned stub forward (not the original).
- **`EnrichFailedError` (new typed exception)** raised by `enrich()` when both native and fallback HTTP paths fail unrecoverably. Caught at the orchestrator boundary ŌåÆ `mark_enrich_failed(stub)`.
- **`parsers/body_fetch.py` (new shared helper)** owns the generic fetch + strip path. Signature roughly `fetch_and_strip(url: str, *, body_selector: str | None, http: ParserHttp) -> str`. Lifted from `LLMEnricher._fetch` + `strip_to_text`. Each parser's `enrich()` can call it directly for fallback mode.
- **`LLMEnricher` shrinks** to: take `(stub, body)`, run `classify_relevance`, run post-LLM Freshness on parsed Header date, write `extracts.json` on `matches=True`. No httpx client. No body strip. No `_fetch`. No `body_selector` argument. No token-cap-overflow stash (moves to body fetch helper since it's a body-pipeline concern, not an LLM-call concern).
- **`Gates Bundle` (new module)** behind a single function `run_gates(stub, *, run_log, metrics, dedup, prefilter, freshness, content) -> Verdict`. Each gate inside checks its own ready-predicate; gates with absent inputs pass silently without writing transcript rows. Single transcript row per stub per bundle invocation summarising the final verdict; per-gate transcripts continue to be written by individual gates when they actually evaluate.
- **`Freshness Gate` runs three times** (was two under ADR-0041): post-discover (cheap stub.posted_date), post-enrich (catches dates the native enrich filled in), post-LLM (catches dates the LLM inferred from body). The two post-stub-update runs are inside the gates bundle; the post-LLM arm stays standalone inside the LLM call site. Per-position transcript `gate_arm` field extends to `"discover" | "post_enrich" | "post_llm"`.
- **`Content Gate` moves** out of `LLMEnricher` into the Gates Bundle. Pre-enrich bundle invocations no-op it (no body); post-enrich invocations evaluate it. Effective customer post-refactor is fallback-mode enrichments (native paths return non-empty by construction or raise to `enrich_failed`).
- **Status row `M enriched` counter resurrects with new semantics**: `M/N enriched` shown only on parsers declaring `has_native_enrich = True`. M increments on `mode == "native"` returns; N increments on every successful `enrich()` call. Dead `metrics.enriched(parser_id)` regains a real call site (orchestrator, after each `enrich()` return). Test `test_parser_row_body_tracks_queries_stubs_enriched` updates accordingly.
- **Bundesagentur parser gains a native enrich** calling `/jobdetails/{ref}`, returning the API's body field and back-filling `posted_date` if the detail response exposes one the search page didn't. Declares `has_native_enrich = True`.
- **stellen.hamburg and jobs-beim-staat parsers** get a one-line `enrich` that delegates to the fallback helper. Both declare `has_native_enrich = False`. No status-row counter shown for them.
- **CONTEXT.md updates**: `Parser`, `LLM Enricher`, `Content Gate`, `Freshness Gate`, `Position Stub`, `Status Display` entries revise to match. New entry **Gates Bundle**. New entry **EnrichResult**. `enriched` counter description on the status row entry updates from "always 0 dead-code" to its new meaning.

## Supersedes / amends

- **Amends ADR-0041**: reverses the "LLM Enricher owns body fetch" and "single-phase Parser API" decisions. Keeps everything else in ADR-0041 ŌĆö Position dataclass retirement, StructuredExtract retirement, Card structure simplification, redirect-following-by-default behavior (now living in the fallback helper instead of inside `LLMEnricher`), `failures/oversized/` and `failures/malformed/` stash patterns.
- **Amends ADR-0025**: Freshness Gate runs at three call sites (was two under ADR-0041's amendment of ADR-0025), with `gate_arm: "discover" | "post_enrich" | "post_llm"`.
- **Amends ADR-0039**: Content Gate's call site moves from inside `LLMEnricher` to the Gates Bundle. Drop condition and `.seen.json`-no-write semantics unchanged.
