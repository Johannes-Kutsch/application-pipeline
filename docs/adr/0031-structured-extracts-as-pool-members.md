# Classifier produces structured extracts; in-domain pool membership lives on dedup status

The **Relevance Classifier**'s output expands from `{in_domain: bool}` to `{in_domain: bool, extract: StructuredExtract | None}`. When `in_domain` is true, `extract` carries a fixed-field structured representation of the listing — `{seniority, work_model, contract_type, key_skills, key_responsibilities, must_have_requirements, notable_caveats}`. Extracts are persisted in a sidecar store (`data/extracts.json`) keyed by a **stable cross-URL identifier** shared across tuple-aliased records. They live as long as the listing stays in the **Pool**, and are deleted when the listing transitions to `selected_by_judge` or `out_of_domain`. The **Match Judge** consumes only extracts — full `raw_description` is no longer carried into the judge prompt. The **Pool** is the implicit set `{url ∈ .seen.json : status == in_domain ∧ parser_re_discovered_today}` — no explicit pool data structure exists.

This ADR extends ADR-0021's classified-in-domain resumption logic into the new normal: every in-domain item naturally resumes across runs because pool membership is a dedup-state fact, not a special crashed-pre-judge recovery hint. Companion ADRs are ADR-0029, ADR-0030, ADR-0032, ADR-0033.

## Why

- **Description tokens were being paid twice per item.** Today, the classifier reads each `raw_description` (per `ClassifyItem` in `src/application_pipeline/llm/types.py:42-46`), emits `in_domain: bool`, and discards everything else. The judge then re-reads the same `raw_description` to produce `MatchVerdict`. With items lingering in the pool across runs (per ADR-0029), the judge would re-read it again every run. The structured extract is paid once at classify time, persists, and is reused — every subsequent judge call sees the compact form instead of the full description.
- **The extract carries enough signal for the judge to rank and to author Cards.** The seven fields plus a free-text `notable_caveats` field (≤200 chars, where the classifier puts things like "remote ausgeschlossen" or "explizit Junior, Deutsch C2 Pflicht") are field-aligned for comparison and tone-aware where the fields fall short. The judge does not need to see the full description again to author `matched` / `missing` / `summary`.
- **Pool membership is *already* on the dedup record.** Today's `classified_in_domain` (ADR-0021) plus the new state-name cleanup makes `status == in_domain` exactly mean "this URL is in the pool." Re-discovery by a parser is the natural availability check — if the parser doesn't yield the URL today, it's not in today's judge list, regardless of how long it has been `in_domain`. No availability HEAD-check, no TTL eviction, no separate pool index.
- **A sidecar extract store keeps `.seen.json` small.** Extracts are ~300-500 tokens of JSON per item. Embedding them in `.seen.json` would inflate the file by orders of magnitude over years of in-domain items, and every Syncthing-synced write would carry the bloat. A sidecar `data/extracts.json` is keyed by a stable id, only carries `status=in_domain` records, and is purged on transition to `selected_by_judge` / `out_of_domain`. Sidecars per-item under `data/extracts/<id>.json` were considered (Q6 (c)) but rejected — too many small files for Syncthing.
- **The stable cross-URL key handles syndication.** Today's tuple-tier dedup (ADR-0004) recognizes the same role under two URLs and aliases them. The extract store keys by the same stable identifier (canonical-url chasing on the alias graph, or a synthetic id — implementation detail) so a role's extract is computed once even when re-discovered under a different URL.

## Considered alternatives

- **Embed extracts in `.seen.json`.** Rejected (Q6 (a)): unbounded growth, every write Syncs the whole blob, and the dedup store becomes a content cache (which ADR-0021 explicitly rejected widening into).
- **Per-item sidecar files (`data/extracts/<id>.json`).** Rejected (Q6 (c)): many small files play badly with Syncthing; one file per pool member is a worse story than a single bounded blob.
- **No extract — judge re-reads `raw_description` from a content cache.** Rejected: re-introduces the description tokens at the judge step on every rediscovery, which is exactly the win the structured extract delivers.
- **Caveman extract (stop-word-stripped key phrases) instead of structured fields.** Rejected (Q4): preserves more raw signal but loses negations ("kein Homeoffice" → "Homeoffice"), and gives the judge a noisier comparison surface. The structured shape with a `notable_caveats` escape hatch covers tone signal without paying the negation-loss cost.
- **Extract-on-demand at judge time** (no persistence, classifier still emits `bool` only). Rejected: doesn't solve the cross-run re-read cost. The whole point of persisting is that a 30-day-old pool member doesn't re-pay description tokens when it's judged again tomorrow.
- **Re-classify on each rediscovery to refresh the extract.** Rejected: the classifier's `bool` output is stable across rediscoveries (the description didn't change), and the extract is a function of the description. Re-classifying re-pays tokens for no new information. Listings whose description meaningfully changes between runs are a non-event for v1.
- **TTL-evict in-domain items after N days.** Rejected for v1 (Q11 (c) deferred): the natural eviction is "the parser stopped re-discovering it" — at that point the record stays `in_domain` in `.seen.json` and never re-enters the judge list, which is effectively the same as eviction. If the dead `in_domain` records become operationally painful (size, observability), add stale-eviction-by-age then.

## Consequences

- **`ClassifyItem` and `RelevanceVerdict` shapes change.** `RelevanceVerdict` becomes `{in_domain: bool, extract: StructuredExtract | None}` where `extract` is non-None iff `in_domain` is true. `StructuredExtract` is a frozen dataclass in `src/application_pipeline/llm/types.py` carrying the seven fields described above (with `key_skills`, `key_responsibilities`, `must_have_requirements` as `list[str]` with bounded length, e.g. ≤10 each).
- **Classifier prompt template** (`src/application_pipeline/templates/prompts/`) gains a "for each in-domain item, also produce the structured extract" section. Output schema becomes `[{"id": "...", "in_domain": true, "extract": {...}}, {"id": "...", "in_domain": false}, ...]`. The `<verdicts>` wrapper tag is unchanged.
- **Extract store** lives at `data/extracts.json`, shaped `{stable_id: StructuredExtract}`. The classify worker writes the extract eagerly (same fsync discipline as `mark_in_domain`) so a crash mid-run doesn't lose work the classifier already paid for. The dedup store's `mark_in_domain` (renamed from `mark_classified_in_domain` per ADR-0029) takes the extract as a kwarg and writes both atomically.
- **Stable id key.** Implementation may choose either canonical-url chasing (each `.seen.json` record stores `canonical_url`, alias writes point to the original) or a synthetic id (UUID minted on first-seen, persisted on the record). Either way, the extract store keys by the same id and tuple-aliased URL records share their extract.
- **Judge call shape.** `LLMExtractor.judge_match` is replaced by `judge_top_n(candidates: list[JudgeCandidate]) -> list[MatchVerdict]` where each `JudgeCandidate` carries `id`, the stable key, and the `StructuredExtract`. The judge sees a list of compact extracts, ranks the top 5, and returns `MatchVerdict` per winner — no `raw_description` ever enters the judge prompt.
- **Daily file rendering still uses the full `raw_description`.** The **Card**'s `## Job Description` section is sourced from the **Position** (re-enriched at the time of the daily run), not from the extract. The extract is for ranking; the description is for reading. **Position** records are *not* persisted across runs — `Position` is reconstructed from `Parser.enrich(stub)` for each in-domain item discovered today, same as today.
- **Status transitions:** `mark_in_domain(stub, extract)` writes status `in_domain` and stores the extract; `mark_selected_by_judge(stub)` writes status `selected_by_judge` and deletes the extract from the sidecar; `mark_out_of_domain(stub)` writes status `out_of_domain` and (no-op) ensures no extract exists.
- **Pool size today** is computed as `len([url for url in today's enriched stubs if .seen.json[url].status == in_domain])`. An optional `pool_size_today` event in `pipeline_orchestrator.events.jsonl` (operator visibility — deferred to implementation).
- **ADR-0021's `judge_resumed` counter is no longer meaningful** as a special case — every in-domain item resumes across runs, that's the new normal. The Run Divider it landed on is also retired (ADR-0030).
- **No back-compat for the old `.seen.json` shape** — extracts simply don't exist for legacy records. Per ADR-0033, the migration policy is "wipe state", so legacy `classified_in_domain` records don't need translation.
