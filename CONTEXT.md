# Application Pipeline

A personal job-discovery and triage pipeline. Fetches listings from a small set of sources, classifies relevance with a local LLM, accumulates an in-domain **Pool** across days, and emits one dated **Daily Results File** per day carrying the **Daily Top-5** **Cards** ranked by the **Match Judge**. Application authoring (CV/cover letter generation) is explicitly out of scope for v1.

## Scope

- **In scope (v1):** Working **Parsers** for **Bundesagentur**, **stellen.hamburg**, and **jobs-beim-staat**, each provably returning **Position Stubs** and capable of enriching them. Smoke-tested standalone on the laptop. No LLM, no orchestration, no **Daily Results File** in v1.
- **In scope (v1.1):** Full pipeline running on the Pi — orchestrator, **Relevance Classifier** (producing **Structured Extracts**), **Match Judge** (one call per run, picks the **Daily Top-5** from the **Pool**), **Deduplication**, daily file, cron firing once per day, Syncthing sync of the daily files to the laptop. Ollama is no longer relevant — Claude CLI is the runtime backend.
- **Out of scope:** **CV** / **Cover Letter** generation, LaTeX pipeline, **Profile** ingestion, additional commercial parsers (LinkedIn / StepStone / Indeed / Adzuna), Playwright, browser automation.

## Language

### Pipeline artifacts

**Config**: A `config.py` in the Pi's Syncthing-synced folder (`~/application-pipeline/data/config.py`) that controls the search — `KEYWORDS`, `SKILLS`, `NEGATIVE_KEYWORDS`, `SOURCES`, `LOCATIONS`, `INCLUDE_REMOTE`, the `USER_INFO_DIR` path, the **Layout** path override, and the Claude settings (`claude_classify_batch_size: int` default 100 per ADR-0016, and an optional `claude_cli_path` override for non-default CLI installs). Plain Python literals, edited in place, with `#` comments. The Pi's copy is materialised on first bootstrap from a template shipped inside the package at `src/application_pipeline/templates/config.py` via `python -m application_pipeline init <dir>`; per ADR-0013, subsequent edits happen on the laptop's paired copy and Syncthing propagates them. Loaded by the **Config Loader**, which returns a frozen typed `Config` dataclass. `KEYWORDS` is a single global list applied to every **Source** (Cartesian product over keyword × source); **Deduplication** absorbs the overlap. No per-source keyword overrides in v1. Each entry in `SOURCES` is a `SourceEntry` carrying its own `max_results` cap (default 1000) controlling how many newest-first listings that **Source** yields per run. `NEGATIVE_KEYWORDS` drives the **Domain Pre-Filter**; entries validate to length ≥ 3. `MAX_LISTING_AGE_DAYS: int` (default 180, validated `≥ 1`) drives the **Freshness Gate**'s `posted_date` age threshold; the gate's `deadline < anchored_today` arm is non-tunable. Cron schedule is not a `Config` field — it lives in the Pi's crontab (per ADR-0033, once daily). `USER_INFO_DIR` (default `user-info/`) is the directory containing the **Triage Profile** markdown files injected into the hardcoded prompts (see ADR-0019). _Avoid_: config file (without qualifier), settings file, search config.

**Layout**: A `layout.py` in the Pi's Syncthing-synced folder (`~/application-pipeline/data/layout.py`) sitting alongside the **Config**, owning all user-tunable cosmetic and structural choices for the **Daily Results File**. Seeded from `src/application_pipeline/templates/layout.py` by the same `init` subcommand that materialises the **Config** (ADR-0013). Carries named placeholder groups and the full `CARD_TEMPLATE` (one shared template — no per-tier dispatch, since **Match Tier** is retired per ADR-0029). Placeholder set excludes `emoji` / `color` / `tier` (gone) and includes `rank` (1..5). Plain Python module loaded at runtime by `load_user_module` (the same machinery as **Config**), validated into a frozen typed `Layout` dataclass. The **Renderer** consumes the `Layout` and substitutes placeholders via `str.format_map`. Edited freely by the user — re-design happens here, not in package code. Errors raise `LayoutError`, a subclass of `UserSettingsError` (the shared base with `ConfigError`). _Avoid_: style file, theme file, template file (when the loaded dataclass is meant).

**Daily Results File**: One dated markdown file per calendar day at `data/results/YYYY-MM-DD.md`, holding the **Daily Top-5** as rendered **Cards** in **Rank** order. The date is **cron-anchored** (per ADR-0030 / ADR-0032) — a run started on day X writes to day X's file even if it crosses midnight while sleeping through a quota window. No `FILE_HEADER`, no preamble, no **Run Divider** — the file starts with the first **Card** and ends after the fifth. If the **Pool** had fewer than 5 in-domain candidates today, the file carries however many cards exist; if the pool was empty, no file is written. Syncthing-synced send-receive from the Pi to the laptop, write-once on the Pi (no run modifies a prior day's file), so sync conflicts are near-zero. _Avoid_: results file (without "daily" qualifier — the trio is retired), `current.md` (gone), dated results file (use "daily").

**Failure Report**: A markdown file written to `data/failures/<timestamp>.md` whenever a run fails — cron-wrapper deploy errors, orchestrator runtime errors, **Match Judge** call failure (no daily file produced), and non-quota classifier errors that left the run with no writeable output. Per ADR-0012. Syncs to the laptop via the same Syncthing folder as the **Daily Results File**. Acknowledged by the applicant deleting the file (Syncthing propagates the delete). Quota errors do **not** trigger a Failure Report — they sleep until the API's stated reset (per ADR-0032). _Avoid_: incident report, error log (the log file is a separate artifact on the Pi).

**Position**: A single job listing surviving the **Relevance Classifier** and the **Match Judge**'s top-5 selection. Identified within its **Daily Results File** by its rendered H1 (`# {company} · {title} · {location_segment}`), not by a numeric counter — the explicit `{rank}` placeholder in the **Card** carries the ordering. _Avoid_: job, listing, vacancy.

**Position Schema**: The frozen-dataclass shape every **Parser** must return from `enrich()` — `Position(stub: PositionStub, raw_description: str, ...)` with optional `salary: str | None` (free string — German pay grades like `"E13 TV-L"` preserved verbatim), `contract_type: Literal["permanent", "fixed-term", "freelance"] | None`, `employment_type: Literal["full-time", "part-time", "internship"] | None`, `work_model: Literal["remote", "hybrid", "on-site"] | None`, `posted_date: date | None`, `deadline: date | None`. Closed-enum fields are `None` when the source exposes no signal — the **Parser** never guesses. `Position` *contains* its `PositionStub` (composition, not inheritance); the stub's `url, title, company, location, source` reach consumers as `position.stub.<field>` (required: `url, title, source`; `company, location` are `str | None`). Both dataclasses live in `src/application_pipeline/parsers/types.py`; **Domain Pre-Filter**, **Renderer**, **Deduplication**, and **LLM Extractor** import from there (parsers do not import from consumers). `Position` and `PositionStub` are both `frozen=True` and never mutated. `Position` records are *not persisted across runs* — they are reconstructed by `Parser.enrich(stub)` on each day's run for items in the **Pool**. The **Match Judge** later attaches a **Match Verdict** (`rank, matched, missing, summary`) to the 5 winners; these are not parser outputs. _Avoid_: output format, dict structure (the schema is dataclass-typed).

**Raw Description**: The full body text of a **Position** as returned by the **Parser**, normalized to plain Unicode with `\n\n` paragraph breaks (no HTML, no Markdown, no entities). Each **Parser** owns the normalization for its source's body shape — block elements become paragraph boundaries, entities decode, whitespace collapses. Empty after normalization is allowed (`raw_description = ""`); the **Domain Pre-Filter** / **Relevance Classifier** will drop it downstream. Input to the **Relevance Classifier** (where the **Structured Extract** is produced from it) and rendered verbatim into the daily file's **Card** for the 5 winners. The **Match Judge** does *not* see the raw description — it ranks on **Structured Extracts** only (per ADR-0031). _Avoid_: description (when full text is meant).

**Structured Extract**: A fixed-field representation of an in-domain **Position**, produced by the **Relevance Classifier** as a side output alongside `in_domain: true`, and consumed by the **Match Judge** as its sole per-candidate input. Carries `seniority`, `work_model`, `contract_type`, `key_skills: list[str]` (≤10), `key_responsibilities: list[str]` (≤10), `must_have_requirements: list[str]` (≤10), and a free-text `notable_caveats: str` (≤200 chars) — the escape hatch for tone, negation ("kein Homeoffice"), and other signals the closed fields miss. Per ADR-0031. Persisted in `data/extracts.json` keyed by a **stable cross-URL identifier** (canonical-url or synthetic id, shared across tuple-aliased records per ADR-0004) so the same role's extract is paid once even when re-discovered under a syndicated URL. The classify worker writes the extract eagerly (same fsync discipline as `mark_in_domain`) so a crash mid-run doesn't lose work already paid for in tokens. Deleted from the sidecar when the **Position**'s status transitions to `selected_by_judge` (winner today) or `out_of_domain` (e.g. manually demoted). _Avoid_: summary (overloaded with the **Match Verdict**'s `summary`), classification (the classifier produces the extract, but the extract is not the classification verdict).

### Filtering & scoring

**Triage Profile**: The applicant's prose self-description plus the rules that decide what counts as in-domain or a good match. Lives in `data/user-info/` as three markdown files — `self-description.md` (candidate background, fed into both LLM call sites), `domain-fit.md` (in/out-of-domain rules, fed into the **Relevance Classifier**), and `match-criteria.md` (hard constraints, location/seniority/remote preferences, fed into the **Match Judge**). Per ADR-0019, the **Prompt Loader** concatenates the relevant files for each call site and the **LLM Extractor** injects the payload into the hardcoded prompt's `{USER_INFO}` slot wrapped in `<user-info>` tags. The user edits these files freely without touching the hardcoded prompt that defines the agent protocol. Language-neutral (German prose; the pipeline is single-language per ADR-0019). Not to be confused with the Phase-2 **CV Profile** used for generating application material. _Avoid_: profile (without qualifier), bio.

**Skill**: A hard-skill or technology item from the applicant's comprehensive list in the **Config**'s `SKILLS`. Rendered as a bullet list into the **Match Judge** prompt's `{skills}` slot at `ClaudeExtractor` construction (one entry per line, `- ` prefix). Per ADR-0026, `SKILLS` is **only** consumed by the judge prompt — the **Domain Pre-Filter** no longer reads it. Stays in `config.py` as a Python list. The LLM's `matched`/`missing` lists are open-vocabulary (no closed-vocab validation against `SKILLS`). _Avoid_: keyword (when matching).

**Keyword**: A search term used to query a **Source**, from the **Config**'s `KEYWORDS`. Distinct from a **Skill** — does not affect judgment. Distinct from `NEGATIVE_KEYWORDS`, which drives the **Domain Pre-Filter**. _Avoid_: skill (when querying), negative keyword (when querying).

**Negative Keyword**: An entry in the **Config**'s `NEGATIVE_KEYWORDS`. Patterns (length ≥ 3) that, if found via case-insensitive substring match in a **Position**'s **title** (body text is not consulted, per ADR-0026), cause the **Domain Pre-Filter** to drop the listing. No rescue mechanism — the prefilter is title-only blacklist with no whitelist offset. _Avoid_: blacklist, exclusion.

**Match Verdict**: The structured output of the **Match Judge** for each of the 5 winners — `{rank: 1..5, matched: list[str], missing: list[str], summary: "2-3 sentences"}`. `matched` and `missing` are short open-vocabulary phrases authored by the LLM (validation is type/length only — no closed-vocabulary check against `SKILLS`). `rank` is the explicit position in the **Daily Top-5** (1 = best match of the day). Drives **Card** content; **Match Tier** is retired per ADR-0029. _Avoid_: score, rating, tier.

**Rank**: The `rank` field of the **Match Verdict** — an integer 1..5 assigned by the **Match Judge** when picking the **Daily Top-5**. Surfaced in the **Card** via a `{rank}` placeholder in the **Layout** (e.g. `**Rank 1/5**` line under `## AI Assessment`). Not a score (no numeric comparability across days); not a tier (no enum). Just an ordering. _Avoid_: tier (retired), score, position (overloaded with **Position**).

**Pool**: The implicit set of in-domain **Positions** eligible for today's **Match Judge** call — every URL in `.seen.json` with `status == in_domain` that was re-discovered by a parser in the current run. There is no separate pool data structure; pool membership is computed each run as `{url ∈ enriched_today : .seen.json[url].status == in_domain}`. An item enters the pool by being classified `in_domain` (status flips from `not_classified` to `in_domain`, and a **Structured Extract** is written to `data/extracts.json`). It stays in the pool, run after run, until it is either picked by the judge (status → `selected_by_judge`, extract deleted) or no longer re-discovered by any parser (the dedup record sticks around but never re-enters today's judge list — natural expiry). No pool size cap in v1 (per ADR-0031); no TTL eviction. _Avoid_: queue (implies FIFO), candidate set (too generic), judge list (use **Pool**).

**Daily Top-5**: The 5 **Positions** the **Match Judge** returns from its single end-of-run call, drawn from today's **Pool**. May be fewer than 5 if the pool was smaller; never more. Surfaced as **Cards** in **Rank** order in the **Daily Results File**. Each Top-5 winner gets `mark_selected_by_judge(stub)` after its **Card** is appended and fsynced, which removes it from the **Pool** for future runs. _Avoid_: top-N (when 5 is meant), shortlist (overloaded).

**Relevance Classifier**: The conceptual name for the call site that decides whether a **Position** is in the applicant's professional domain (AI / Data / Game Dev / SWE) and, when in-domain, produces its **Structured Extract**. Implemented as a call to `LLMExtractor.classify_relevance_batch` — there is no wrapper class. Per ADR-0031, the per-batch response shape is `[{"id": "...", "in_domain": true, "extract": {...}}, ...]` for in-domain items and `[{"id": "...", "in_domain": false}, ...]` for out-of-domain ones. Batched (default 100 items per call, per ADR-0016); may stay pipelined with parsers (per ADR-0017) or run serially after the parser phase — implementation may choose, since one-run-per-day removes the wall-clock pressure. On `ClaudeUsageLimitError` the orchestrator sleeps until the API's stated reset time + 2 minutes (per ADR-0032) and retries the failed batch — no degraded-success exit. A malformed batch response (bad JSON, length/id mismatch) fails the whole batch — none of its items are marked seen, and the failure is appended to `data/logs/llm_classify_relevance.events.jsonl` with the full prompt + response captured in `data/logs/llm_classify_relevance.transcripts.jsonl`. Per ADR-0025, `llm_classify_relevance` and `llm_judge_match` are each their own log component; per ADR-0028 every component identifier carries a layer prefix (`parser_`, `llm_`, `pipeline_`). Runs **only on Positions that survive the Domain Pre-Filter** and whose `.seen.json` status is `not_classified`. **Off-domain Positions get `mark_out_of_domain(stub)` immediately and are never written to a Daily Results File.** _Avoid_: filter (ambiguous), gate.

**Freshness Gate**: A deterministic stage that runs after `Parser.enrich()` and before the **Relevance Classifier**, dropping **Positions** that are temporally invalid. Per ADR-0034. Drop conditions: `posted_date is not None and (anchored_today - posted_date).days > MAX_LISTING_AGE_DAYS`, **or** `deadline is not None and deadline < anchored_today`. `None` on a field means "no signal, don't drop on that field alone"; if both are `None` the gate passes. `anchored_today` is the cron-anchored logical date (per ADR-0030), shared with the **Daily Results File**. Drops write the new terminal-skip dedup status `expired`; on an `in_domain → expired` transition the URL's entry in `data/extracts.json` is deleted (same cleanup as `selected_by_judge`). Runs on **every** enrich including Pool re-discovery, so pool items whose source keeps surfacing them naturally age out. Its own log component `pipeline_freshness` (per ADR-0024/0028) emits per-Position transcripts (`url, title, source, posted_date, deadline, anchored_today, age_days, passes, reason ∈ {passed, too_old, deadline_passed, too_old_and_deadline_passed}`) and a per-run `event=run_complete` aggregate; its own **Agent Row** in the **Status Display** between `pipeline_prefilter` and `llm_classify_relevance`. Future-dated `posted_date` (negative age) passes silently — parser data hygiene is not this gate's job. _Avoid_: staleness filter, expiry gate, date filter.

**Domain Pre-Filter**: A deterministic module that runs before the **Relevance Classifier**, dropping **Positions** whose **title** matches any **Negative Keyword**. Title-only, blacklist-only — there is no whitelist mechanism and no body-text matching (per ADR-0026). Substring match (case-insensitive, after `normalize()`). Pure, stateless, pre-LLM, single responsibility. Function-shaped: the **Pipeline Orchestrator** holds a precomputed `list[str]` normalized blacklist and dispatches per **Position**. Drops are written to the dedup store with status `out_of_domain` (per ADR-0029's status renaming — same status the **Relevance Classifier** writes for LLM-rejected items). The module enumerates *all* matching keywords on `PreFilterVerdict` so the orchestrator can record a per-position decision into `data/logs/pipeline_prefilter.transcripts.jsonl` and emit a per-keyword end-of-run summary into `pipeline_prefilter.events.jsonl`. Each transcript record carries `url, title, source, passes, reason ∈ {passed, blacklist_drop}, blacklist_matches, title_len`. Aggregate stats count *positions matched* (not raw occurrences) and surface a `NEGATIVE_KEYWORDS_dead` list of zero-match keywords for config pruning. The Pre-Filter assumes its input has already been keyword-filtered upstream (per the **Parser** invariant). _Avoid_: filter (ambiguous), gate, classifier.

**Match Judge**: The conceptual name for the call site that picks the **Daily Top-5** from today's **Pool**. Implemented as a single call to `LLMExtractor.judge_top_n(candidates)` per run — there is no wrapper class, and there is no per-item judge call. Per ADR-0029, the call takes a list of `JudgeCandidate`s (each carrying a stable id and the candidate's **Structured Extract**) and returns up to 5 **Match Verdicts** (`{id, rank, matched, missing, summary}`). The judge sees only structured extracts — no `raw_description` is ever in the judge prompt. **Skills** and the **Triage Profile**'s match-criteria reach the judge via the prompt's `{skills}` and `{USER_INFO}` slots as today. On `ClaudeUsageLimitError` the orchestrator sleeps and retries the same call (per ADR-0032). On any non-quota error the run fails without writing a daily file and a **Failure Report** is emitted (per ADR-0030). Runs only on the days the **Pool** is non-empty.

### Deduplication and run state

**Deduplication**: The process of skipping **Positions** seen in a previous run or already yielded earlier in the current run. Three-tier: an in-run ephemeral URL set owned by a per-run dedup scope (`RunScopedDedup`, obtained from the **Deduplication Store** via its `run_scope()` context manager — scoped to one `run()` invocation, sitting in front of the persistent store to absorb the Cartesian `KEYWORDS × LOCATIONS × SOURCES` overlap without paying duplicate `enrich()` cost), plus the persistent **Deduplication Store** with two tiers: exact-match on URL, plus exact-match on the `(company, title, location)` tuple lowercased. The tuple tier only fires when **all three fields are non-`None`** on both sides. When the tuple matches under a new URL (a syndicated copy), the **Deduplication Store** records an alias entry under the new URL — copying the original record's `status` and `first_seen` — so subsequent runs hit the cheap URL tier without re-paying the tuple lookup (ADR-0004). `first_seen` answers "when did this *role* first appear", not "when did this URL first appear". `is_seen` returns a `SeenResult` variant; `url_hit` and `tuple_hit` skip; `in_domain` enriches and routes the stub directly into the **Pool** for today's judge call (no classify pass); `miss` processes from scratch. The alias write is performed inside `is_seen`.

**Dedup status enum** (per ADR-0029 / ADR-0031, supersedes the prior `off_domain | kept | enrich_failed | external_redirect | classified_in_domain` set):
- `not_classified` — first-contact write when a stub enters the dedup store; eligible for the classifier next time it is re-discovered.
- `out_of_domain` — written by both the **Domain Pre-Filter** (title hit a **Negative Keyword**) and the **Relevance Classifier** (LLM verdict `in_domain: false`). Terminal-skip.
- `in_domain` — written by the classifier on `in_domain: true`, alongside the **Structured Extract** in `data/extracts.json`. Means "currently in the **Pool**"; re-discovery on a later run routes the stub directly into that day's judge candidates without re-classifying.
- `selected_by_judge` — written after the judge picks the item, after the **Card** is appended and fsynced to the daily file. Terminal-skip; the extract is deleted from the sidecar on this transition.
- `expired` — written by the **Freshness Gate** when `posted_date` exceeds `MAX_LISTING_AGE_DAYS` or `deadline` is before the cron-anchored date (per ADR-0034). Terminal-skip. May transition from `not_classified` (first-contact drop after enrich) or from `in_domain` (Pool re-discovery drop); the latter also deletes the **Structured Extract** from `data/extracts.json`.
- `enrich_failed` — parser's `enrich()` raised `ParserError`; terminal-skip (parser burned its retry budget).
- `external_redirect` — parser emitted an `ExternalRedirect` payload (per ADR-0015 / ADR-0028); terminal-skip.

**Error semantics for `mark_*`** (orchestrator behavior, single-writer Pi):
- **Pre-filter drop** → `mark_out_of_domain(stub)` immediately. No LLM cost paid.
- **Classifier non-quota error** (Claude CLI gone, timeout, malformed JSON, etc.): the whole batch fails; **do not mark the batch's items**; the orchestrator continues with whatever other batches succeeded and still calls the judge on whatever in-domain items exist (per ADR-0030). Next run retries the un-marked items.
- **Classifier `ClaudeUsageLimitError`** (per ADR-0032): the orchestrator parses the reset time from the API's 429 result text, sleeps until `reset_time + 2 minutes` (or fallback `next_top_of_hour + 2 minutes`), and retries the failed batch. No `degraded_reason`; no Failure Report. Cron-anchored logical day (ADR-0030) handles midnight crossings.
- **Judge non-quota error**: no daily file is written; a **Failure Report** is emitted under `data/failures/<timestamp>.md`; the **Pool** is intact (no `selected_by_judge` transitions happened) and tomorrow's run sees today's pool plus tomorrow's arrivals.
- **Judge `ClaudeUsageLimitError`**: sleep and retry, same as classifier.
- **Per-card append succeeds, then judge response processing fails mid-loop**: each card is appended + fsynced + `mark_selected_by_judge(stub)` called atomically per winner. A failure between cards leaves earlier winners marked, later winners unmarked (back in the pool for tomorrow) — bounded, acceptable.
- **`enrich()` raises `ParserError`** → `mark_enrich_failed(stub)`. Terminal-skip.

State lives in `.seen.json` (synced to the laptop via Syncthing alongside the **Daily Results Files** for durability, per ADR-0002), shaped as `{url: {company_lc, title_lc, location_lc, status, first_seen}}` (any of `company_lc, title_lc, location_lc` may be `None`); the fuzzy index is built in-memory at load, not persisted. `DeduplicationStore` exposes seven narrow methods (`mark_out_of_domain`, `mark_in_domain` — takes the **Structured Extract** as a kwarg, `mark_selected_by_judge`, `mark_expired`, `mark_enrich_failed`, `mark_external_redirect`, plus the first-contact `mark_not_classified`) — one per status value — so each call site is greppable and unambiguous. The pipeline is single-writer at the process level (per ADR-0002): `.seen.json` is mutated by the main thread (and any classify/judge worker threads if pipelined); guarded by a `threading.Lock` inside `DeduplicationStore`. The daily file is single-writer per phase. Parser threads remain pure producers — they yield **Position Stubs** into a queue and never touch dedup or the daily file. **Migration:** per ADR-0033, deploying the new model wipes the previous `.seen.json` and the old trio (`green.md` / `amber.md` / `red.md`); no automatic translation of legacy statuses. _Avoid_: duplicate filtering, URL filtering.

### Display

**Card**: The expanded view rendered for **every** **Daily Top-5** winner. Structure: an H1 headline `# {company} · {title} · {location_segment}` (no tier emoji, no color span); `location_segment` is the literal `location` for on-site or location-only listings (`Hamburg`), `{location} (Hybrid)` / `{location} (Remote)` when a non-on-site `work_model` is known, `Unknown Location` when `location` is `None`, and `Unknown Location (Hybrid|Remote)` for the `None`-location + non-on-site combination. Below the H1: a meta line of `posted_date · contract_type · employment_type` (segments with `None` values omitted; whole line omitted if all three are `None`), a `**Salary:** {salary}` line (omitted when `salary` is `None`), then an `## AI Assessment` section opening with a `**Rank {rank}/5**` line, followed by the **Match Verdict**'s 2–3 sentence `summary` and `**Matched:**` / `**Missing:**` bulleted lists (each list and its label omitted when empty), then a `## Job Description` section carrying the verbatim `raw_description` (whole section omitted when `raw_description == ""`), then a horizontal rule and a bare-autolink URL footer. Headings are English. The `matched` / `missing` lists come directly from the **Match Judge**'s verdict on the structured extract — no second LLM call.

**Renderer**: A pure module exposing `render(position, verdict, layout) -> str`. Always uses the **Layout**'s `CARD_TEMPLATE`. Builds a placeholder dict (raw fields + named placeholder groups + `rank`), substitutes via `str.format_map`, returns the rendered block. No file I/O, deterministic on inputs. No tier-derived placeholders (per ADR-0029). _Avoid_: formatter, presenter.

**Results File Manager**: The only module that reads or writes the **Daily Results File**. Function-shaped: the **Pipeline Orchestrator** holds the day's `Path` (`data/results/{cron_anchored_date}.md`, per ADR-0030) and dispatches. Module surface: `ensure_initialized(path)` (just `mkdir(parents=True)`; no header write — files start empty, no `FILE_HEADER`), `append(path, rendered_block)` (verbatim write + `flush` + `os.fsync`, propagates `OSError` unwrapped). `data_dir` is the parent of the loaded `config.py` (per ADR-0022). _Avoid_: writer, output manager.

### Observability

**Log Artifacts**: The set of files written to `data/logs/` during a run, laid out **by reader** rather than by component (per ADR-0024). Four file types: `<comp>.events.jsonl` (one structured row per step emitted by component `<comp>`; the run-end metrics row that used to live on the retired Run Divider now lands here as `event=run_complete` per ADR-0030); `lifecycle.jsonl` (single shared file carrying status-display `registered` / `phase_changed` / `removed` events); `run.log` (single shared file carrying tracebacks and end-of-run `SUMMARY OF SESSION` blocks); `<comp>.transcripts.jsonl` (LLM components only — full prompt/response payloads). Per ADR-0028, every component identifier carries a layer prefix — `parser_`, `llm_`, or `pipeline_`. For LLM code, **the component is the call site** (`llm_classify_relevance`, `llm_judge_match`) — not the implementation class. _Avoid_: log file (without qualifier — there are four kinds), debug log.

**Run Log**: The single per-run instance that writes the **Log Artifacts** to `data/logs/`. Constructed once at the **Pipeline Orchestrator** entrypoint from `cfg.logs_dir` and threaded as a constructor kwarg into every component that emits log events: `RunMetrics`, `ClaudeExtractor`, the `_StatusDisplay` core, the `_OutboundDispatcher`, and every **Parser** (via the parser's context-manager construction, which forwards the handle into its `ParserHttp`). Owns the directory path, the file-naming convention, the JSONL envelope shape, and the `=== <component>  <ts>  <kind> ===` header convention in `run.log`. Filesystem errors propagate as `OSError`; there is no silent no-op branch. Safe to share across threads (each method opens, writes, and closes its file per call; no shared mutable state on the instance). _Avoid_: log writer (too generic), logger (overloaded with `logging.Logger`).

**Status Display**: A live, in-process view of pipeline progress, sitting alongside the **Log Artifacts** and the **Failure Report**. Ephemeral — exists only while `run()` is executing, exists to answer "what is the pipeline doing right now". A `StatusDisplay` Protocol has two implementations: `RichStatusDisplay` (a `rich.Live` table that repaints in place at ~4 Hz, selected when `sys.stdout.isatty()`) and `PlainStatusDisplay` (line-by-line `print` of register / phase-transition / remove events, selected when stdout is not a TTY — Pi cron). The display surfaces one row per pipeline stage: an overall `pipeline` phase row (top), a transient `startup` phase row, one **Agent Row** per configured **Parser** (keyed by `parser_<parser_type>`), and **Agent Rows** for `pipeline_dedup`, `pipeline_prefilter`, `llm_classify_relevance`, and `llm_judge_match`. Body strings are formatted by **RunMetrics**. `register()` and `update_phase()` also emit a row into `data/logs/lifecycle.jsonl` (per ADR-0024). _Avoid_: progress bar (implies known total), TUI (overloaded), dashboard.

### Sources & extraction

**Source**: A configured job board or API in the **Config**'s `SOURCES`, declared as a `SourceEntry` carrying a **Parser Type** and a per-source `max_results` cap. _Avoid_: site, board.

**Parser**: A Python module in `src/application_pipeline/parsers/` that fetches **Positions** from one **Source**. Two-phase API: `discover(query)` returns cheap **Position Stubs** from list-pages; `enrich(stub)` fetches the detail page and returns a full **Position** conforming to the **Position Schema**. The orchestrator runs **Deduplication** between the phases. Each **Parser** is a context manager owning a single `httpx.Client`; no shared mutable module-level state. The **Pipeline Orchestrator** owns the Cartesian expansion of `KEYWORDS × LOCATIONS [+ remote]` and emits one `discover(ParserQuery(keyword, location, max_results))` call per combination per source; `location` is a sealed **Location** (`City(name) | Remote`). Each parser declares its **Location Coverage** as module-level symbols and dispatches via `parsers.location.resolve()` (per ADR-0014). **Source ↔ Parser is one-to-one**. **Parsers own all per-host pacing**; the orchestrator never sleeps between queries. Parsers may emit `ExternalRedirect` from `enrich`. **Keyword-match invariant** (per ADR-0026): every stub yielded from `discover()` is expected to match its `ParserQuery.keyword` in the title. _Avoid_: scraper, fetcher.

**Position Stub**: The cheap result of `Parser.discover()` — `url, title, company, location, source`. Required: `url, title, source`. Optional (`str | None`): `company, location`. Sources that don't expose one of the optional fields leave it `None`. _Avoid_: preview, summary.

**External Redirect**: A payload emitted by a **Parser**'s `enrich()` instead of a `Position` when the detail page carries an outbound URL to a third-party board **and no usable body to serve**. Carries the original `PositionStub` and the `outbound_url` string. The **Pipeline Orchestrator** marks the stub `status: "external_redirect"` and increments `RunSummary.external_redirects`. Per ADR-0015 / ADR-0028. _Avoid_: redirect stub, forwarded listing.

**Parser Type**: The string on a `SourceEntry` identifying which **Parser** to use — maps directly to a filename in `parsers/`. _Avoid_: adapter.

**Location**: The geographic slot the **Pipeline Orchestrator** passes to a **Parser** via `ParserQuery.location`. A sealed sum type with two arms: `City(name: str)` for a named place from `Config.LOCATIONS`, and `Remote` for the orchestrator's remote-only slot emitted when `INCLUDE_REMOTE=True`. _Avoid_: place, geo (when the field is meant).

**City**: The `City(name)` arm of **Location**. `name` is the user-typed string from `Config.LOCATIONS`, normalized via `normalize()` before lookup. No central catalog — the set of resolvable city names is the dynamic union of configured **Sources**' `serves()` predicates. _Avoid_: town, place.

**Remote**: The `Remote` arm of **Location**. Each **Parser** decides what `Remote` means for its **Source** via `remote_wire()`. _Avoid_: homeoffice, work-from-home (when the type variant is meant).

**Location Coverage**: The module-level **Protocol** every **Parser** module satisfies — `serves(name: str) -> bool`, `to_wire(name: str) -> str`, `serves_remote: bool`, `remote_wire() -> Any`. Each parser is the local authority on its coverage; there is no global city catalog. The **Config Loader** validates `LOCATIONS` at load time and rejects unresolvable entries with `ConfigError`. Per ADR-0014. _Avoid_: location map, slug table.

**LLM Extractor**: A Python interface (Protocol) with two methods — `classify_relevance_batch(items: list[ClassifyItem]) -> (list[RelevanceVerdict], CallUsage)` and `judge_top_n(candidates: list[JudgeCandidate]) -> (list[MatchVerdict], CallUsage)`. `ClassifyItem` carries an opaque string `id`, `title`, and `raw_description`; the response is id-keyed (`[{"id": "...", "in_domain": bool, "extract": {...} | absent}, ...]`). `RelevanceVerdict` is `{in_domain: bool, extract: StructuredExtract | None}` (extract non-None iff in-domain). `JudgeCandidate` carries the stable id, the candidate's **Structured Extract**, and the `Position`'s `stub` fields the judge needs for tie-breaking (title, company, location). The hardcoded prompts live inside the package (`src/application_pipeline/templates/prompts/`). The **Triage Profile** is injected via the prompt's `{USER_INFO}` slot. **Skills** are bound at construction. The v1 production implementation is `ClaudeExtractor`, shelling out to the **Claude Code CLI** (`claude -p - --output-format json --model <alias>` plus `--effort <level>` where applicable) as a fresh subprocess per call. Per ADR-0018 / ADR-0027 the call shape is: `haiku` (no `--effort`) for the **Relevance Classifier**, `haiku` with `--effort medium` for the **Match Judge**. Structured outputs are wrapped in semantic XML tags (`<verdicts>` for both call sites under the new shape) and parsed via the **Agent Output Protocol** (ADR-0018). On `ClaudeUsageLimitError`, the implementation parses the 429 reset time from the result text and raises `ClaudeUsageLimitError(reset_time)` so the orchestrator can sleep (per ADR-0032). Pi-only at runtime. _Avoid_: LLM, model (without qualifier).

**Agent Output Protocol**: A project-agnostic pure module under `application_pipeline/llm/` that extracts a structured JSON payload from a Claude Code CLI response whose `result` field carries free-form text with the payload wrapped in a semantic XML tag (per ADR-0018). Exposes `extract_json_block(text: str, tag: str) -> Any` plus `AgentOutputProtocolError(kind: "tag_missing" | "json_malformed")`. Finds the rightmost `</tag>`, walks backward through preceding `<tag>` openings, strips an optional surrounding markdown fence, then `json.loads`. Returns the first candidate that parses; raises on no tag / no parse. The **LLM Extractor** catches the error at its call-site boundary and re-raises as the existing `ExtractorBatchMalformedError` / `ExtractorMalformedJSONError`. _Avoid_: output parser (too generic), response handler (overloaded with HTTP).

**Pagination**: Successive page fetches per **Source**, preferably ordered newest-first by posted_date, until either the **Source** signals no next page **or** the per-Source `max_results` cap is reached. Per ADR-0020 there is no dedup-driven discover early-stop. `SKIP_AND_END_QUERY` (per ADR-0009) is only emitted on `run_state.is_aborted`. Each `SourceEntry` in the **Config** carries its own cap (default 1000).

### Invocation

The full pipeline runs as a single Python entry point on the Pi: `python -m application_pipeline` (package per `pyproject.toml`, entry via `src/application_pipeline/__main__.py`). On the Pi, cron fires **once per day** (per ADR-0033) — wrapped by `flock -n /var/lock/application-pipeline.lock` (single-writer enforcement) and by the deploy wrapper that handles tag-pull + staging-clone + symlink flip (per ADR-0011). A run that sleeps through a quota window (ADR-0032) may overlap the next cron fire, which `flock -n` silently skips — acceptable; absence of the next day's daily file is the operator signal. The laptop is used only during development to smoke-test individual **Parsers** in isolation (no LLM, no **Deduplication**, no daily file). Note: the `pycastle/` directory in this repo is an unrelated RALPH Loop coding-agent plugin used to *build* this project; it is not this project's Python package, but its quota-handling logic (`pycastle/services/claude_service.py:88-145`) is the reference port for ADR-0032's parse-and-sleep behavior.

## Relationships

- The **Daily Results File** is one dated markdown file per calendar day, containing the **Daily Top-5** **Cards** in **Rank** order. No accumulation across days; no reset semantics (each day is its own file).
- A **Position** belongs to exactly one **Source**, was fetched by exactly one **Parser**, and (if in-domain) has its **Structured Extract** produced by the **Relevance Classifier** at first-contact.
- A **Position** is written into a **Daily Results File** only if it passes the **Domain Pre-Filter**, then the **Freshness Gate** (per ADR-0034), then the **Relevance Classifier** (status flips to `in_domain`), then is picked into the **Daily Top-5** by the **Match Judge** (status flips to `selected_by_judge`). The **Freshness Gate** also re-runs on Pool re-discovery, so pool items can transition `in_domain → expired` before today's judge call.
- The **Pool** is the set of `status == in_domain` URLs that were re-discovered by a parser in the current run. Pool membership is an emergent property of `.seen.json` + today's parse output, not a stored data structure.
- The **Domain Pre-Filter** runs before the **Relevance Classifier** and never bypasses the LLM gate; it only drops listings into `out_of_domain`.
- The **Match Judge** runs once per run on the **Pool**, takes **Structured Extracts** plus construction-bound **Skills** and the injected **Triage Profile**, and returns up to 5 **Match Verdicts** with explicit **Rank**.
- The **Triage Profile** lives in `data/user-info/` and reaches the LLM via the hardcoded prompt's `{USER_INFO}` slot (per ADR-0019); **Skills** are bound at `ClaudeExtractor` construction and reach the LLM via the `{skills}` slot; `NEGATIVE_KEYWORDS` lives in the **Config** and reaches the **Domain Pre-Filter** directly (per ADR-0026).

## Flagged ambiguities

- **"Filter"** — used to mean two different things: the **Domain Pre-Filter** (deterministic drop) and the **Relevance Classifier** (LLM-based discard). These are distinct stages; never use the bare word "filter".
- **"Match Tier"** — retired. The judge picks the **Daily Top-5** by **Rank**; there is no tier enum. See [ADR-0029](docs/adr/0029-match-tier-retired-judge-picks-top-n.md). The old [ADR-0003](docs/adr/0003-match-tier-judged-by-llm.md) is fully superseded.
- **"Match Score"** — never existed in v1; not to be confused with **Rank**. **Rank** is an ordering within a single day's **Daily Top-5**, not a numerical comparability across days.
- **"Profile"** — disambiguate. The v1 **Triage Profile** (prose, lives in `data/user-info/`, injected into the prompts' `{USER_INFO}` slot, drives LLM judgment) is distinct from the Phase-2 **CV Profile** (structured, used to generate application material). Never use "profile" without a qualifier.
- **"Summary" vs "Raw Description" vs "Structured Extract"** — three things, often confused: `summary` is the 2–3 sentence judge-authored line on the **Match Verdict**, written for the **Card**; `raw_description` is the full original text from the **Parser**, rendered verbatim into the **Card**'s `## Job Description` section; **Structured Extract** is the classifier-authored, JSON-fielded form of the listing that the **Match Judge** ranks on. The judge never sees `raw_description`. The classifier never sees the `summary`. Use the precise term every time.
- **"Source" vs "Parser"** — a **Source** is a config entry (what to search); a **Parser** is the module (how). One-to-one in v1.
- **"Pool" vs "Daily Top-5"** — the **Pool** is the eligible set entering the judge; the **Daily Top-5** is the selected output. Never call the input "the top-5" or the output "the pool".
