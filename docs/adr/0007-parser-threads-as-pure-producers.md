# Parser threads are pure producers; main thread owns all writes

The **Pipeline Orchestrator** runs each **Parser** instance on its own thread (one per `parser_type`) so that three sources can fetch concurrently without contending on a single host's pacing budget. Within a parser thread everything stays sequential (per-host 0.5 s pacing). Across parser threads, work is parallel.

The threads are **pure producers**: the parser thread runs `discover()` *and* `enrich()`, but it never touches **Deduplication** state or the **Results File**. It pushes **Position Stubs** onto an outbound `queue.Queue` and blocks on an inbound `queue.Queue` for the orchestrator's "enrich this stub / skip it" decision. Only the orchestrator's main thread reads `.seen.json`, writes `.seen.json`, runs the **Relevance Classifier** and **Match Judge**, renders, appends to `results/current.md`, and calls `mark_seen`.

## Why

- **Per-host pacing stays inside the parser.** Each parser owns one `httpx.Client` and one `_throttle()` helper. Letting only one thread (the parser's own) fetch from that source means the 0.5 s sleep is never raced. No locks, no shared rate-limiter, no module-level mutable state.
- **Network and LLM are different bottlenecks.** Network on Pi LAN is fast (~50–500 ms per request). Ollama on Pi 5 is slow (several seconds to tens of seconds per call). Putting LLM work on the main thread keeps it serial (one Ollama call at a time — the chosen Qwen 3 models do not parallelize on this hardware) while parser threads keep prefetching list pages and detail pages in the background. The two pipelines run at their natural rates without one starving the other.
- **The "single-writer at the file level" invariant becomes literal.** `.seen.json` and `results/current.md` are touched by exactly one thread (the main thread). No `threading.Lock` around `DeduplicationStore.is_seen` / `mark_seen`. No interleaved appends to `results/current.md`. Per ADR-0002, the pipeline is single-writer at the process level; this ADR sharpens it to the thread level without adding lock surface.
- **Parser interface stays unaware of threading.** The `Parser` Protocol exposes `discover(query)` and `enrich(stub)`. Neither method takes a queue, a callback, or a dedup-state argument. The standalone `python -m application_pipeline.parsers.<name>` dev script calls both methods on the main thread with no orchestrator at all. Threading is purely an orchestrator-side concern.
- **No shared mutable state means no thread-safety burden inside parsers.** Each parser instance is self-contained; module-level constants are immutable; the `httpx.Client` is per-instance. The `Parser` Protocol contract is "instances must be safe to use from a single dedicated thread for the lifetime of the orchestrator run." Stronger guarantees (multi-thread re-entrant) are not required.

## Considered alternatives

- **Each parser thread does its own dedup checks and Results File writes, protected by `threading.Lock`s.** Rejected: spreads the locking surface across three parsers plus the dedup store plus the Results File Manager. Easy to miss a lock and corrupt `.seen.json` or interleave Results File writes. The producer/consumer split eliminates the lock surface entirely.
- **Run enrich on the main thread; parser threads only do `discover()`.** Rejected: the main thread would block on every detail fetch (~50 unseen stubs × 0.5 s pacing × 3 sources = ~75 sequential seconds per run). All three parser threads would idle while the main thread sleeps on a single host. Also forces per-host pacing state to leak out of the parser into a main-thread-callable form — breaks "no shared mutable module-level state."
- **Run enrich on a `ThreadPoolExecutor` per source.** Rejected for v1: parallel detail fetches within one source violate per-host pacing without adding back the shared rate-limiter we deliberately removed. If a source one day proves it can take parallel detail-fetches, that source's parser can manage its own internal pool — without changing the orchestrator's single-thread-per-parser contract.
- **`asyncio` end-to-end with `httpx.AsyncClient`.** Rejected per the parsers PRD's round-1 resolution (sync httpx today; door open later). An async rewrite is a one-time large change that pays off when the workload is I/O-bound and the I/O is fan-outable. With one Ollama call serializing on the main path and per-host pacing serializing per parser, fan-out is bounded — async would re-platform the codebase for marginal gain.
- **Single-threaded fully sequential pipeline.** Rejected: three sources serial on per-host pacing × Cartesian product of keywords × locations is wall-clock-painful even on a personal pipeline. Concurrency across sources is the cheap win.

## Consequences

- The orchestrator (out of scope for the parsers PRD #19; deferred to a follow-on issue) gains a producer/consumer skeleton built on `queue.Queue` and `threading.Thread`. One thread per `parser_type` plus the main thread.
- The `Parser` Protocol contract pins **"no shared mutable module-level state"** as a binding rule. Parser modules may have module-level *constants* (e.g., `_DISPLAY_NAME`) and module-level pure functions implementing the **Location Coverage** protocol (`serves`, `to_wire`, `serves_remote`, `remote_wire` — per ADR-0014) but no caches, no global counters, no module-level `httpx.Client`.
- `DeduplicationStore` and the Results File Manager keep their current single-thread design — no `threading.Lock`, no atomicity helpers beyond the existing `os.fsync` pattern.
- `Config` does not gain a `MAX_PARSER_CONCURRENCY` knob. The number of parser threads equals the number of distinct `parser_type` strings in `Config.SOURCES` — bounded naturally by the number of registered parsers.
- The `python -m application_pipeline.parsers.<name>` dev script remains useful and unaffected — it exercises a parser on the main thread without the orchestrator, perfect for fixture re-capture and manual drift checks (the live counterpart to the offline-by-default test suite).
- Back-pressure between threads is implicit: the parser thread pushes one stub then blocks on the dedup-decision queue. The main thread is always one stub ahead of the parser, never thousands. No bounded queue tuning required.
