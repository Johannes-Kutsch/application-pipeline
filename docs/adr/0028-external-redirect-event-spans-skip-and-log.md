# `external_redirect` event spans skip-and-log

The `external_redirect` row in `<parser>.events.jsonl` carries a `skipped: bool` field and is emitted for two distinct outcomes from the same call site:

- `skipped=true` — the parser detected an off-domain outbound URL on a detail page with **no usable body**, returned `ExternalRedirect(stub, outbound)` to the orchestrator, the orchestrator marked the stub `external_redirect` in the **Deduplication Store**, and bumped `RunSummary.external_redirects`. This is the existing ADR-0015 path, originally for `jobs_beim_staat_html`.
- `skipped=false` — the parser detected an off-domain outbound URL on a detail page that **does carry usable body text**, recorded the outbound URL via `parser_log.record`, and returned a normal `Position` for downstream processing. Dedup state is whatever the rest of the pipeline writes; `RunSummary.external_redirects` is **not** bumped.

The first parser to exercise the `skipped=false` path is `bundesagentur_api`, whose `enrich()` response can carry an `externeURL` field alongside a non-empty `stellenangebotsBeschreibung` (~10% of items in a representative Python/Hamburg sample). `jobs_beim_staat_html` continues to emit only `skipped=true` (its detection point is the absence-of-body wrapper page; the keep-and-log case does not arise for it).

## Why

- **The thing we want to evaluate is the population of outbound hosts, not the population of skips.** Issue #268 was framed as "store all redirections parsers come across so the maintainer can evaluate what makes sense to implement [next as its own parser]." That evaluation is `jq '.outbound' <parser>.events.jsonl | hostname-extract | sort | uniq -c`. Splitting the population across two event names (`external_redirect` vs a separate `external_partner_link`) means every cross-host query becomes a union of two streams, with no analysis upside — the partitioning is by *our* downstream action, not by anything intrinsic to the outbound URL. Single event name + `skipped` field keeps the dominant query one-liner.
- **The "skip" outcome and the "keep" outcome are different decisions even though they share a signal.** For `jobs_beim_staat_html` the detail page is a wrapper around a third-party post — there is no `Position` to build, so we skip. For `bundesagentur_api` the API often returns *both* an `externeURL` and a real German job description — there is a `Position` to build, and skipping it would silently drop ~10% of real Bundesagentur listings the user would otherwise see. Reusing `ExternalRedirect` (the sticky-skip payload) for both would require either (a) accepting the lost listings or (b) overloading `ExternalRedirect` to mean "maybe skip, maybe not" and pushing the policy onto the orchestrator. Splitting on body presence at the parser keeps the payload semantics clean: `ExternalRedirect` means "nothing to enrich here, sticky-skip me."
- **`RunSummary.external_redirects` stays narrowly defined.** The counter is the operator-visible answer to "how many listings did we decline to enrich into the pipeline because they were wrappers?" Bumping it for the keep-and-log case would change that answer to "how many externeURL signals did we encounter?", which is what the events log already answers more precisely. The Run Divider should report decisions, not encounter counts.
- **Detection point per parser is shape-driven, not policy-driven.** `jobs_beim_staat_html` detects at `enrich()` (the wrapper shape is only visible on the detail page). `bundesagentur_api` *could* detect at `discover()` (the `externeURL` field is on search items too), but the skip decision requires the body, which is only available after `enrich()`. So Bundesagentur emits its `external_redirect` event at `enrich()` time regardless of branch — both `skipped=true` and `skipped=false` are emitted from the same code site, with the body-emptiness predicate (`strip_html(...).strip() == ""`) selecting the branch.
- **The `skipped` boolean is a stable discriminator on disk.** Once `bundesagentur_api.events.jsonl` rows accumulate on the Pi, renaming or restructuring this event becomes painful (laptop-side analysis scripts grow against the field shape). Locking the shape now — `event="external_redirect"`, fields `stub_url`, `outbound`, `skipped` — avoids a later migration.

## Considered alternatives

- **Two distinct event names** (`external_redirect` for the skip path, `external_partner_link` for the keep path). Rejected. Cleaner type-level separation, but every cross-host analysis becomes a union of two streams, and the partitioning is by our downstream action rather than by anything the outbound URL itself encodes. The dominant query is "what hosts do we see, period" — that should be one event, one filter.
- **Skip every `externeURL`-bearing Bundesagentur listing** (apply jobs-beim-staat's policy symmetrically). Rejected. Throws away ~10% of real Bundesagentur listings that have usable German body text in the same payload as the outbound link. The savings are imaginary — these listings are real positions, not wrappers.
- **Keep every `externeURL`-bearing Bundesagentur listing and forget the outbound URL.** Rejected — this is the status quo, and issue #268's whole point is to *collect* outbound URLs for future parser-prioritisation analysis.
- **Bump `RunSummary.external_redirects` on the keep-and-log path too.** Rejected on definitional grounds — see "Why" above. The counter exists to report decisions, not encounter counts; encounter counts are already answered by the events file.
- **Add a parallel `RunSummary.external_partner_seen` counter for the keep case.** Rejected as premature. Until the operator actually wants a Run Divider number for "encountered-but-kept" populations, the events log is the right venue. Adding a new field to the Run Divider locks an on-disk schema for a use case that hasn't been validated yet.
- **Pre-skip `externeURL`-bearing Bundesagentur stubs at `discover()` to avoid the detail fetch.** Rejected — the skip decision requires the body length, which requires the detail fetch. We could pre-log the outbound URL at discover-time as a separate event, but that doubles the on-disk noise (one row for "saw it" and one row for "decided about it") for no analytic upside. One event per stub, emitted at the decision point, is enough.
- **Generalise to `stellen_hamburg_api` as well.** Rejected on shape — surveying real responses (Python/Hamburg keyword sweep across several broad terms) found 100% on-domain `PositionURI` values, no `JobPosting.url` divergence, and external hrefs only appearing inside body text (not as wrapper markers). Adding detection code here would be code that never fires. The "generalise to other parsers" framing of #268 narrows to Bundesagentur only.

## Consequences

- **`bundesagentur_api.enrich` adds `externeURL` detection.** After parsing the detail response, the parser computes `outbound = data.get("externeURL")` and `body = strip_html(data.get("stellenangebotsBeschreibung") or "").strip()`. If `outbound` is non-empty:
  - If `body == ""`: `parser_log.record("bundesagentur_api", "external_redirect", stub_url=stub.url, outbound=outbound, skipped=True)`; return `ExternalRedirect(stub, outbound)`.
  - Else: `parser_log.record("bundesagentur_api", "external_redirect", stub_url=stub.url, outbound=outbound, skipped=False)`; continue to build and return the normal `Position`.
  If `outbound` is empty, no event is emitted — current behaviour unchanged.
- **`jobs_beim_staat_html` adds the `skipped=True` field to its existing event call** so the event shape is uniform across parsers. No behavioural change — the parser still skips every detected case.
- **No new payload types, no orchestrator changes, no dedup-schema changes.** The orchestrator's existing `_handle_external_redirect` path serves the `skipped=true` case unchanged; the `skipped=false` case never reaches the orchestrator (a normal `Position` does).
- **`RunSummary.external_redirects` is unchanged.** It continues to count `ExternalRedirect` payloads only — i.e. the `skipped=true` population.
- **CONTEXT.md updates.**
  - **External Redirect** entry: clarify that the `ExternalRedirect` payload is emitted only when the parser determines there is no usable body to serve. A separate `external_redirect` event with `skipped=false` may also be emitted by a parser that detected an outbound URL but still produced a normal `Position`; this event is for outbound-host analysis and does not affect dedup state or `RunSummary.external_redirects`.
  - **Position Schema** entry: no change.
  - **Log Artifacts** entry: no schema change; the `<comp>.events.jsonl` semantics already allow free-form fields per event.
- **ADR-0015 stands.** Its jobs-beim-staat-specific detection, payload-shape, and orchestrator-flow consequences are unchanged. This ADR extends the meaning of the `external_redirect` *event name* (which ADR-0015 introduced) to a second outcome without altering the `ExternalRedirect` payload's semantics.
- **Smoke-test guard for `externeURL`.** A new smoke test (marked `smoke`, excluded from default runs per the project's offline-first test policy) hits the Bundesagentur API and asserts that at least one `externeURL`-bearing item appears in a broad search; this is the canary that catches Arbeitsagentur dropping or renaming the field. Default-run tests use fixture payloads.
- **Follow-up out of scope here.** Picking which third-party host to implement as its own parser (the original motivation for #268) is downstream analysis work — a future ADR or issue triggered by the accumulated `external_redirect` rows, not a consequence of this one.
