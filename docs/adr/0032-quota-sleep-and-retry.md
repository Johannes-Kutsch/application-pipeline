# Quota handling: parse Anthropic's stated reset time and sleep through it

When the Claude CLI returns `api_error_status: 429` (usage limit), the pipeline parses the human-readable reset time from the response's `result` text, sleeps until `reset_time + 2 minutes`, and retries. If the reset time cannot be parsed (or the `result` text is missing), it falls back to **next top of hour + 2 minutes**. There is no retry budget cap — the pipeline sleeps as long as the API tells it to. On wake, all held LLM calls retry; on a second quota hit, the same parse-and-sleep loop fires again. The previous policy (ADR-0017's "LLM workers degrade to no-ops; run ends successfully with `degraded_reason='usage_limit'`") is superseded.

This ADR supersedes ADR-0017's degraded-success exit clause. The classify-pipelined-with-parsers half of ADR-0017 may still apply at implementation time; the failure-on-quota half is replaced. Companion ADRs: ADR-0029, ADR-0030, ADR-0031, ADR-0033.

## Why

- **Degrading to no-ops loses data on the day the user most wants output.** Under ADR-0017, a quota hit on a busy day meant *no* daily file. The applicant had no signal except a `degraded_reason` field that they would not see (it lives on the Run Divider, which ADR-0030 also retires). For a v1 pipeline whose entire purpose is to surface 5 listings a day, "skip today silently" is a much worse failure mode than "today's file is delayed a few hours."
- **Anthropic's 429 payload already carries the reset time.** The CLI's JSON envelope on a quota error includes a `result` text like `"...your usage limit resets May 20, 3pm (UTC)"` or `"...resets 3pm (UTC)"` (no date when same-day). The local `pycastle/` plugin already parses this format and sleeps until that time + 2 min buffer. The parsing logic and regex are battle-tested in pycastle and directly portable — same wire format, same buffer, same fallback.
- **The 2-minute buffer is the operationally-validated lag.** pycastle's `claude_service.py:50-59` adds `+2min` because Anthropic's quota counters take a beat to actually open the window after the stated time. Anything shorter risks the immediate-retry hitting the same 429.
- **No retry budget cap is the right v1 trade-off.** A cap (Q10 (ii) 6h, (iii) N retries) would mean the pipeline can give up while the quota is still rolling shut, ending with a Failure Report instead of a daily file. The applicant's stated preference is "delayed file" over "no file with explanation." If quota outages routinely exceed 24h, the cron-overlap behavior (the sleeping run blocks the next cron via `flock -n`) will surface that operationally — at which point this ADR can be revisited with real data.
- **Single-account is sufficient for v1.** pycastle supports a multi-credential pool (`_AccountPool` in `claude_service.py:31-78`) so an exhausted account can be skipped in favor of a fresh one. application-pipeline runs against one Claude CLI install with one bound account; the pool degenerates to a single account. Porting the pool API but with one entry is simple enough that adding more accounts later is a config change, not a refactor.

## Considered alternatives

- **Keep ADR-0017's degrade-to-no-op exit.** Rejected for the reason above — the daily-file model amplifies the cost of skipping a day. Under the previous trio-and-multi-fire model, a degraded run still left `kept` items in the file from before the quota hit; under one-fire-per-day, a degraded run leaves nothing.
- **Cap retry budget at 6 hours; write Failure Report on cap.** Considered (Q10 (ii)). Rejected: arbitrary number, no data to back it. The cron-overlap behavior gives a natural operational signal if outages exceed a day, and the Failure Report path still fires on non-quota errors. Adding a cap solely for quota would be a hedge against a problem that has not been observed.
- **Sleep a fixed interval (e.g. 15 min) and retry, regardless of stated reset.** Rejected: would churn on Anthropic's actual reset cycle and waste both wall-clock time (sleeping longer than needed if reset is sooner) and tokens (retrying earlier than needed and hitting 429 again).
- **Fail-fast on quota, let cron retry tomorrow.** Rejected: cron is once-per-day (ADR-0033), so "retry tomorrow" means 24h skip. The whole reason to handle quota inline is to fit the daily output into a quota-aware schedule the operator can't tune more finely than once-per-day.
- **Multi-account pool from day one.** Rejected (Q10 single-account): the pool surface is only useful if the operator has multiple Claude subscriptions. For a hobbyist single-applicant pipeline that's unlikely; designing for it speculatively adds config surface (`Config.CLAUDE_ACCOUNTS: list[tuple[str, str]]`) for no current need.

## Consequences

- **New module `src/application_pipeline/llm/quota.py`** (or equivalent) holds the 429 result-text parser. The regex and month table are direct ports from `pycastle/services/claude_service.py:88-145` — same Anthropic wire format. `parse_reset_time(result_text: str) -> datetime | None`.
- **`ClaudeUsageLimitError`** carries the parsed `reset_time: datetime | None`. The `LLMExtractor` implementation raises it from `classify_relevance_batch` and `judge_top_n` when the CLI exits with 429.
- **Sleep happens at the orchestrator level**, not inside `ClaudeExtractor`. The orchestrator catches `ClaudeUsageLimitError`, computes `wake_time = (reset_time or _next_top_of_hour(now)) + timedelta(minutes=2)`, calls `time.sleep((wake_time - now).total_seconds())`, and re-issues the failed call. This mirrors `pycastle/orchestrator.py:278-282`.
- **The classifier batching model interacts cleanly with sleep.** When a classify batch raises `ClaudeUsageLimitError`, the orchestrator holds *that batch's items unmarked* (per ADR-0017's per-batch-failure rule, which stays), sleeps, then retries the same batch. Pipelined classify workers (ADR-0017) may stay or be serialized — implementation detail.
- **The judge call retries the same way.** On 429 from the single end-of-run judge call, the orchestrator sleeps and re-issues with the same `JudgeCandidate` list (ADR-0031). The pool is stable during sleep (no new parsing happens), so the retry input is deterministic.
- **Non-quota LLM errors (timeouts, malformed JSON, schema overrun, ClaudeCliNotFoundError) are unchanged.** They do not trigger sleep; classifier batch errors fail that batch and leave items unmarked for tomorrow; judge errors trigger Failure Report and no daily file.
- **`degraded_reason` field is removed entirely.** No runs end "successfully degraded" — they either complete (with however many cards the pool produced) or fail (with a Failure Report). The Run Divider that carried `degraded_reason` is also retired by ADR-0030.
- **Cron-anchored logical day (ADR-0030) handles midnight crossings.** A run started by the 02:00 May 18 cron that sleeps until 09:00 May 18 (or worse, 03:00 May 19) still writes to `data/results/2026-05-18.md`.
- **Cron overlap is accepted.** `flock -n /var/lock/application-pipeline.lock` (existing) means a sleeping run blocks the next cron fire, which is then silently skipped. An outage exceeding the cron interval produces "yesterday missing, today present" — an acceptable operator signal for an unbounded sleep policy.
- **Sleep duration is logged.** `event=quota_sleep, reset_time=..., wake_time=..., duration_s=...` to `pipeline_orchestrator.events.jsonl` (ADR-0024) so the operator can grep for "how often was the pipeline blocked on quota and for how long."
- **Multi-account pool deferred.** When the operator adds a second Claude subscription, lift the pool from pycastle in a follow-up ADR.
