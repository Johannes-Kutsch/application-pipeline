# Skills as a grouped, dual-consumed pool

`user-info/search-terms/skills.md` is the single source of truth for skill names, dual-consumed by the pipeline (`SearchTerms.SKILLS` → judge prompt's `{skills}` slot) and by `/write-cv` (CV's skills section). The file gains structure on top of ADR-0032's flat-bullet shape: H2 headings are skill **groups** (also the `\cvitem{<group>}{...}` category label in the rendered CV), bullets are atomic skills, pandoc-style `{...}` attributes annotate both. Groups carry `always` (always rendered) and per-jobtype `relevance` (`games=high, mle=low`, …); items carry `always` only. The pipeline loader ignores headings + attributes and harvests every bullet body as a flat list — backward-compatible with ADR-0032's parse contract. `/write-cv` reads the full structure: always-groups always render; the LLM picks zero-or-more additional groups by relevance against the listing's jobtype; within an included group every `always` item renders plus zero-or-more LLM-picked items; excluded groups contribute nothing (item-`always` is a within-group floor, not a group-promoting trigger). Output order matches file order. The `skills_block` slot in **CV Slot-Map** stays — `/write-cv` mechanically assembles `\cvitem{<group>}{<comma-joined skills>}` lines into the slot body from the pool; the LLM never freehand-authors skills.

The primary motivation is factual consistency: bounding what Claude can claim about the applicant. Resume sections (`Berufserfahrung` / `Ausbildung` / `Projekte`) already use the **Content Pool** for the same reason; `skills_block` was the last surface where the LLM could invent items (e.g. "Rust (advanced)" when only basics are held). Relevance-weighted selection is a secondary benefit that falls out of the same shape. The pipeline already needs the flat list for the judge's `{skills}` prompt slot; reusing the same file removes the duplicate-list maintenance burden that would otherwise grow.

## Considered alternatives

- **Whole `\cvitem` rows as pool items** (parallel to existing resume pool granularity: one macro = one renderable row) — rejected: skill *emphasis* varies per listing, so freezing whole rows means authoring N variants per group (`\skillsLanguagesML` vs `\skillsLanguagesBackend`). Atomic skills with group composition at write time express the same variation with one row per group + per-item `always` flags.
- **Item-level `relevance:`** — rejected: groups are the unit of LLM selection, so relevance at item level is unused. If items inside a group diverge in relevance enough to need per-item scoring, that's the signal to split them into two groups.
- **Two files: flat `skills.md` for the pipeline + a sibling TeX pool for `/write-cv`** — rejected: the explicit goal of #473 is single-source-of-truth. Two files re-introduces drift that the issue exists to fix.
- **Skills live inside `content_pool.tex`** as a fourth section (`% ===== Skills =====`) — rejected: that file lives under `user-info/cv/` and the pipeline doesn't read TeX. Skills live with the other pipeline-readable search terms in `user-info/search-terms/`, where the pipeline already reads.
- **Retire the `skills_block` slot; `cv_template.tex` reads `skills.md` directly at compile time** — rejected: breaks the invariant that `cv.tex` is the complete per-listing artifact (it would no longer fully describe its own output), and forces `cv_template.tex` to grow a markdown parser.
- **`skills_choice.txt` sidecar carrying only the LLM's group picks; assembly at compile time** — rejected: same invariant-break as above plus a second per-listing artifact for the operator to track.
- **Item-`always` promotes its group into the selection** — rejected: tangles the two layers. If a skill must always ship, mark its group `always`. Keeping item-`always` as a within-group floor makes the selection rules independent across the two layers.

## Consequences

- Supersedes in part **ADR-0032**: `skills.md` body shape is no longer "flat `-` bullets only" — it gains H2 group headings and `{...}` attributes. The pipeline loader's contract (harvest bullet bodies, treat the file as a flat list) is preserved; headings and attributes are ignored at parse time.
- Amends **ADR-0030**: `skills_block` slot semantics shift from free-authored to mechanically assembled by `/write-cv`. The slot still appears in `cv.tex` with its rendered `\cvitem` rows verbatim — the artifact remains self-contained.
- The `cv_skeleton.tex` `skills_block` example is replaced with an instructional comment describing the assembly behaviour; no hand-authored example.
- CONTEXT.md updates: **Skill**, **SearchTerms**, **CV Slot-Map** entries reflect the structured-file shape and the assembly path; a new **Skill Group** entry names the unit of LLM selection.
- No migration code. Operators relocating to ADR-0032's layout add the H2 group structure to their new `skills.md` by hand. Pre-existing flat `skills.md` files (zero groups) parse fine — every bullet becomes an ungrouped skill, which is degenerate for `/write-cv` (no `\cvitem` row has a category) but valid for the pipeline. The operator-visible signal is an empty skills section in the rendered CV.
