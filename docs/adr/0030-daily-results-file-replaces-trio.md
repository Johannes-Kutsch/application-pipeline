# Daily results file replaces the green/amber/red trio; FILE_HEADER and Run Divider retired

The **Results File** stops being a rolling trio (`green.md` / `amber.md` / `red.md`) that accumulates across runs. It becomes one dated markdown file per calendar day under `data/results/YYYY-MM-DD.md`, holding exactly the **Daily Top-5** **Cards** (or fewer if the run had fewer winners). The hardcoded **FILE_HEADER** is dropped — daily files have no preamble, just the **Cards** in **Rank** order. The **Run Divider** is dropped — daily-file presence-or-absence is the heartbeat ("yesterday's file is missing" is the operator signal), and metrics that used to live in the divider move to `pipeline_orchestrator.events.jsonl` per ADR-0024.

This ADR supersedes the trio shape and the per-tier reset semantics established by the previous Results File language in `CONTEXT.md`. The companion ADRs are ADR-0029 (tier retired), ADR-0031 (extracts and pool), ADR-0032 (quota handling), and ADR-0033 (one run per day).

## Why

- **A dated file per day is the smallest artifact that answers the applicant's question.** The applicant's daily question is "what are today's best 5 matches?" — a dated file containing exactly that and nothing else is the honest shape. The previous trio answered a different question ("everything ever judged, bucketed by tier") that the applicant did not actually ask.
- **Per-tier reset semantics were a workaround for an accumulating-file problem that no longer exists.** The trio's reset story ("delete `green.md` to reset only green; `.seen.json` is unaffected") existed because the file grew unboundedly and the user occasionally wanted a fresh start. Daily files self-reset by date — yesterday's file is read once and then ignored; reset is implicit. The reset-hint half of `FILE_HEADER` goes with it.
- **The Run Divider was a multi-fire-per-day heartbeat.** It existed because cron fired several times per day and the operator needed to tell "this run crashed" from "this run wrote nothing because the pool was thin." Under one run per day (ADR-0033), daily-file presence answers the same question with less surface: file exists → run completed; file absent → run failed or sleeping. The metrics the divider carried (`classify_*`, `judge_*`, `dedup_*` counts, timing) belong in operator-only artifacts (`pipeline_orchestrator.events.jsonl`); they were never read by the applicant.
- **Schema-version marker was speculative.** The other half of `FILE_HEADER` was a schema-version comment for a future format change that has not happened in v1 and is unlikely to land before another major refactor. Dropping it now and re-adding it later costs nothing.
- **Syncthing semantics simplify.** The trio required the laptop-as-read-only-consumer convention plus per-file sync-conflict handling. Daily files are write-once on the Pi and never re-touched after the day's run completes — the conflict surface is much narrower. The `*.sync-conflict-*.md` warning still applies but is now a near-zero event.

## Considered alternatives

- **Single rolling `top5.md` that gets overwritten daily.** Rejected: loses history. The applicant who skips a day or wants to compare "yesterday's top 5 vs today's" cannot. Dated files give cheap history for free.
- **Daily file plus a permanent `archive.md` that accumulates everything.** Rejected: dated files *are* the archive. Nothing in v1 needs a single concatenated view; if a future analysis script wants one, it can `cat data/results/*.md`.
- **Keep the Run Divider in the daily file as an end-of-day metadata block.** Rejected: it conflates the applicant-facing artifact with the operator-facing artifact. The two have different readers, different update cadences, and different reasons-for-being. Splitting cleanly — metrics to `events.jsonl`, cards to `<date>.md` — is worth the small extra surface.
- **Hardcode a per-day file header carrying the date and pool-size summary (e.g. "Judged 47 candidates, top 5 below").** Rejected: the filename is the date already; the pool-size info belongs in operator logs, not in the applicant-facing artifact. Adding it would re-create the reason `FILE_HEADER` originally existed — to carry operator metadata into the user-facing file — which is the inversion this ADR is unwinding.
- **Date by wall-clock-at-write time rather than cron-anchored logical day.** Rejected: see ADR-0032 — a run that crosses midnight (e.g. starts at 23:30, sleeps through a quota window, finishes at 03:00) needs an unambiguous "what day is this file" rule. Cron-anchored (start-of-run date) gives one file per cron fire; wall-clock-at-write makes file-naming dependent on sleep duration, which makes the next morning's cron unclear about whether to write or skip.

## Consequences

- **Results File Manager** (`src/application_pipeline/results/manager.py`): the trio-path machinery is removed. Module surface becomes `ensure_initialized(path)` (just `mkdir parents=True`; no header write — files start empty) and `append(path, rendered_block)` (unchanged: write + flush + fsync). Path resolution: `data_dir / "results" / f"{cron_anchored_date}.md"`.
- **FILE_HEADER** constant is removed from the package. No replacement.
- **Run Divider** rendering is removed. The orchestrator's end-of-run metric emission moves to `pipeline_orchestrator.events.jsonl` (ADR-0024) as a single `event=run_complete` row with the fields the divider used to carry (`classify_calls`, `classify_input_tokens`, `judge_input_tokens`, `dedup_*`, `elapsed_s`, ...). `degraded_reason` is no longer needed (ADR-0032 removes the degraded-success exit).
- **Cron-anchored logical day** is the source of truth for the daily file's date. The orchestrator captures `run_started_at` at entry and derives `date_for_file = run_started_at.date()`; everything writes to the same path regardless of how long the run takes or whether it crosses midnight.
- **`<5` candidates:** if the pool has fewer than 5 in-domain items, the daily file carries however many cards exist. **`=0`:** no file written. No empty file. No placeholder card. Tomorrow's run is unaffected.
- **Layout** loses `FILE_HEADER`-related concerns (there were none — `FILE_HEADER` was always hardcoded inside the package). The `CARD_TEMPLATE` placeholder set shrinks per ADR-0029.
- **`CONTEXT.md` updates:** Results File entry rewrites; Run Divider entry deletes; Match Tier entry deletes; Card entry simplifies (no more tier-aware Layout); new entries for **Daily Results File**, **Pool**, **Daily Top-5**, **Rank**.
- **Failure semantics:** judge call failure (any cause including quota that exceeded the patience of ADR-0032) → no daily file is written → a **Failure Report** lands in `data/failures/<timestamp>.md` (ADR-0012, unchanged). Classifier partial failure → run continues to judge on whatever classified successfully (per ADR-0029's single-judge-call model).
- **Migration:** the existing trio (`green.md` / `amber.md` / `red.md`) and `.seen.json` are wiped on first new deployment (per ADR-0033). No automatic conversion. The applicant accepts that historical kept-records do not survive the cutover — they were not load-bearing.
