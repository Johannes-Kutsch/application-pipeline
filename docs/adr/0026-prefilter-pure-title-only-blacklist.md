# Domain Pre-Filter becomes a pure title-only blacklist

The **Domain Pre-Filter** is reduced to a single rule: drop a **Position** if any **Negative Keyword** matches its title (case-insensitive substring, after `normalize()`). The whitelist mechanism is removed entirely — there is no rescue via `SKILLS` or `INCLUSION_KEYWORDS`, no body-text matching on either side. `INCLUSION_KEYWORDS` is retired from the **Config** schema; legacy values in user configs are silently ignored with one INFO log line. `SKILLS` keeps its remaining role (the `{skills}` slot in the **Match Judge** prompt) and stops being a prefilter input. This ADR supersedes ADR-0005 (`0005-domain-prefilter-gates-llm.md`).

## Why

- **The whitelist became redundant.** ADR-0005 designed the Pre-Filter as a volume reducer against the firehose of "all listings on a source". In practice every parser in v1 queries its source with `Config.KEYWORDS`, so every listing reaching the Pre-Filter has already passed an upstream keyword filter. The whitelist was filtering an already-filtered stream — a tautology layer.
- **Empirical confirmation.** Over the last 1800 Pre-Filter decisions (`synched/logs/prefilter.events.jsonl`): `whitelist_only=576 (32%)` and `whitelist_rescue=372 (21%)` together meant the whitelist fired on 53% of decisions. The pathology was visible directly in the log: a "Learning Experience Designer" listing passed via `whitelist_only` because the body contained the substring `git`. The whitelist was firing on incidental tech-stack mentions in body text, not on genuine in-domain signals.
- **Body-scope matching was the proximate cause.** Both whitelist and blacklist matched against `title + raw_description`. Body matches generated most of the false rescues (the `git`-in-body case) and would also have generated most of the false drops if the blacklist were strengthened. Title is high-signal: a real SWE role rarely has a Negative Keyword in its title. Restricting both lists to title-only collapses the false-positive surface on both sides.
- **The Pre-Filter should not redo the judge's job.** ADR-0005's framing — "the LLM stays the authority on in-domain" — implied the Pre-Filter is a cheap, high-precision pre-cut. The whitelist drift turned it into a semantic gate that competed with `classify_relevance` for the same decision. The cleaner shape is: title-blacklist for unambiguous off-domain listings ("Sales Manager", "Werkstudent"), nothing else. Body text is the judge's job.
- **Cost is a real concern but not the dominant lever.** Per the token-cost grilling session (issue #319), the dominant cost lever is moving the **Match Judge** off Sonnet (see ADR-0027); the Pre-Filter cleanup adds a small additional saving (~$0.14/run estimated) on top. The justification for this ADR is primarily architectural — semantic drift, dead config surface, dual-consumer cleanup of `SKILLS` — not raw cost.
- **`SKILLS` stops being a dual-consumer.** `SKILLS` was both the judge prompt's `{skills}` slot *and* part of the prefilter whitelist. The dual role meant any prefilter-side decision to drop a generic skill (`python`, `git`) would also weaken the judge's view of the applicant. Removing the prefilter consumer eliminates the coupling; `SKILLS` becomes a clean single-use input to the judge.
- **`INCLUSION_KEYWORDS` becomes dead config.** With no whitelist, `INCLUSION_KEYWORDS` has no consumer. Removing it from the **Config** dataclass keeps the schema honest. The user's existing `synched/config.py` lives in a Syncthing-synced folder and is not regenerated by package upgrades, so a stale `INCLUSION_KEYWORDS = [...]` line will outlive this ADR for a while — the loader silently ignores unknown attributes with one INFO log line per run (`config has unused field 'INCLUSION_KEYWORDS' — safe to remove, see ADR-0026`). Hard-erroring on a harmlessly-extra field would block the run for no operator benefit; the INFO line gives a documented path to clean up at the operator's pace.
- **False-negative risk is explicit and accepted.** Under the old rule, a "Senior Data Engineer" listing whose body said "we don't take juniors" was rescued by `SKILLS`. Under the new rule it drops (if `junior` is in `NEGATIVE_KEYWORDS`). Mitigation: keyword-match is *title-only* now, so a body-text mention of `junior` no longer drops the listing — only a title like "Junior Engineer" does. The collateral surface is much smaller than a naive whitelist-removal would suggest. The user explicitly accepted higher prefilter false-negative rate in exchange for cost savings during the #319 grilling.

## Considered alternatives

- **Tighten the existing whitelist (drop generic single-word tokens like `git`, `python`, `sql`).** Rejected: requires a new prefilter-only opt-out mechanism because `SKILLS` is also consumed by the judge prompt, so it can't be pruned without weakening the judge's view of the applicant. New surface area for a clean-up task is wrong-direction. The empirical signature shows the whitelist isn't doing useful work at any scope, not just on generic tokens.
- **Restrict whitelist matching to title only, keep the rescue mechanism.** Rejected after the first grilling pass identified the deeper drift: with the upstream parsers already keyword-filtering, the rescue mechanism is solving a problem that no longer exists. Title-only would have papered over the symptom without fixing the cause.
- **Keep the whitelist on title-only and drop the blacklist's body scope to match.** Equivalent on the blacklist side, but leaves the redundant whitelist in place. Same critique.
- **Remove the Pre-Filter entirely; let the **Relevance Classifier** judge every listing.** Rejected: ADR-0005's original justification stands for the title-blacklist core. Title-level off-domain listings ("Sales Manager", "Werkstudent", "Praktikant") should not pay classify tokens. The deterministic title-blacklist is the cheap discard the LLM call can't be.
- **Move blacklist to title-only but keep body scope on the whitelist.** Rejected: this is the asymmetric shape that the old rule already had (whitelist could rescue from body matches even when blacklist hit body). Inverting it makes the asymmetry incoherent. Both lists are now title-only, symmetrically.
- **Add an embedding-based gate (BGE-M3 etc.) before classify.** Deferred (as ADR-0005 originally noted). A clean-shape title-blacklist achieves the immediate goal at zero dependency cost; the embedding upgrade remains a future swap behind the same Pre-Filter module surface.
- **Hard-error on legacy `INCLUSION_KEYWORDS` in user configs.** Rejected: the Config lives outside the package in the user's Syncthing folder. Blocking the run on a stale field after a package upgrade is hostile and offers no actionable error context — the operator hasn't done anything wrong, the field just stopped being read. Silent-ignore + INFO log respects the project's existing "log unparseable inputs at INFO" stance and matches the precedent from ADR-0019 about graceful config evolution.

## Consequences

- **`Domain Pre-Filter` module simplifies.** Single rule: `for kw in NEGATIVE_KEYWORDS: if kw.casefold() in normalize(title).casefold(): drop`. No two-list logic, no whitelist enumeration, no rescue branch. Per-position transcript record's `reason` enum collapses from `{no_hit, whitelist_only, whitelist_rescue, blacklist_drop}` to `{passed, blacklist_drop}`. The `whitelist_matches` field is removed; `blacklist_matches` and `title_len` remain. `body_len` is removed (body is no longer read by the Pre-Filter). The `*_dead` end-of-run summary keeps the `NEGATIVE_KEYWORDS_dead` list and drops the corresponding whitelist tracking.
- **`Config` schema loses `INCLUSION_KEYWORDS`.** Removed from the typed dataclass. The Config Loader silently ignores the attribute when present on a legacy `synched/config.py`, emitting one INFO line per run via the orchestrator's config-events log: `config has unused field 'INCLUSION_KEYWORDS' — safe to remove, see ADR-0026`.
- **`SKILLS` becomes single-consumer.** Used only by the **Match Judge** prompt's `{skills}` slot at `ClaudeExtractor` construction. The prefilter import of `SKILLS` is removed. No change to the user's `SKILLS` content — only its plumbing.
- **CONTEXT.md updates.** The `Inclusion Keyword` term is removed. `Domain Pre-Filter`, `Negative Keyword`, `Skill`, `Keyword`, `Config`, and the relationships section are updated to reflect title-only, blacklist-only matching.
- **Run Divider unchanged at the field level**, but the prefilter pass-rate distribution will shift visibly. Expected drop rate moves from ~26% to materially higher (estimated ~47% based on the 1800-decision log sample); judge call volume per run drops proportionally. The user should expect a one-time visible cost step-down on the first run after deploy.
- **Future firehose parsers.** A parser that cannot apply `KEYWORDS` on the wire must apply it client-side as a title-substring filter before yielding stubs from `discover()`. The orchestrator and Pre-Filter assume every yielded stub matches its `ParserQuery.keyword`. This invariant is parser-local (like **Location Coverage** per ADR-0014), not enforced by the Pre-Filter. Documented in CONTEXT.md under `Parser`. No v1 parser is firehose; this is an architectural commitment, not code to write now.
- **Test surface shrinks.** Existing prefilter tests for `whitelist_only` / `whitelist_rescue` paths are deleted, not rewritten. New tests assert: (1) a title containing only a Negative Keyword drops; (2) a body containing only a Negative Keyword passes (title-only scope); (3) a title with no Negative Keyword passes regardless of body content; (4) the Config Loader silently accepts a legacy `INCLUSION_KEYWORDS = [...]` and logs INFO.
- **ADR-0005 is superseded.** Its volume-reduction goal and "LLM stays the authority on in-domain" framing carry forward. The whitelist-precedence design, the language-detection-then-prefilter ordering (already obsolete per ADR-0019's single-language stance), and the `inclusion_keywords` / `negative_keywords` dual-list shape are retired.
