# Classifier runs a parallel worker pool of fixed size N

The **Relevance Classifier** runs N solo `claude -p` calls concurrently from a fixed-size pool of classify worker threads. N is a `Config` knob (`claude_classify_parallelism: int`, default 4). The per-call wire shape (solo, `<verdict>`-wrapped, prompt via stdin) and failure semantics from ADR-0036/ADR-0038 are preserved verbatim; only the pool size changes from 1 to N.

A single **Quota Wall** coordinates the workers against ADR-0023's sleep-through-reset behaviour. When any worker observes a `ClaudeUsageLimitError`, it publishes `reset_time` to a shared wall; before firing each new call, every worker checks the wall and blocks until past it. In-flight calls at the moment the wall is raised are allowed to finish — they will most likely 429 themselves, re-queue, and be sleep-blocked at their next dequeue. Exactly one `event=quota_sleep` row is written per orchestrator-wide wall raise (not one per worker).

Supersedes ADR-0036's explicit rejection of "Parallel solo worker pool (N>1 in flight)". Solo-call protocol (ADR-0036), stdin-combined prompt (ADR-0038), and worker→`DeduplicationStore` lock (ADR-0014) all survive unchanged.

## Why

- **The serial-budget premise from ADR-0036 no longer holds.** ADR-0036 rejected N>1 with "17-min serial wall-clock is well within budget." On 2026-05-22 the operator reported classify wall-clock running into multiple hours, dominated by per-call latency (model + subprocess startup + prompt size), not by 429 sleeps. Per-call latency is the lever a parallel pool actually moves — quota-bound runs would not benefit and could be made worse by parallelism, but this run is not quota-bound.
- **N concurrent solo calls compose cleanly with the prompt cache.** Anthropic's 5-min cache key matches the identical prefix every classify call ships. The first call within any 5-min window pays the cold-prefix tax; subsequent calls — parallel or serial — hit the warm cache. With N=4 cold-firing simultaneously at run start, the first N calls all miss the cache once; this is amortised over the thousands of subsequent classify calls per run. The cost margin vs. fully-warmed serial is negligible at personal-pipeline scale.
- **Fixed N, not ramp.** A "fire one, wait for warm, then fan out to N" ramp adds startup machinery to save at most N cold-miss prefixes per run. At the daily volume this is not worth the complexity; flat N from the first item.
- **N in `Config`, not hardcoded.** Different hosts have different Pi class (subprocess concurrency cost) and different sustained classify volumes; the knob lets the operator tune without a code edit. Default 4 is the recommended starting point; the operator drops it if median per-call latency rises (CPU/memory pressure) or if `quota_sleep` count rises (per-minute rate-limit pressure).
- **Shared Quota Wall, not per-worker sleeps.** Independent per-worker sleeps under policy (b) waste effort: every worker has to re-discover the wall through its own 429. A shared wall lets the *first* 429 protect the remaining N-1 workers from also burning a request to discover it. Cancelling in-flight calls (policy (c)) would add subprocess-cancellation plumbing for negligible gain — those in-flight calls cost nothing extra once submitted and naturally re-queue.
- **One `quota_sleep` log row per wall.** Per-worker rows would multiply the same wall event by N and complicate eval reconstruction. The orchestrator-wide row matches today's single-worker shape and remains a clean cut-point for prompt-tuning evals.

## Considered alternatives

- **Keep N=1 (status quo).** Rejected: ADR-0036's premise of "17-min serial wall-clock" no longer holds at current daily volume; multi-hour runs are not within budget.
- **Adaptive N (start at 1, ramp up after observing first cache hit).** Rejected: extra startup state machine for a one-time per-run saving of ≤N cold prefixes; not worth the complexity at this volume.
- **Per-worker independent quota sleeps (policy b).** Rejected: every worker re-discovers the wall through its own 429, wastes requests, and produces N quota_sleep rows for one quota event.
- **Cancel in-flight calls when the wall is raised (policy c).** Rejected: subprocess-cancellation plumbing for negligible gain; the in-flight calls cost nothing extra and naturally re-queue.
- **Hardcoded N in the orchestrator.** Rejected: removes the operator's ability to tune for host class or per-minute rate-limit pressure without a code change.
- **Move to a faster model instead of parallelising.** Rejected as orthogonal: model choice is ADR-0015's concern; parallelism is a multiplicative win on top of any model choice. Both levers may eventually pull.
- **Re-introduce batching from ADR-0014.** Rejected: ADR-0036's three robustness wins (whole-batch blast radius, lost-in-the-middle attention, JSON brittleness) all stand. Parallelism gets the throughput win without the robustness cost.

## Consequences

- **`Config` gains `claude_classify_parallelism: int`** (default 4, validated `≥ 1`). Materialised in `src/application_pipeline/templates/config.py` as a commented line carrying the default + tuning guidance ("raise for faster runs; lower if median per-call latency rises or `quota_sleep` count rises"). Loaded by the **Config Loader** into the frozen typed `Config`. `claude_classify_parallelism=1` recovers the ADR-0036 single-worker behaviour exactly — no code branch required.
- **Orchestrator constructs N `_ClassifyThread` instances** on the existing single `classify_queue`. Workers race `queue.get()`; first dequeued item wins. Worker thread names suffixed `classify-worker-0..N-1`. `_NO_MORE_BATCHES` sentinel is enqueued N times (one per worker) at parser end-of-life so each worker exits cleanly. `classify_thread.join()` becomes a loop over the N workers; the first worker's `exc` (if any) is raised after the join loop completes.
- **New `QuotaWall` helper** at module scope of `orchestrator.py` (or `llm/quota.py`). Surface: `raise_wall(reset_time: datetime) -> bool` (returns True iff this call was the first to raise the current wall — used to gate the single `quota_sleep` log row), `wait_if_blocked() -> None` (blocks until past the wall; cheap no-op on the common path), `is_active() -> bool`. Implemented with `threading.Condition` + a `datetime | None` field guarded by the same lock. Idempotent under concurrent `raise_wall` calls — only the first raise within a given window writes a log row and triggers the orchestrator-side sleep accounting.
- **Quota retry path**: a worker that catches `ClaudeUsageLimitError` calls `wall.raise_wall(exc.reset_time)`; if it was the first raiser, the orchestrator logs `event=quota_sleep, reset_time=..., wake_time=..., duration_s=...` once. The worker then calls `wall.wait_if_blocked()` (which sleeps until `reset_time + 2min` per ADR-0023) and retries the same solo call. Other workers hit `wait_if_blocked()` at the top of their next iteration and block on the same condition variable.
- **`DeduplicationStore` lock unchanged.** ADR-0014's `threading.Lock` already serialises `mark_*` writes; N=4 contention is bounded by classify call latency (orders of magnitude longer than lock acquisition).
- **Run Log thread-safety unchanged.** `RunLog` is already documented "safe to share across threads — each method opens, writes, closes per call." N concurrent `events.jsonl` / `transcripts.jsonl` writes line-atomically via the existing open-write-close-per-call pattern.
- **`RunMetrics` thread-safety.** The classify counters (`classify_calls`, `classify_input_tokens`, etc.) must be incremented under a lock or via atomic adds; today's single-writer assumption is broken. Audit `RunMetrics.classify_batch_complete` / `classify_batch_failed` / `classify_batch_dequeued` and add an internal `threading.Lock` covering the classify counter group if not already present.
- **Status Display row for `llm_classify_relevance`**. One row for the call-site, body string formatted from the aggregate counters; no per-worker rows. The body string already counts "running / done / failed" — semantics unchanged at N>1.
- **Ordering is non-observable.** The **Match Judge** runs end-of-run on the assembled pool (ADR-0020/0022); per-item completion order does not affect downstream behaviour. `extracts.json` writes go through the dedup-store lock and are race-safe.
- **CONTEXT.md updates** (follow-up slice):
  - `Relevance Classifier`: "A pipelined single classify worker pulls one item at a time" → "A pool of N classify workers (N = `claude_classify_parallelism`, default 4) pulls items concurrently".
  - `Config`: gains `claude_classify_parallelism: int` (default 4, `≥ 1`).
  - New `Quota Wall` glossary entry under "Filtering & scoring" or a new "Concurrency" subsection — the shared coordination object the pool checks before each call.
- **Operator monitoring**: after the first few post-change runs the operator checks `llm_classify_relevance.events.jsonl` for (i) end-to-end classify wall-clock drop, (ii) per-call median duration ≤ ~130% of the serial baseline, (iii) `quota_sleep` count per run not meaningfully higher. If (ii) or (iii) regress, lower `claude_classify_parallelism`. No formal SLO; this is a watch-the-dashboard bar at personal scale.
- **No migration.** Hosts on `0.x` without the new `Config` field get default N=4 on next run (`Config` materialises defaults). Hosts wanting to recover serial behaviour set `claude_classify_parallelism = 1`.
