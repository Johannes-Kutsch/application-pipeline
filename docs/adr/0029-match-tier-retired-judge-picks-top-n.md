# Match Tier retired; judge picks top-N from the in-domain pool

The **Match Judge** no longer assigns a `tier ∈ {green, amber, red}` to each in-domain **Position**. Instead, the judge takes the **entire in-domain pool for the current run** and returns the top **N=5** **Positions** ranked against the **Triage Profile** and **Skills**, with each winner carrying its **Match Verdict** (`matched`, `missing`, `summary`) plus a new explicit `rank: 1..N`. The **Match Tier** concept is gone — from `MatchVerdict`, from the **Layout**'s placeholder vocabulary, from the **Results File** trio, and from the **Run Divider** vocabulary it took with it.

This ADR supersedes ADR-0003 (Match Tier judged by LLM) in full. The companion ADRs are ADR-0030 (daily results file), ADR-0031 (structured extracts as pool members), ADR-0032 (quota sleep-and-retry), and ADR-0033 (one cron run per day).

## Why

- **Per-item judging is the dominant token cost.** Every in-domain **Position** today causes a `judge_match` call carrying the full `raw_description` to the judge — and on a slow week, the same `raw_description` is judged again on the next run if the URL gets re-discovered. The pool concept (ADR-0031) plus a single top-N judge call collapses N per-item calls into 1, and the per-item description tokens stop being paid on every rediscovery.
- **Tier was a coarse ranking with no comparative signal.** The judge could call ten listings "green" without any way to say *which* green was best. The applicant reads `green.md` top-down already — making rank explicit gives them what they were already doing implicitly. The amber/red files were dead surface in practice: the applicant reads green, and amber/red are mostly noise. Replacing the trio with "top 5 of the day" is the simpler artifact.
- **Drops three pieces of dead complexity.** With tier gone: the trio of files collapses to one daily file (per ADR-0030); the **Layout**'s `emoji` / `color` / `tier` placeholders stop being meaningful; the **Renderer**'s tier-derived placeholder dict shrinks; the **Match Tier** ambiguity-flag in `CONTEXT.md` retires. None of this is migration scaffolding worth preserving.
- **The single judge call is bounded by the pool, not by the run's inflow.** Today's per-item judge pays N calls for N in-domain items. The new shape pays exactly one judge call per run, regardless of pool size, with the per-item input being a compact structured extract (per ADR-0031) rather than a full description. The judge stays a single end-of-run call — no judge worker queue, no per-item parallelism, no judge-stage retry cost amplification.

## Considered alternatives

- **Keep tier alongside rank.** Rejected: tier was a bucketing decision masquerading as a ranking decision. Once rank exists explicitly, tier carries no information not in the rank, and keeping it would force the **Layout** to continue exposing tier-derived placeholders that exist only for backwards compatibility.
- **Top-N as a configurable cap (`judge_top_n` in `Config`).** Rejected for v1: 5 is the right size for "what the applicant reads over morning coffee" and a knob with no clear right answer adds surface area for no operational benefit. If demand for larger or smaller cards-per-day appears, lift it then.
- **Two-stage judge: rank pass on extracts, second per-winner pass for `matched`/`missing`/`summary` on full descriptions.** Considered during design (the "(c-ii)" option). Rejected: the structured extract (per ADR-0031) carries enough field-aligned content (`seniority`, `work_model`, `key_skills`, `key_responsibilities`, `must_have_requirements`, `notable_caveats`) for the judge to author 2-3 sentences and matched/missing without seeing the full description again. The second pass would re-pay description tokens on the 5 winners for no fidelity gain that the applicant would notice in the **Card**.
- **Tier-based file split kept; judge still picks top-N per tier.** Rejected: re-introduces the trio's surface (three files, three reset semantics, the **Run Divider** appearing in all three) for a ranking model that no longer needs it. The whole motivation for the trio was "the applicant reads green first, so don't bury it under amber/red volume" — top-N solves that directly without splitting.

## Consequences

- `MatchVerdict` (`src/application_pipeline/llm/types.py`) drops `tier`, adds `rank: int` (validated `1 ≤ rank ≤ N`). `MatchTier` enum is removed.
- `LLMExtractor.judge_match` signature changes: from `judge_match(raw_description) -> (MatchVerdict, CallUsage)` to a batched form taking the pool (`list[JudgeCandidate]` where each candidate carries the stable id and structured extract from ADR-0031) and returning `list[MatchVerdict]` of length ≤ N. The single-call `judge_match` is gone; the wrapper-class story (no wrapper) stays.
- The judge prompt template moves from "score one listing" to "select top 5 from the list below, output `<verdicts>[...]</verdicts>` with one entry per winner including `id, rank, matched, missing, summary`". The **Agent Output Protocol** machinery (ADR-0018) is unchanged.
- **Renderer** drops the `emoji` / `color` / `tier` placeholders. The **Layout**'s `CARD_TEMPLATE` loses those three placeholder slots; a `{rank}` placeholder appears (`Rank 1/5`-style). Existing user `layout.py` files referencing removed placeholders raise `LayoutError` on load — accepted: this is a single-shot user migration, not a back-compat concern.
- **Card** structure: H1 unchanged; meta line unchanged; **`## AI Assessment`** section now opens with a `**Rank 1/5**` line (or whatever the **Layout** chooses), then summary, then matched/missing; **`## Job Description`** unchanged.
- **Run Divider** vocabulary loses tier-derived counters (none existed beyond green/amber/red filing counts in practice — those are dropped). Run Divider itself is also retired entirely by ADR-0030.
- The **Match Tier** ambiguity in `CONTEXT.md` retires. New language: **Rank**, **Daily Top-5**, **Pool**.
- `.seen.json` `status` field: `kept` is renamed `selected_by_judge`; `off_domain` is renamed `out_of_domain`; `classified_in_domain` is renamed `in_domain` and its semantics shift (ADR-0031). `enrich_failed` and `external_redirect` are unchanged. No automatic migration — see ADR-0033.
- ADR-0003 is fully superseded. ADR-0021's "judge_resumed" Run-Divider counter is moot (Run Divider is retired); the underlying classify→judge resumption it enabled is now subsumed by the pool model (ADR-0031), so the resumption is no longer a special case.
