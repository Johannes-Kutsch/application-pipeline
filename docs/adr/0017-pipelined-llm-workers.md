# Classify and judge run on dedicated workers, pipelined with parsers

The **Pipeline Orchestrator** runs the **Relevance Classifier** and the **Match Judge** on two dedicated worker threads that pipeline with the parser threads. As soon as a per-language buffer reaches `claude_classify_batch_size` items, the main thread hands that batch to the classify worker; in-domain survivors stream straight to the judge worker, which renders, appends-fsyncs, and marks seen — all while parsers are still discovering. A `ClaudeUsageLimitError` no longer aborts the run: the LLM workers degrade to no-ops, the parser side keeps populating `.seen.json` (dedup hits, prefilter drops, enrich failures), and the run ends successfully with a `degraded_reason="usage_limit"` field on the **Run Divider**.

This ADR supersedes ADR-0016's "Usage-limit error mid-batch aborts the run" and its "pipelining: classify → judge surviving items → next classify" paragraph, and amends ADR-0007's "only the orchestrator's main thread reads `.seen.json`, writes `.seen.json`, runs the Relevance Classifier and Match Judge, renders, appends to `data/results/current.md`, and calls `mark_seen`" — the writer-thread invariant is relaxed in a structured way (see Consequences).

## Why

- **The status-display screenshot that motivated this work is the giveaway.** Mid-run, parsers are still walking pages (`16/72`, `27/72`, `0/72` queries) while `classify_relevance` and `judge_match` sit empty. The current implementation drains *all* parsers before touching the LLM, which means classify + judge wall-time stacks on top of parser wall-time instead of overlapping. On the Pi this turns a 20-minute pipeline into a 40-minute one for no reason — both stages are I/O-bound on entirely different remotes.
- **ADR-0016 already specified the overlap; the code never delivered it.** ADR-0016: *"as soon as a batch reaches `claude_classify_batch_size` items… the orchestrator sends the classify request, then iterates surviving in-domain items judging each one and appending+fsync+mark-seen before pulling more stubs."* Today's `orchestrator.run()` does the buffering+slicing *after* `with ExitStack(): … threads.join()`, which is the opposite of pipelining. This ADR makes the code match the spec — and goes one step further by putting classify and judge on their own threads instead of inlining them into the main loop.
- **Three threads, not two.** The main thread stays a pure parser-outbound consumer (preserving the back-pressure shape from ADR-0009: parser pushes, main thread decides `ENRICH/SKIP/SKIP_AND_END_QUERY` synchronously). Classify gets its own thread because each Claude call is many seconds on the wire — blocking the main thread on classify reintroduces the same starvation problem we set out to fix. Judge gets its own thread because Claude judges one listing at a time (per ADR-0016, judge stays scalar); fanning judge calls behind a queue means classify-of-batch-N+1 can already be in flight while judge is still chewing batch-N's in-domain survivors.
- **Per-language buffers stay on the main thread.** The main thread builds `de_buffer` and `en_buffer` as survivors emerge from the prefilter (the language partitioning logic lives next to the prefilter that already produces it). When a buffer hits `claude_classify_batch_size`, the main thread slices off exactly `batch_size` items and `put`s them on the classify queue as a homogeneous-language batch. The classify worker is a dumb consumer — given a batch, classify it; push in-domain survivors onto the judge queue. End-of-run: the main thread flushes any non-empty residual buffers as final undersized batches before sending the sentinel.
- **Writer ownership is split by phase, not arbitrated by locks.** `.seen.json` now has three potential writers (main: enrich-failed / external-redirect / dedup-hit-related entries; classify worker: off-domain marks; judge worker: kept marks). It is the only artifact that genuinely sees concurrent writes — guarded by a single `threading.Lock` inside `DeduplicationStore`. `data/results/current.md` retains a single writer *at any given time*, but ownership rotates by phase: the main thread writes the file header at startup and the **Run Divider** at end-of-run (after `t.join()`-style barriers); the judge worker is the sole writer during the run itself. No lock on the Results File Manager — phase separation does the work a lock would.
- **Usage-limit is "the run got less done than it could have," not "the run failed."** ADR-0012 makes `data/failures/<ts>.md` the user-facing alert surface. Filing a failure report every time the Pi hits a daily Claude quota — which on the steady-state pipeline is plausible *several times a day* during a busy job-search window — would train the user to ignore the directory, the opposite of what ADR-0012 wants. Instead: the LLM workers see `ClaudeUsageLimitError`, set a shared `degraded_event`, drain their queues to no-ops (still popping items so the senders don't block), and exit cleanly. The main thread keeps consuming parsers' outbound. Dedup hits, prefilter drops, enrich failures, and external-redirect marks continue to land in `.seen.json` — so the *next* cron tick re-parses the same listings cheaply (dedup short-circuits URLs already seen; prefilter is cheap), and only the unclassified survivors re-pay enrich. The Run Divider records `degraded_reason="usage_limit"` plus partial counters; absence of forward progress shows up as flat `kept=` counts in the Divider history, which is greppable.
- **Sentinel cascade for shutdown, identical on happy and degraded paths.** When `parsers_remaining` is empty, the main thread flushes any non-empty buckets to the classify queue (one final undersized batch per language) and pushes a `_NO_MORE_BATCHES` sentinel. The classify worker drains its queue (processing or no-op'ing each item depending on `degraded_event`), forwards a `_NO_MORE_JUDGES` sentinel, and exits. The judge worker drains its queue and exits. Main thread `join()`s both, writes the Run Divider, returns the `RunSummary`. The degraded path differs only in that workers `continue` past each non-sentinel item instead of processing it — no second teardown protocol to maintain. `ClaudeUsageLimitError` on the main thread (a parser-side Claude call doesn't exist today, but if a future step adds one) sets the same flag.
- **Abort propagation for non-quota exceptions.** Any other unhandled exception in a worker (an `ExtractorError` that escapes, an `OSError` from `fsync`) is captured on the worker's `self.exc`, sets a fatal `abort_event`, and the main thread re-raises it after `join()`. The main loop's `outbound.get()` becomes `outbound.get(timeout=0.25)` so it can periodically check `abort_event` even when parsers are quiet. ADR-0012's failure-report path fires for these — they are crash-class events, not quota-class.
- **Status display is thread-safe internally.** Both workers update their own rows (`classify_relevance`, `judge_match`) directly via `status_display.update_body(...)`. The display module gains an internal `threading.Lock` around table mutations (rich `Live` is not documented thread-safe; the plain implementation's `print` is line-atomic but multiple threads interleaving still want explicit ordering). The "only the main thread mutates display state" line in CONTEXT.md is softened accordingly.
- **Item ordering in `current.md` follows judge-completion order.** Position numbers are run-scoped and have never been stable identifiers (URL is). With overlapping classify and judge, items land in `current.md` as the judge worker finishes them — a function of parser speed, classify batch boundaries, and judge wall time. The live status display already reflects this order; the numbered file simply matches it. Accepted explicitly; not a regression.

## Considered alternatives

- **Keep classify and judge inlined on the main thread, just trigger classify when a buffer fills.** Rejected: while the (multi-second) Claude call runs, the main thread cannot pop the parser outbound queue, so the parsers' producer threads block on `outbound.put` (lockstep contract from ADR-0009 means each parser is one stub ahead). End-state wall-clock is barely better than the current "wait for all parsers, then process" implementation. The point of this work is *overlap*, not a different serialization order.
- **One LLM worker thread doing classify *and* judge serially, with writes routed back to main via a result queue.** Rejected: the main thread already has one input queue (`outbound`); adding a second queue forces a multiplex pattern (`outbound.get(timeout)` + `result_queue.get_nowait()` in a poll), which is the kind of code that grows subtle bugs and obscures data flow. Phase-separated writer ownership (this ADR) preserves the "single writer at any given time" property without the multiplex.
- **Three threads with judge driven by a `ThreadPoolExecutor` for true judge fan-out.** Rejected for v1: Claude per-account concurrency is low and unstable, and parallel judges would compete with the next classify batch for the same quota. The pipeline (classify-N → judge-N → classify-N+1) is the right grain.
- **Treat usage-limit as a failure (ADR-0016's current text), and write a failure report.** Rejected for the reasons in the bullet above: noisy, trains the user to ignore the alert directory, and the next cron tick already retries via dedup short-circuit.
- **Preserve language-grouped output order by buffering rendered blocks and appending at end-of-run.** Rejected: defeats the streaming-output property that justified the redesign. The live display would race ahead of `current.md`, and a crash between "all done" and "flushed buffered blocks" would lose kept items.
- **Move the language-buffer ownership into the classify worker (push individual survivors onto its queue).** Rejected: spreads batching logic across the thread boundary, complicating the end-of-run flush dance. The main thread already partitions by language for the prefilter; keeping the buffers there is a smaller delta from today's code.

## Consequences

- **`DeduplicationStore` gains a `threading.Lock`.** `is_seen` and `mark_seen` acquire it. ADR-0007's "no `threading.Lock` around `DeduplicationStore.is_seen` / `mark_seen`" sentence is reversed; the lock is added now because there are genuinely concurrent writers. Per-call overhead is unmeasurable (Python `Lock` is a few hundred ns; the protected critical section is a dict lookup + a JSON-line append).
- **`mark_seen`'s magic-string `status` parameter is removed in favor of four narrow methods on `DeduplicationStore`: `mark_off_domain(stub)`, `mark_kept(stub)`, `mark_enrich_failed(stub)`, `mark_external_redirect(stub)`.** Today's four call sites pass bare strings; with three threads each potentially marking, magic-string drift would be a silent bug class. Narrowing the interface makes "who marks `kept`?" a one-grep question and gives the lock site one method body per status instead of one shared method gated on a string arg. The on-disk format (`status: "off_domain" | "kept" | "enrich_failed" | "external_redirect"`) is unchanged.
- **`ResultsFileManager` keeps its current non-locked design.** Phase separation (main thread writes header pre-run and Divider post-run; judge worker writes during the run, after `parsers_remaining` is empty would still be before the Divider — but `join(judge)` precedes the Divider write) makes the file effectively single-writer per phase.
- **ADR-0007's writer-thread invariant is amended.** The relaxation is structured, not free-for-all: parser threads remain pure producers (no change); the main thread is no longer the *sole* writer of `.seen.json` or `data/results/current.md`; instead `.seen.json` is multi-writer behind a lock, and `data/results/current.md` is single-writer per phase with judge as the run-phase writer. ADR-0007's "single-writer at the file level becomes literal" claim is downgraded to "single-writer-per-phase for the Results File, lock-protected for the seen-state."
- **`ClaudeUsageLimitError` semantics flip.** ADR-0016's "Usage-limit error mid-batch aborts the run. The orchestrator writes `data/failures/<ts>.md`, exits non-zero" is superseded by: workers set `degraded_event`, drain queues to no-ops, run ends successfully, Run Divider gains `degraded_reason="usage_limit"`. Exit code stays 0. ADR-0012's failure-report path is reserved for crash-class exceptions (`OSError`, unhandled `ExtractorError`, etc.).
- **`Run Divider` grows a conditional field.** `degraded_reason=<reason>` appears only when the run was degraded (so non-degraded Dividers stay short). For v1 the only reason is `usage_limit`; future degradations (e.g., `claude_cli_missing`) reuse the same slot. CONTEXT.md's Run Divider paragraph gains this field.
- **New queue topology, extending ADR-0009:** in addition to the shared `outbound` queue and per-parser `inbound[parser_id]` queues, the orchestrator owns one `classify_queue` (main → classify worker, items = `_ClassifyBatch` plus `_NO_MORE_BATCHES` sentinel) and one `judge_queue` (classify → judge worker, items = `_JudgeJob` plus `_NO_MORE_JUDGES` sentinel). `_ClassifyBatch` wraps `language: str` and `positions: list[tuple[Position, LanguageResolution]]`; `_JudgeJob` wraps `position: Position`, `resolution: LanguageResolution`, and `item_id: str` (the opaque classify-batch id, reused for transcripts log cross-reference per ADR-0016). Typed dataclasses replace anonymous tuples so the queue contract is greppable and extensible. Queues are unbounded; back-pressure is implicit (classify is the slow stage, so the judge queue stays short; the classify queue is bounded by batch granularity since the main thread only enqueues whole batches).
- **A `_RunState` object replaces individual `degraded_event` + `abort_event` flags.** Fields: `degraded_reason: str | None`, `fatal_exc: BaseException | None`, `degraded_at: datetime | None`, `degraded_by: str | None` (thread name). Properties: `is_degraded`, `is_aborted`. Methods: `set_degraded(reason: str, by: str)`, `set_aborted(exc: BaseException)`. The consolidated object gives forensic information (when and where degradation tripped) for free and prevents the flag-set growing one Boolean at a time as future degraded modes appear. Workers and main thread share one `_RunState` instance; writes are guarded by an internal `threading.Lock`.
- **`StatusDisplay` implementations gain internal thread-safety.** `RichStatusDisplay` wraps its `Live` table mutation in a `threading.Lock`; `PlainStatusDisplay` does the same around its `print`. The `register / update_phase / update_body / remove` surface is unchanged. CONTEXT.md's "only the main thread mutates display state" sentence is replaced by "the display is thread-safe; any thread may update its own row."
- **`_BatchStats` is split along thread-ownership lines.** Each worker thread owns the counters it mutates: `_ClassifyStats` (classify worker — `classify_calls`, `classify_items`, `classify_*_tokens`, `classify_total_s`, `classify_failed`, `classifier_dropped`, `classify_cost_usd`) and `_JudgeStats` (judge worker — `judge_calls`, `judge_failed`, `judge_total_s`, `judge_*_tokens`, `judge_cost_usd`, `written`, `green`, `amber`, `red`, `written_per_source`, `errored`). Each stat object is mutated only by its owning thread; the main thread reads them after `join()`. No lock — the join is the memory barrier. The combined Run Divider numbers are computed on the main thread by summing across both stats objects. The type signature itself documents the seam.
- **A `_QueueWorker` base class captures the shared classify/judge worker pattern.** Both workers do the same dance: set `current_stage` once at thread start, loop on `queue.get()` until sentinel, wrap the loop in `try/except → self.exc = exc; run_state.set_aborted(exc)`, no-op past items when `run_state.is_degraded`. `_QueueWorker` lives next to `_ParserThread` in `orchestrator.py`, takes the input queue + sentinel + stage name, and exposes a single `_process(item)` hook. `_ClassifyThread` and `_JudgeThread` override `_process` only. The abort + degraded behavior is implemented once.
- **Tests need a new shape.** Existing tests that pre-loaded all survivors and then ran classify+judge inline still work for the inlined `_process_batch` helper. New integration tests must drive a real `_ClassifyThread` and `_JudgeThread` (or test doubles) over a populated queue and assert ordering-agnostic outcomes (kept set, off-domain set, residual seen-state). Add a usage-limit-degraded-run test: inject a `ClaudeUsageLimitError` on the second classify batch, assert run exits 0, Divider has `degraded_reason="usage_limit"`, partial counters reflect the abort point, dedup-store has the expected pre-abort marks plus all parser-side marks (dedup hits, enrich failures) for stubs the parsers continued to produce after the abort.
- **No new `Config` field.** Queue sizes are unbounded; abort behavior is hardcoded; degraded behavior is hardcoded. Tunables can come later if either choice proves wrong.
- **End-of-run order in `current.md` is no longer language-grouped.** Documented above; not a regression — order was never specified as stable.
