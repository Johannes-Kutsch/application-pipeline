# LLM call sites are log components, not classes

> **Clarifies [ADR-0024](0024-log-convention-split-streams.md):** the "one file per component" rule still holds; this ADR defines `component` for LLM code as the *call site* (`classify_relevance`, `judge_match`), not the implementation class (`ClaudeExtractor`).

Each LLM call site is its own log component. `classify_relevance` and `judge_match` each own a `<call_site>.events.jsonl` plus a `<call_site>.transcripts.jsonl`. There is no `claude_extractor.*` stream — the implementation class shared by both call sites is not a log component. A row's home is determined by the call site it concerns, regardless of which module wrote it (the orchestrator's judge worker thread and `ClaudeExtractor._invoke` both write into `judge_match.events.jsonl`).

The **Run Divider**'s Claude metrics split per call site to match: `classify_input_tokens`, `classify_output_tokens`, `classify_cache_read_tokens`, `classify_cost_usd`, `judge_input_tokens`, `judge_output_tokens`, `judge_cache_read_tokens`, `judge_cost_usd`. The previous aggregate fields (`claude_input_tokens`, `claude_output_tokens`, `claude_cache_read_tokens`, `claude_cost_usd`) are removed; the aggregate is `classify_* + judge_*` and can be re-derived by readers that want it.

## Why

- **Today's on-disk layout is inconsistent.** Successful calls land in `claude_extractor.events.jsonl` / `claude_extractor.transcripts.jsonl` (filed by writer module). Failed calls land in `classify_relevance.transcripts.jsonl` / `judge_match.transcripts.jsonl` (filed by call site). The orchestrator's judge worker thread writes to `judge_match.events.jsonl`, while `classify_relevance.events.jsonl` doesn't exist at all. No single file tells the full story of one call site — exactly the readability failure ADR-0024 was meant to fix.
- **"What was classify doing vs what was judge doing" is the dominant question.** The two call sites use different models (`haiku` vs `sonnet`), different effort levels, different prompt shapes, different cost profiles, and different failure modes. Mixing them into one stream forces a `jq '.event == "classify_relevance_batch"'` filter on every analysis. Filing by call site makes the filter the filename.
- **Carrying the split into the Run Divider keeps the story consistent end-to-end.** If the per-call-site separation matters enough to own its file, it matters enough to own its divider field. Eight Claude fields in an HTML-comment block is still trivially greppable.
- **New call sites compose cleanly.** A future LLM call site (e.g., a re-write or expand step) gets its own pair of files automatically, with no special-casing. The "class is the component" framing would have forced every new call site to either share `ClaudeExtractor`'s stream or get a wrapper class.

## Considered alternatives

- **One file per call site, but fold events and transcripts into a single `<call_site>.jsonl`.** Rejected for the same reason ADR-0024 rejected the equivalent for non-LLM components: prompt/response payloads are large strings with a different reader (prompt debugging) than step-trace events (`jq` aggregation). Mixing them bloats the analysis stream.
- **One file per component, where component is `claude_extractor`.** Rejected: the failure path's existing per-call-site filing is the right instinct — recovering "what did classify cost over the last week" from a mixed stream is strictly worse than from a per-call-site stream. The asymmetry today is a bug worth fixing toward the per-call-site side, not the per-component side.
- **Per-call-site files plus a `claude_extractor.events.jsonl` aggregate stream.** Rejected: double-writing every row to satisfy readers that want the aggregate. The aggregate is trivially `cat classify_relevance.events.jsonl judge_match.events.jsonl | jq …`; paying disk and write-amplification for the convenience isn't worth it on a Pi.
- **Keep the divider's aggregate Claude fields alongside the new per-call-site fields.** Rejected: redundant, and the aggregate is mechanically derivable from the per-call-site fields. The "afraid to commit" shape.
- **Treat `claude_extractor` as a component but document a per-call-site override of ADR-0024 for it.** Rejected: a special-case override in the log convention is exactly what created today's mess. Defining "component" call-site-wise for LLM code keeps ADR-0024's rule intact and removes the special case.

## Consequences

- **`ClaudeExtractor._invoke`'s success-path writers** stop using the module-level `_COMPONENT_ID = "claude_extractor"` and route to `site.component_id` (matching what the failure path already does). The constant `_COMPONENT_ID` goes away.
- **`claude_extractor.events.jsonl` and `claude_extractor.transcripts.jsonl` are no longer produced.** Existing files from prior runs become stale historical artifacts; operator deletes manually if desired. No automatic migration.
- **`classify_relevance.events.jsonl` starts existing** — every classify batch call (success or failure) produces a row. `classify_relevance.transcripts.jsonl` (already exists for failures) gains successful-call rows.
- **`judge_match.events.jsonl` gains the success path** alongside the orchestrator worker's existing error rows. Mixed event shapes in one file are fine — the events schema is already "free-form keys per row" per ADR-0024.
- **Run Divider field set changes.** Remove `claude_input_tokens`, `claude_output_tokens`, `claude_cache_read_tokens`, `claude_cost_usd`. Add `classify_input_tokens`, `classify_output_tokens`, `classify_cache_read_tokens`, `classify_cost_usd`, `judge_input_tokens`, `judge_output_tokens`, `judge_cache_read_tokens`, `judge_cost_usd`. `RunMetrics` aggregation must keep per-call-site running totals (it already has the inputs via `CallUsage.cost_usd` and the call-site identity at emit time).
- **CONTEXT.md updates.** The **Log Artifacts** entry gains the per-call-site rule for LLM components. The **Relevance Classifier** entry's mention of `classify_relevance.transcripts.jsonl` is joined by `classify_relevance.events.jsonl`. The **Run Divider** entry's Claude field list is replaced with the per-call-site split.
- **ADR-0016's reference to `data/logs/classify_relevance.log`** is already superseded by ADR-0024 (per-component `.log` files were consolidated into `run.log`); the new per-call-site `classify_relevance.events.jsonl` is now where structured per-batch step events live.
