# Cron fires once per day; deployment wipes `.seen.json` and the old trio

The Pi's cron schedule for `application_pipeline` reduces from multi-fire-per-day to a single fire per day. Each fire is one logical run — parsers walk every **Source**, classifier processes everything `not_classified`, judge picks the **Daily Top-5**, daily file is written, run ends. Deployment of this refactor wipes the existing `.seen.json`, `data/extracts.json` (if any), and `data/results/{green,amber,red}.md`. No automatic state migration is provided. Historical kept records do not survive the cutover.

Companion ADRs: ADR-0029, ADR-0030, ADR-0031, ADR-0032.

## Why

- **One judge call per day means one cron fire per day is sufficient.** Under ADR-0029 the judge is a single end-of-run call producing the day's 5 cards. Running cron multiple times per day would either re-issue the judge call (filling the daily file with progressively more cards as the day went on — model (b) in Q7) or fire-but-skip-the-judge (model (c)). Either of those is a strictly more complex contract than "once a day, one file, five cards." Simplicity wins where the more sophisticated cadence had no operational benefit the applicant would observe.
- **Parser freshness is no longer a forcing function.** Under the previous design, multi-fire-per-day cron caught listings posted between fires so the trio could surface them quickly. Under daily output, "caught at 9am" vs "caught at 5pm" both surface in *tomorrow's* file; the intra-day latency was load-bearing only because the trio updated continuously. With the daily-file artifact, intra-day cadence is invisible to the applicant.
- **Quota interaction is cleaner with one fire per day.** ADR-0032's sleep-and-retry has unbounded sleep duration. With multi-fire cron, a sleeping run blocks several subsequent fires (one per cron interval). With once-per-day cron, a sleeping run blocks one fire — at most one missing daily file before the operator notices. The flock semantics match the operator's mental model.
- **Migration is not worth the engineering cost.** Three options were considered (Q11): auto-rename statuses (`off_domain` → `out_of_domain`, etc.), wipe everything, or hybrid (keep terminal `out_of_domain`/`enrich_failed`/`external_redirect` records, wipe `classified_in_domain`/`kept`). The applicant chose **wipe**. The "preserve don't-re-classify-noise" win of the hybrid option costs perhaps one day of classifier tokens to recover organically — not worth the migration code, the schema-version handling, or the load-time branch.
- **Cron timing is a Pi-side knob, not a `Config` knob.** Cron fires when the operator configures crontab — there is no `Config.cron_schedule` field. Recommended: an early-morning hour (e.g. 03:00 local) so the daily file is on the laptop when the applicant wakes up, but this is a setup-doc recommendation, not a code contract.

## Considered alternatives

- **Keep multi-fire cron with one judge call per day, gated on "no file for today yet" (Q7 (c)).** Rejected: the cost saved (parser HTTP requests amortized across the day rather than bursted at one hour) is small for a hobbyist pipeline querying three German sources with their own pacing. The orchestrator-level complexity (per-fire idempotence, "has today's judge run yet" check, partial pool state across fires) is meaningful. Once-per-day is simpler at every layer.
- **Multi-fire cron with `selected_by_judge` flips happening mid-day (Q7 (b)).** Rejected: produces up to `5 × cron_fires` cards per day, which is more output than the applicant wants and re-introduces the green-burial problem in a new shape. Top-5 should mean top-5.
- **Auto-migrate old statuses.** Rejected (Q11): the rename map is straightforward but the semantic shift on `classified_in_domain` → `in_domain` requires backfilling extracts, which requires re-running the classifier on every legacy in-domain record. That's effectively a wipe-with-extra-steps.
- **Hybrid migration (keep terminal records, wipe transitional ones).** Considered (Q11 (c)). Rejected by applicant: preserves a small token-cost win in exchange for code that exists exactly once and is then dead. Wipe is the cleaner story.
- **Migration via a one-shot CLI subcommand the operator invokes manually.** Rejected: same conclusion as auto-migrate, with an extra step. If the wipe is acceptable, the CLI subcommand has no audience.

## Consequences

- **Cron line on the Pi** changes from the current multi-fire pattern to once daily. Example: `0 3 * * * /home/pi/application-pipeline/current/.venv/bin/python -m application_pipeline` wrapped by the existing flock and deploy-wrapper (ADR-0011). The exact hour is operator-tunable.
- **Deployment of this refactor includes a state-wipe step.** Either documented in `docs/pi-setup.md` ("before first run of the new pipeline, `rm data/.seen.json data/results/green.md data/results/amber.md data/results/red.md`") or wrapped in the deploy script — implementation detail.
- **No load-time branch for legacy statuses.** The config loader / dedup store can assume the new status enum (`not_classified | out_of_domain | in_domain | selected_by_judge | enrich_failed | external_redirect`). A legacy value in a leftover `.seen.json` raises a load error pointing at the wipe instruction. No silent translation.
- **`docs/pi-setup.md` updates** to describe the once-daily cron line, the wipe step, and the daily file location.
- **First-day-after-cutover behavior:** parsers walk every **Source** as if nothing existed. Every stub is `not_classified`; the classifier processes the full first-run volume in one pass (potentially hitting quota — ADR-0032 handles that). The judge picks 5. By day 3 the pool is in a steady state.
- **Operator signal for "did cron fire today?":** `data/results/YYYY-MM-DD.md` existence is the primary signal (per ADR-0030). If absent, the next-cheapest signal is `pipeline_orchestrator.events.jsonl` (newest event row's timestamp). If the orchestrator never started, no signal exists — same as today.
- **Implementation note on existing thread machinery:** ADR-0017's pipelined classify-worker-with-parsers may stay if the implementation finds the parser/classify overlap valuable for wall-clock; with one fire per day and bounded inflow, serializing parsers → classifier → judge is also acceptable. The classify worker queue is not load-bearing in the new model. No ADR commitment either way.
- **CLAUDE_ACCOUNTS / pool config** is not added (per ADR-0032, deferred). `Config` schema does not gain any new fields from this ADR — the changes are operational (cron, wipe), not configurational.
