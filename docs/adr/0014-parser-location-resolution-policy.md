# Parser location resolution as a per-parser predicate + transform, validated at config load

The orchestrator â†’ parser seam carries a typed `Location = City(name) | Remote` instead of `str | None`. Each **Parser** module declares its geographic coverage and wire-format translation locally, via four module-level symbols: `serves(name: str) -> bool`, `to_wire(name: str) -> str`, `serves_remote: bool`, and `remote_wire() -> Any`. A shared helper `parsers/location.py` exposes a single `resolve(location, parser_module) -> Resolved | NotServed | RemoteWire` that parsers call once at the top of `discover` and pattern-match on. The **Config Loader** validates at load time that every entry in `LOCATIONS` is served by at least one configured **Source**, and that `INCLUDE_REMOTE=True` is satisfied by at least one source whose `serves_remote` is true; mismatches raise `ConfigError`.

## Why

- **The orchestrator â†’ parser contract carries the semantic anchor in its type.** `query.location is None` meant "remote-only call" only because CONTEXT.md said so; readers had to know the rule. A sealed `Location` makes the meaning structural â€” `case Remote()` reads itself, the type checker enforces exhaustiveness, and a future variant forces every parser to handle it explicitly.
- **The duplication being retired is the slug tables, not "location handling" in general.** Each parser previously carried a 28â€“33-row `_LOCATION_SLUGS` dict where ~half the rows were umlaut-variant duplicates (`kĂ¶ln` and `koeln` both mapping to one wire form). The decision is to delete the city listings entirely â€” not to consolidate them into a shared catalog. Parsers declare a predicate (does the source serve this name?) and a transform (how to write it on the wire), nothing else. A nationwide parser like Bundesagentur has `serves = lambda _: True`; a geo-pinned parser like stellen.hamburg has `serves = lambda n: normalize(n) == "hamburg"`.
- **Honest user-facing failure on typos.** With no city catalog at all, `LOCATIONS = ["Atlantis"]` would silently return zero results across all sources â€” indistinguishable from "real city, no jobs today." Load-time validation closes that gap: the union of `serves` predicates across configured sources defines the dynamic pool of resolvable cities; anything outside it raises `ConfigError: no configured source serves location 'Atlantis'`. The pool grows when a parser broadens its predicate and shrinks when a parser is removed from `SOURCES` â€” no global list to drift.
- **Remote stays per-parser because the wire shape genuinely differs.** Bundesagentur adds an extra query parameter (`arbeitszeit=ho`); jobs-beim-staat uses a different value in the same slot (`place=homeoffice`); stellen.hamburg has no remote facet at all. Forcing a uniform wire shape would either lose information or lie about the data. `remote_wire()` returns whatever the parser needs to send â€” opaque to the helper, dispatched by the parser. The shared concern (does any configured source serve remote at all?) is validated via `serves_remote: bool`, which is uniform.
- **The helper is the seam, not the policy.** `parsers/location.py` owns the sealed `Location` type, the `Resolved | NotServed | RemoteWire` result type, the `resolve()` dispatch, and the load-time validation hook. It does not own a city catalog, a normalization map, or any per-source knowledge. Parsers stay the local authority on their own coverage; the helper just enforces a consistent pattern for asking.

## Considered alternatives

- **Keep `ParserQuery.location: str | None` and codify the meaning in an ADR only.** Rejected: the sentinel-`None` pattern is exactly what the sealed type fixes. The rule "None means remote" only lives in prose; a new parser author has to read CONTEXT.md to find it, and a type checker cannot enforce exhaustive handling.
- **Add an `Anywhere` variant alongside `City` and `Remote`.** Rejected: no config knob emits it today, no current parser would treat `Anywhere` differently from `Remote`, and the question "what does this source do for `Anywhere`?" has no acceptance criterion anchoring an answer. Adding it speculatively forces three parsers to make a meaningless decision. Exhaustive matching with `assert_never` makes adding the variant later cheap.
- **Shared canonical city catalog in `parsers/location.py`, parsers declare subsets.** Rejected: duplicate truth. The actual coverage is what each parser supports; forcing parsers to be subsets of a separate list means two places to update for every change, and the central catalog becomes either the union (effectively redundant) or a curated subset (drift bait).
- **Coverage declared on `SourceEntry` in `Config`.** Rejected: the wire-form translation (`KĂ¶ln` for Bundesagentur, `koeln` for jobs-beim-staat) is parser-specific knowledge that has no business living in `Config`. Splitting "what cities does this source serve?" from "how do we write each one?" puts related decisions in two files.
- **Orchestrator pre-resolves locations into a `ResolvedQuery` before calling `discover`.** Rejected: the orchestrator would have to import each parser module's predicate to build the worklist, coupling it to geographic knowledge it otherwise doesn't touch. The Cartesian construction in `_run_pipeline` stays geo-blind.
- **ASCII-fold umlauts in the helper (`Ă¶â†’oe`, `ĂĽâ†’ue`, â€¦) so user input accepts both `KĂ¶ln` and `Koeln`.** Rejected: each parser's wire form is one canonical string per city, and the user types one spelling per city in their config. Sending both forms either doubles API calls or commits the parser to picking one â€” knowledge that's already source-specific. Honest failure on a typo'd second spelling is preferred over magic input rewriting.
- **Drop coverage knowledge entirely; let the source's API return zero for unknown cities.** Rejected after pushback (good idea, wrong conclusion): works for nationwide parsers but breaks geo-pinned ones â€” stellen.hamburg would burn one HTTP round-trip per non-Hamburg location per run rather than short-circuit. Also forfeits the typo-detection that load-time validation provides. The middle ground (`serves` predicate + `to_wire` transform, no city list) keeps the user-facing simplicity while preserving honest failure modes.
- **`COVERAGE: dict[str, str]` per parser instead of `serves()` + `to_wire()`.** Rejected: re-introduces the slug-table maintenance burden the policy was designed to retire. The predicate-plus-transform form expresses Bundesagentur's "I serve all of Germany" as one lambda instead of an enumerated 50-row dict, and expresses stellen.hamburg's "I am geo-pinned" as a one-line predicate.
- **Parser self-gates inside `discover` (`if not serves(name): return`) without going through the helper.** Rejected as the long-term shape: each parser would carry a three-line gate block, exactly the duplication the policy was meant to retire. Routing through `resolve()` makes the gate one `match` statement and dispatches `City` / `Remote` / `NotServed` uniformly.

## Consequences

- **`ParserQuery.location` type changes from `str | None` to a sealed `Location = City(name) | Remote`.** The type lives in `parsers/types.py` next to `ParserQuery`. Every `ParserQuery(...)` construction site updates â€” orchestrator, parser dev scripts, every parser test. Test constructors get a one-line wrapper.
- **A new module `src/application_pipeline/parsers/location.py`** exposes the sealed `Location` type, the sealed result type `Resolved(wire: str) | NotServed | RemoteWire(payload: Any)`, the `resolve(location, parser_module) -> Resolved | NotServed | RemoteWire` function, and the load-time validation entrypoint used by the **Config Loader**.
- **Each parser module exports four symbols at module level** conforming to a `LocationCoverage` `Protocol`: `serves(name: str) -> bool`, `to_wire(name: str) -> str`, `serves_remote: bool`, `remote_wire() -> Any`. The `Any` on `remote_wire` is deliberate â€” the wire shape is heterogeneous (extra params dict, place value, etc.) and the parser is the one that interprets its own return value.
- **Each parser's `discover` opens with one `match` statement** on `resolve(query.location, _module)`: `Resolved(wire)` â†’ call API with the wire form; `RemoteWire(payload)` â†’ parser-specific remote handling; `NotServed` â†’ return empty generator. The `_LOCATION_SLUGS` constants disappear.
- **`Config Loader` validation grows two new checks**, both before the first fetch: every `LOCATIONS` entry must satisfy `any(src.serves(loc) for src in configured_sources)`, and `cfg.include_remote` requires `any(src.serves_remote for src in configured_sources)`. Failures raise `ConfigError` with a "did you meanâ€¦?" hint computed via `difflib.get_close_matches` against the union of `serves`-resolvable inputs.
- **Runtime "this source doesn't serve this city" is no longer an error or a warning.** Load-time validation catches typos; runtime mismatches (e.g. stellen.hamburg called for Berlin in a multi-source config) are an expected outcome of Cartesian expansion and resolve to `NotServed` â†’ empty generator â†’ INFO log line for observability.
- **The `_LOCATION_SLUGS` example in ADR-0007** is replaced with `_DISPLAY_NAME` â€” the slug tables disappear, but the underlying rule ("parser modules may have constants but no shared mutable state") stands.
- **CONTEXT.md updates** documented in the relevant entries: **Parser**, **Position Stub**, **Source**, and the new **Location** / **City** / **Remote** / **Location Coverage** terms. The **API Parser** and **WebFetch Parser** terms are removed (orthogonal cleanup â€” the API/HTML distinction was descriptive flavor that didn't gate any behaviour, and jobs-beim-staat's REST migration into a JSON-envelope-with-HTML-payload shape made the labels actively misleading).
