# Hardcoded prompts with externalized user-info; single-language pipeline

LLM prompts are split into two parts. The **agent protocol** — task framing, output-tag instruction (`<verdicts>` / `<verdict>`), JSON shape, and the synthetic example — is **hardcoded** inside the package at `src/application_pipeline/templates/prompts/` and ships as part of the codebase. The **applicant-specific content** — candidate self-description, in/out-of-domain rules, hard match criteria — lives in `data/synched/user-info/` as three markdown files and is injected into the hardcoded prompt at runtime via a single `{USER_INFO}` slot wrapped in `<user-info>` tags. The **Prompt Loader** concatenates the appropriate files per call site:

- **Relevance Classifier** receives: `self-description.md` + `domain-fit.md`.
- **Match Judge** receives: `self-description.md` + `match-criteria.md`, plus the existing `{skills}` slot fed from `Config.SKILLS`.

In the same change, the pipeline drops per-language prompt files and all language threading: one prompt per call site in German, regardless of listing language. The `langdetect` dependency, `LanguageResolution`, `resolve_language`, the `language` field on **Position Stub**, the `language` argument on `LLMExtractor` methods, and the per-language buffer split in the **Pipeline Orchestrator** are all removed. The hardcoded prompt instructs Claude to respond in German regardless of input language.

## Why

- **Drift caused a silent production failure.** The whole prompt — protocol + applicant content — was authored as a single editable file in the synced folder. The agent-protocol footer (the `<verdicts>`-tag instruction and example) was deleted during user edits, producing `tag_missing: <verdicts> block missing or malformed` on every classify call. Separating the protocol (hardcoded) from the content (user-editable) makes this class of drift structurally impossible: the user can't edit the protocol because they don't have the file.
- **The protocol is generic; the content isn't.** The output-tag contract, JSON shape, and tag-wrapping convention are properties of the **Agent Output Protocol** (ADR-0018), not properties of one applicant's job search. They belong with the code that parses the output. The applicant's self-description and domain rules are deeply personal ("Senior-Tiefe in der Spielebranche", "Bosch Corporate Research") and don't generalise — they belong with the applicant's configuration.
- **Three files, not one, because the content has three concerns.** `self-description.md` is shared between both call sites (the classifier needs to know the candidate is a senior Unity dev to judge "junior ML role" fit; the judge needs the same context). `domain-fit.md` is classify-only. `match-criteria.md` is judge-only. Splitting along consumer boundaries means each call site gets only what it needs without forcing the user to duplicate the candidate profile across files.
- **`SKILLS` stays in `config.py`.** It has two consumers — the `judge_match` `{skills}` slot and the **Domain Pre-Filter** substring whitelist (concatenated with `INCLUSION_KEYWORDS`). It's a flat list of 22 strings already, well-grouped with comments. Extracting it to `skills.md` would force the **Config Loader** to parse markdown bullets back into `list[str]` for the pre-filter — parsing complexity for no editorial gain over a Python list.
- **Per-language separation was costing more than it bought.** Claude is bilingual at parity for the classify task (concept-level binary decision) and for the judge task (the `summary` field's language is fixed by an explicit instruction in the prompt). Maintaining `de` and `en` variants of every prompt and threading a `language` field through **Position Stub** → **Position** → **Domain Pre-Filter** → **LLM Extractor** → batch buffers → transcripts was paying real maintenance cost (and was where the original drift entered: ADR-0006's "parallel edits across the language pair" mitigation broke down in practice). The applicant's traffic is German-dominant; English listings remain classifiable by a German-framed prompt without measurable quality loss.
- **The `<user-info>` tag wrapping makes prompt structure scannable.** A Claude transcript shows clearly which span is system-authored vs user-authored, which helps post-hoc debugging when a classification looks wrong.

## Considered alternatives

- **Keep one editable prompt file per call site (status quo, fix the drift by re-adding the deleted tag block).** Rejected: re-creates the conditions for the next drift incident. The applicant edits the prompts often enough that protocol-drift will recur; structural separation is the durable fix.
- **Startup validation that detects drift.** Considered: add a `Prompt Loader` check that the loaded prompt contains the expected output-tag instruction. Rejected as standalone: catches drift at boot rather than preventing it, and the error message ("your prompt is missing the `<verdicts>` instruction") still puts the user in the protocol-editing business. Implementing it would solve the symptom; the structural split solves the cause.
- **Concatenate user-info as a leading prose block (no `<user-info>` tag wrapping).** Rejected: the wrapping is cheap and makes the boundary explicit in the rendered prompt — easier for the model to localise context, easier for a human reading a transcript.
- **Keep per-language prompts; only restructure the user-info split.** Considered as a smaller increment. Rejected because the user-info split mostly *enables* the language collapse: once the applicant's prose is one language-neutral payload injected into a fixed protocol, maintaining `de` and `en` protocol-prompt variants is pure overhead.
- **Per-call-site user-info files (`user_info.classify.md` and `user_info.judge.md`).** Rejected: forces the candidate self-description to live in two files, re-introducing duplication and drift — exactly the failure mode this ADR is escaping.
- **Single shared user-info file (`user_info.md`) sent to both call sites.** Rejected: the `domain-fit` rules are irrelevant to the **Match Judge** and the `match-criteria` rules are irrelevant to the **Relevance Classifier**. Sending the union to both wastes Haiku tokens on classify and adds prompt noise that the model has to ignore. Three files with a per-call-site concatenation strategy is the cleanest cut.
- **A `langdetect`-based filter (drop non-German listings entirely).** Rejected: would discard the long tail of English-language tech roles posted by Hamburg/Stuttgart multinationals, which are exactly the listings the applicant cares about.

## Consequences

- `Config` gains `USER_INFO_DIR: pathlib.Path` (default `user-info/`); `PROMPTS_DIR` is removed (the prompts are now package-internal, not user-configurable).
- The package ships templates for the three user-info files at `src/application_pipeline/templates/user-info/`, materialised into `data/synched/user-info/` by the existing `python -m application_pipeline init <dir>` command (per ADR-0013).
- `Prompts` dataclass collapses from `dict[Literal["de", "en"], PromptTemplate]` to a single `PromptTemplate` per call site. `CLASSIFY_RELEVANCE_SLOTS` becomes `{ITEMS, USER_INFO}`; `JUDGE_MATCH_SLOTS` becomes `{skills, raw_description, USER_INFO}`.
- The **Prompt Loader** reads the hardcoded prompt files from the package (via `importlib.resources`), reads the user-info files from `Config.USER_INFO_DIR`, concatenates per call site, and renders the `{USER_INFO}` slot with `<user-info>...</user-info>` wrapping. Missing or empty user-info files raise `PromptError` at startup.
- `LLMExtractor.classify_relevance_batch` and `judge_match` lose their `language` parameter. Transcripts no longer carry a `language` field.
- `langdetect` is removed from `pyproject.toml`. `src/application_pipeline/language.py` is deleted. `LanguageResolution`, `resolve_language`, the `language` field on **Position Stub**, and the per-language batching split in the orchestrator are removed.
- The **Run Divider** loses any per-language counters (none currently exposed, but the segmentation hooks come out).
- Existing prompts in `data/synched/prompts/` are deleted as part of migration; the applicant authors fresh `user-info/` content from the seeded templates.
- The hardcoded prompts instruct Claude to "respond in German regardless of input language" so the **Match Verdict**'s `summary` field stays consistent for the **Renderer**.
- Re-introducing language separation later (a third language, or a quality regression on English listings) means re-introducing the file split, the threading, and the buffer split. The risk is real but graded: the change is mechanical and reversible by reverting this ADR's commit set.
- ADR-0006 (per-language prompts via file-naming) is deleted as part of this change — superseded in full.
