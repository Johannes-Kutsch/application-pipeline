# Claude CLI: pinned model and tag-wrapped output

The **LLM Extractor**'s only production implementation (`ClaudeExtractor`) pins `--model` (and `--effort` where applicable) per call site, and treats every `claude -p` response as agent prose with a structured payload wrapped in a semantic XML tag — `<verdicts>` for the **Relevance Classifier**, `<verdict>` for the **Match Judge** — rather than treating the envelope's `result` field as raw JSON. A small project-agnostic **Agent Output Protocol** module extracts the payload via tag-anchored walk-back and regex fence-strip, then `json.loads`. Model and effort live as `ClaudeExtractor` module-level constants — `haiku` for classify (no `--effort`), `sonnet` with `--effort medium` for judge — not as fields on the user-editable `Config`. This ADR tightens ADR-0016's "fresh subprocess per call" line (which left model selection implicit) without invalidating it.

## Why

- **Incident #240: every classify batch since the recent Claude CLI default-behavior change was failing.** The CLI began emitting a Haiku→Sonnet orchestration by default; the Sonnet stage wrapped its JSON in a ` ```json … ``` ` markdown fence; our parser called `json.loads` on the envelope's `result` field and died at char 0 on the leading backtick. The pipeline ran nightly, the **Run Divider** recorded `classify_batches_failed > 0`, and the **Results File** stayed empty. Two independent fixes both happen to be necessary: pinning the model collapses the orchestration to a single deterministic responder; tag-anchored parsing tolerates whatever wrapping the chosen model happens to do.
- **`--model` is a contract, not a hint.** Without `--model`, our call inherits whatever default behavior the CLI ships today. We have no commitment from Anthropic that the default won't change again — and the failure mode when it does is silent (the envelope is well-formed, the `result` field is well-formed, only the inner content is wrong). Passing `--model haiku` for classify and `--model sonnet` for judge makes the call shape deterministic across future CLI updates. Bare aliases (`haiku`, `sonnet`) over dated IDs (`claude-haiku-4-5-20251001`) because point-release renames should be transparent — if our cost or quality story ever depends on the dated id, we revisit then.
- **Per-call-site reasoning load drives the choice.** The **Relevance Classifier** is a yes/no domain decision over 100 items per batch — Haiku is right by capacity, and extended thinking is wasted budget on a binary task, so classify omits `--effort` entirely. The **Match Judge** is open-vocabulary `matched`/`missing` extraction plus a 1-2 sentence summary that weighs seniority, location, and domain fit — Sonnet with `--effort medium` is right. These are call-site decisions, not user preferences. Surfacing them to `Config` would invite the operator to "tune" them in ways that silently change cost and verdict quality.
- **Tag-anchored parsing is the durable defense, fence-stripping is the recovery.** Telling the model "respond with a JSON array" leaves the parse algorithm anchored at the *start* of the response — any preamble, code fence, or commentary at character 0 breaks parsing forever. Telling the model "wrap your output in `<verdicts>` tags" gives both ends a named handle: the parser finds the rightmost `</verdicts>` and walks back through preceding `<verdicts>` openers until one parses. Stray substrings inside the JSON payload (an opening tag mentioned in a job description, agent commentary that includes the tag name) don't derail extraction because the algorithm tries every candidate. Inside each candidate, a regex strips an optional surrounding markdown fence — handling ` ```json\n…\n``` `, bare ` ```\n…\n``` `, trailing whitespace. The fence-strip is a quiet recovery, not a prompt-level instruction: we never tell the model "don't wrap in fences" because prompt-level negative instructions are unreliable across model updates.
- **The Agent Output Protocol module is project-agnostic on purpose.** It exports `extract_json_block(text, tag)` and `AgentOutputProtocolError`. It imports nothing from the project's domain types. Call-site-specific validation — `RelevanceVerdict` length-match against input `ClassifyItem` ids, `MatchVerdict` field shape — stays in `ClaudeExtractor`. This keeps the module a deep, rarely-changing core that any future Claude call site can reuse without pulling in classify/judge specifics. The unrelated RALPH-Loop plugin in `pycastle/` couples its analogous module to its own prompt taxonomy because pycastle is the only client; we have potentially multiple clients (a third LLM call site is not unthinkable) so we keep the boundary clean from day one.
- **Two failure modes, two `envelope_error_class` values.** `tag_missing` and `json_malformed` are operationally distinct — the first signals "model didn't follow output instructions" (prompt-side fix), the second signals "model emitted the tag but produced garbage inside" (model-capability or truncation fix). `result_not_json` from the pre-incident invoker is **removed** — there is no longer a transport-layer JSON parse step. The forensics taxonomy on `ClaudeCliError.envelope_error_class` becomes `{envelope_not_json, envelope_not_object, cli_nonzero_exit, empty_result, tag_missing, json_malformed}` — six values, each pointing at a distinct fix path. The transcripts logs at `synched/logs/{classify_relevance,judge_match}.transcripts.jsonl` continue to be the operator's primary forensic surface.
- **The invoker stays dumb; the extractor owns the parse pipeline.** `ClaudeCliInvoker.call()` does subprocess + envelope-shape validation + exit-code interpretation only, and returns a `ClaudeResponse` whose payload is `raw_response: str` (no `parsed_result` field anymore). `ClaudeExtractor` calls `extract_json_block(response.raw_response, "verdicts" | "verdict")` per call site, then runs its existing shape validation on the returned value. Two reasons: subprocess-talking and content-parsing are independently changeable concerns (a future `--output-format stream-json` migration would only touch the invoker; a future third call site would only touch the extractor and prompts); and the test surface stays sharp — the protocol module is unit-testable against fixture strings without any subprocess plumbing.
- **Prompts gain one worked example per call site, not multiple.** The current **Match Judge** prompt has no shape example at all — it lists fields prosaically and trusts the model to infer the dict shape. That latent fragility is closed by adding one concrete `<verdict>{…}</verdict>` block. The **Relevance Classifier** prompt already had a skeleton with `<id>` placeholders; it gets the same treatment with a fully rendered two-item example. One example, not many, because both call sites have exactly one structural shape. The **Triage Profile** prose at the top of each prompt file is preserved verbatim — only the closing output section changes — to keep the prompt-cache prefix stable after a one-time miss on first run.
- **No four-tag (`<task>`/`<context>`/`<workflow>`/`<output>`) restructure.** Pycastle's ADR-0017 adopts a full structural-tag vocabulary for prompts; this ADR adopts only the *output-anchor* tag wrapping. The four-tag restructure is orthogonal to fixing the incident and would inflate the diff without changing any behavior the operator can observe. Revisit in a future ADR if a third call site lands and the prompt files start to drift in structure.

## Considered alternatives

- **Pin only the model; keep the naive `json.loads(raw_response)` parser.** Rejected: a tracer-bullet probe in the incident showed that even after pinning, occasional fence-wrapped responses still occur (the model fences when it decides the JSON deserves emphasis). Without tag anchoring we trade today's deterministic failure for an intermittent one — strictly worse for forensics.
- **Strip fences; keep the bare-JSON prompt instruction.** Rejected: leaves the parse algorithm anchored at character 0. Any future preamble (a politeness apology, a "Here is the JSON:" header, a chain-of-thought leak) breaks parsing at the boundary again. The tag is the durable anchor; the fence-strip is just a quiet recovery once we're already inside the tag body.
- **Surface model and effort to `Config` so the operator can A/B different choices.** Rejected: invites silent quality and cost regressions, and conflates "search shape" preferences (KEYWORDS, SKILLS, LOCATIONS) with internal call-site engineering choices. The two have different change cadences and different review surfaces — `Config` edits are user-driven and Syncthing-propagated; model choice is a code change with a PR.
- **Embed `extract_json_block` inside `ClaudeCliInvoker` with a `tag` parameter.** Rejected: blurs the invoker's responsibility from "talk to the CLI safely" to "talk to the CLI safely *and* know about our prompt's output convention". The invoker stops being reusable across protocols. Pycastle deliberately drew this same boundary; we follow.
- **Push call-site validation (id/length match, `MatchVerdict` shape) into the protocol module to centralize parsing.** Rejected: forces the project-agnostic module to import `ClassifyItem`, `RelevanceVerdict`, `MatchVerdict` and grow a function per call site. The module stops being a deep, rarely-changing core — every new call site adds a new function with new validation logic. Pycastle gets away with this because it has one client (itself); we want the option of a third call site without re-bending the module.
- **Add a "do not wrap your output in markdown fences" instruction to the prompts.** Rejected: prompt-level negative instructions are unreliable across model updates; multiple anecdata across the team show models occasionally producing the negated behavior more reliably than the requested one. The parser-side fence-strip is the durable defense.
- **Two-slice staging: ship model pinning first as an emergency stabilization, then ship tag-wrapping in a follow-up.** Rejected: the test-suite churn from removing `ClaudeResponse.parsed_result` doesn't split cleanly across PRs, and the emergency slice depends on an unverified bet that Haiku-as-responder never fences. One tracer-bullet slice landing model pinning + protocol module + prompt updates + transcript taxonomy update + test rewrites lands the codebase in a coherent end-state in one merge.
- **Adopt pycastle's four-tag prompt structure (`<task>` / `<context>` / `<workflow>` / `<output>`) as part of the same change.** Rejected for scope: orthogonal to the incident, no observable behavior change, doubles the prompt-file diff for no operational gain. Revisit in a future ADR if and when the prompts diverge.

## Consequences

- **`ClaudeCliInvoker.call()` signature changes.** `model: str` becomes a required keyword argument (no default); `effort: str = ""` is optional with empty-string sentinel meaning "omit the `--effort` flag entirely". Returns `ClaudeResponse(raw_response, usage, cost_usd, duration_s, session_id)` — `parsed_result` is removed. Every call site and every test fixture is updated in one go.
- **`ClaudeExtractor` gains three module-level constants.** `_CLASSIFY_MODEL = "haiku"`, `_JUDGE_MODEL = "sonnet"`, `_JUDGE_EFFORT = "medium"`. Changing them is a code change with a PR; there is no runtime configuration surface.
- **New module: `application_pipeline/llm/agent_output.py`.** Exports `extract_json_block(text: str, tag: str) -> Any` and `AgentOutputProtocolError(kind: Literal["tag_missing", "json_malformed"])`. Tested standalone in `tests/test_agent_output.py` against fixture strings: naked JSON, fence-wrapped JSON, tag-wrapped JSON, preamble-padded JSON, stray-substring inside payload, missing tag entirely, malformed JSON inside tag.
- **Prompt templates updated.** `templates/prompts/classify_relevance.{de,en}.md` close with a `<verdicts>`-tag instruction and one rendered two-item example; `templates/prompts/judge_match.{de,en}.md` close with a `<verdict>`-tag instruction and one rendered example. **Triage Profile** prose preserved verbatim. One-time prompt-cache invalidation on first run after deploy; cache stabilises immediately.
- **Forensics taxonomy revised.** `ClaudeCliError.envelope_error_class` values are now `{envelope_not_json, envelope_not_object, cli_nonzero_exit, empty_result, tag_missing, json_malformed}`. `result_not_json` is retired. Transcripts log shape is otherwise unchanged — `_write_failure_transcript` writes the same JSON-lines record, with `raw_response` included so the operator can see what the model actually emitted on protocol failures.
- **`Pipeline Orchestrator` error handling is untouched.** `AgentOutputProtocolError` is caught at the **LLM Extractor** boundary and re-raised as `ExtractorBatchMalformedError` (classify) or `ExtractorMalformedJSONError` (judge), so the orchestrator's existing "do not mark seen, retry next run" path fires unchanged. **Run Divider** counters (`classify_batches_failed`, `classify_items_abandoned`, `judge_items_abandoned`) keep their existing semantics and emit values from the existing code paths.
- **`ClaudeUsageLimitError` is unaffected.** ADR-0017's degraded-mode behavior is independent of the output-protocol change and continues to fire on the same `result.lower()` / stderr phrase-matching logic in the invoker.
- **Bigger one-time diff than ADR-0016 implies.** The "fresh subprocess per call" line in ADR-0016 implied an argv shape that this ADR updates. ADR-0016 is not formally superseded — its batching, caching, and id-keyed-response contracts all stand — but readers should treat ADR-0018 as the authoritative source for the call shape going forward.
- **No new operator-visible commands or flags.** The CLI invocation, the **Run Divider** field set, and the per-component log file paths are unchanged from the operator's perspective. The only externally visible difference is that classify cron ticks stop failing.
