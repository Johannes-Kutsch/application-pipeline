# Application Pipeline

A personal job-discovery and triage pipeline. Fetches listings from a small set of sources, classifies relevance with a local LLM, scores fit against the applicant's skill list, and emits a single rolling shortlist as markdown. Application authoring (CV/cover letter generation) is explicitly out of scope for v1.

## Scope

- **In scope (v1):** Working **Parsers** for **Bundesagentur**, **stellen.hamburg**, and **jobs-beim-staat**, each provably returning **Position Stubs** and capable of enriching them. Smoke-tested standalone on the laptop. No LLM, no orchestration, no **Results File** in v1.
- **In scope (v1.1):** Full pipeline running on the Pi — orchestrator, **Relevance Classifier**, **Match Judge**, **Deduplication**, rolling **Results File**, cron, Syncthing sync of the **Results File** to the laptop. Ollama lives only on the Pi.
- **Out of scope:** **CV** / **Cover Letter** generation, LaTeX pipeline, **Profile** ingestion, additional commercial parsers (LinkedIn / StepStone / Indeed / Adzuna), Playwright, browser automation.

## Language

### Pipeline artifacts

**Config**: A `config.py` in the Pi's Syncthing-synced folder (`~/application-pipeline/data/config.py`) that controls the search — `KEYWORDS`, `SKILLS`, `NEGATIVE_KEYWORDS`, `SOURCES`, `LOCATIONS`, `INCLUDE_REMOTE`, the `USER_INFO_DIR` path, the **Layout** path override, and the Claude settings (`claude_classify_batch_size: int` default 100 per ADR-0016, and an optional `claude_cli_path` override for non-default CLI installs). Plain Python literals, edited in place, with `#` comments. The Pi's copy is materialised on first bootstrap from a template shipped inside the package at `src/application_pipeline/templates/config.py` via `python -m application_pipeline init <dir>`; per ADR-0013, subsequent edits happen on the laptop's paired copy and Syncthing propagates them. Loaded by the **Config Loader**, which returns a frozen typed `Config` dataclass. `KEYWORDS` is a single global list applied to every **Source** (Cartesian product over keyword × source); **Deduplication** absorbs the overlap. No per-source keyword overrides in v1. Each entry in `SOURCES` is a `SourceEntry` carrying its own `max_results` cap (default 1000) controlling how many newest-first listings that **Source** yields per run. `NEGATIVE_KEYWORDS` drives the **Domain Pre-Filter**; entries validate to length ≥ 3. (Per ADR-0026, `INCLUSION_KEYWORDS` was retired — the schema no longer carries it; the **Config Loader** silently ignores the field if it remains in a legacy `synched/config.py` and emits one INFO log line per run pointing at the ADR.) `USER_INFO_DIR` (default `user-info/`) is the directory containing the **Triage Profile** markdown files injected into the hardcoded prompts (see ADR-0019). _Avoid_: config file (without qualifier), settings file, search config.

**Layout**: A `layout.py` in the Pi's Syncthing-synced folder (`~/application-pipeline/data/layout.py`) sitting alongside the **Config**, owning all user-tunable cosmetic and structural choices for the **Results File**. Seeded from `src/application_pipeline/templates/layout.py` by the same `init` subcommand that materialises the **Config** (ADR-0013). Tier emoji, tier color hex are retained on `Layout` for use *inside* the `CARD_TEMPLATE` (no longer the headline, but available to any user-added section), plus named placeholder groups and the full `CARD_TEMPLATE` (one shared template for all tiers — no `HEADLINE_TEMPLATE` anymore). `FILE_HEADER` is **not** a Layout field — it is hardcoded inside the package (alongside `ResultsFileManager`) as an internal contract carrying the schema-version marker and reset hint, not something the user edits. Plain Python module loaded at runtime by `load_user_module` (the same machinery as **Config**), validated into a frozen typed `Layout` dataclass. The **Renderer** consumes the `Layout` and substitutes placeholders via `str.format_map`. Edited freely by the user — re-design happens here, not in package code. Errors raise `LayoutError`, a subclass of `UserSettingsError` (the shared base with `ConfigError`). _Avoid_: style file, theme file, template file (when the loaded dataclass is meant).

**Results File**: A trio of rolling files under `data/results/` — `green.md`, `amber.md`, `red.md` — one per **Match Tier**, that accumulate **Positions** since the last manual reset. Splitting by tier exists because the green shortlist is what the applicant actually reads first, and a single mixed file buries it under amber/red volume. Reset is **per-tier**: deleting `green.md` resets only green; the next run materialises a fresh one (the shared `FILE_HEADER` from **Layout** is written into whichever file is missing). `.seen.json` is unaffected by file deletion — that's the separate "memory" knob. Synced from Pi to laptop via Syncthing (send-receive) — the laptop is a **read-only consumer by convention**: laptop-side edits will be clobbered by the next Pi append, and `*.sync-conflict-*.md` files indicate the user edited a file in place (treat as user error, log a warning). To annotate or archive, copy the file out of the synced folder first. Each file contains two kinds of block: rendered **Position** blocks (one per kept position whose **Match Tier** equals the file's tier) and **Run Divider** HTML-comment blocks (one per completed run, appended at end-of-run to **all three** files by the **Pipeline Orchestrator** via `Results File Manager.append()`). Appending the divider to all three keeps the heartbeat-is-absence invariant intact per file (any file missing its expected divider signals a crashed run) and lets the operator spot "no green for N runs" at a glance from the divider cadence in `green.md`. Positions are **not numbered** — each Card's H1 (`# {company} · {title} · {location_segment}`) is its own identifier within the file. _Avoid_: dated results file, output file, `current.md` (the old single-file name is gone).

**Run Divider**: An HTML-comment block appended to `current.md` at the end of each completed run by the **Pipeline Orchestrator** — carries timestamp, tag, per-source success counts, kept count, failure count, and timing/cost metrics (`classify_calls`, `classify_items`, `classify_total_s`, `classify_input_tokens`, `classify_output_tokens`, `classify_cache_read_tokens`, `classify_cost_usd`, `judge_calls`, `judge_total_s`, `judge_input_tokens`, `judge_output_tokens`, `judge_cache_read_tokens`, `judge_cost_usd`, `elapsed_s`), and dedup tier counts (`dedup_url_hits`, `dedup_tuple_hits`, `dedup_run_hits`, `dedup_misses`). When the run was degraded by a non-fatal LLM stop (per ADR-0017 — only `usage_limit` in v1), an additional `degraded_reason=<reason>` field appears; non-degraded runs omit the field. Three further conditional fields appear only when non-zero, orthogonal to `degraded_reason` (a run can have abandoned batches without being degraded): `classify_batches_failed=<n>` and `classify_items_abandoned=<n>` (count of classify batches that raised after invoker classification and the underlying item count), and `judge_items_abandoned=<n>` (count of in-domain Positions whose `judge_match` call failed). Abandoned items are not marked seen, so the next cron tick retries them — the counters exist so the operator can grep run history for invisible-data-loss events that the previous "silent batch_error" defect masked. A fourth conditional field, `judge_resumed=<n>` (also omitted when zero), counts stubs whose classify stage was skipped on this run because a prior run had already marked them `classified_in_domain` (per ADR-0021) — the operator-visible proof that the classify-stage idempotence is doing real work. With batched classify (per ADR-0016) `classify_calls` is batch count and `classify_items` is the underlying listing count; `classify_cache_read_tokens` being non-zero is the proof that the prompt-cache TTL is firing across consecutive batches. The Claude metrics split per call site (`classify_*` / `judge_*`) per ADR-0025; the previous aggregate `claude_*` fields are gone. A run that crashes before reaching end-of-run never writes its divider — absence is a signal. _Avoid_: footer (implies single trailing block), status line.

**Failure Report**: A markdown file written to `data/failures/<timestamp>.md` whenever a run fails (cron-wrapper deploy errors or orchestrator runtime errors). Syncs to the laptop via the same Syncthing folder as the **Results File**. Acknowledged by the applicant deleting the file (Syncthing propagates the delete). Per ADR-0012. _Avoid_: incident report, error log (the log file is a separate artifact on the Pi).

**Position**: A single job listing surviving the **Relevance Classifier** and **Deduplication**. Identified within its **Results File** by its rendered H1 (`# {company} · {title} · {location_segment}`), not by a numeric counter. _Avoid_: job, listing, vacancy.

**Position Schema**: The frozen-dataclass shape every **Parser** must return from `enrich()` — `Position(stub: PositionStub, raw_description: str, ...)` with optional `salary: str | None` (free string — German pay grades like `"E13 TV-L"` preserved verbatim), `contract_type: Literal["permanent", "fixed-term", "freelance"] | None`, `employment_type: Literal["full-time", "part-time", "internship"] | None`, `work_model: Literal["remote", "hybrid", "on-site"] | None`, `posted_date: date | None`, `deadline: date | None`. Closed-enum fields are `None` when the source exposes no signal — the **Parser** never guesses. `Position` *contains* its `PositionStub` (composition, not inheritance); the stub's `url, title, company, location, source` reach consumers as `position.stub.<field>` (required: `url, title, source`; `company, location` are `str | None`). Both dataclasses live in `src/application_pipeline/parsers/types.py`; **Domain Pre-Filter**, **Renderer**, **Deduplication**, and **LLM Extractor** import from there (parsers do not import from consumers). `Position` and `PositionStub` are both `frozen=True` and never mutated. The **Match Judge** later attaches a **Match Verdict** (`tier, matched, missing, summary`); these are not parser outputs. _Avoid_: output format, dict structure (the schema is dataclass-typed).

**Raw Description**: The full body text of a **Position** as returned by the **Parser**, normalized to plain Unicode with `\n\n` paragraph breaks (no HTML, no Markdown, no entities). Each **Parser** owns the normalization for its source's body shape — block elements become paragraph boundaries, entities decode, whitespace collapses. Empty after normalization is allowed (`raw_description = ""`); the **Domain Pre-Filter** / **Relevance Classifier** will drop it downstream. Input to the **LLM Extractor**. _Avoid_: description (when full text is meant).

### Filtering & scoring

**Triage Profile**: The applicant's prose self-description plus the rules that decide what counts as in-domain or a good match. Lives in `data/user-info/` as three markdown files — `self-description.md` (candidate background, fed into both LLM call sites), `domain-fit.md` (in/out-of-domain rules, fed into the **Relevance Classifier**), and `match-criteria.md` (hard constraints, location/seniority/remote preferences, fed into the **Match Judge**). Per ADR-0019, the **Prompt Loader** concatenates the relevant files for each call site and the **LLM Extractor** injects the payload into the hardcoded prompt's `{USER_INFO}` slot wrapped in `<user-info>` tags. The user edits these files freely without touching the hardcoded prompt that defines the agent protocol. Language-neutral (German prose; the pipeline is single-language per ADR-0019). Not to be confused with the Phase-2 **CV Profile** used for generating application material. _Avoid_: profile (without qualifier), bio.

**Skill**: A hard-skill or technology item from the applicant's comprehensive list in the **Config**'s `SKILLS`. Rendered as a bullet list into the **Match Judge** prompt's `{skills}` slot at `ClaudeExtractor` construction (one entry per line, `- ` prefix). Per ADR-0026, `SKILLS` is **only** consumed by the judge prompt — the **Domain Pre-Filter** no longer reads it. Stays in `config.py` as a Python list (not extracted to markdown) — the typed list is the right shape for the prompt-slot bind. Not used in any formula — there is no `Skills ∩ Requirements` matcher, and the LLM's `matched`/`missing` lists are open-vocabulary (no closed-vocab validation against `SKILLS`). _Avoid_: keyword (when matching).

**Keyword**: A search term used to query a **Source**, from the **Config**'s `KEYWORDS`. Distinct from a **Skill** — does not affect judgment. Distinct from `NEGATIVE_KEYWORDS`, which drives the **Domain Pre-Filter**. _Avoid_: skill (when querying), negative keyword (when querying).

**Negative Keyword**: An entry in the **Config**'s `NEGATIVE_KEYWORDS`. Patterns (length ≥ 3) that, if found via case-insensitive substring match in a **Position**'s **title** (body text is not consulted, per ADR-0026), cause the **Domain Pre-Filter** to drop the listing. No rescue mechanism — the prefilter is title-only blacklist with no whitelist offset. _Avoid_: blacklist, exclusion.

**Match Verdict**: The structured output of the **Match Judge** for an in-domain **Position** — `{tier: green | amber | red, matched: [...], missing: [...], summary: "2-3 sentences"}`. `matched` and `missing` are short open-vocabulary phrases authored by the LLM (validation is type/length only — no closed-vocabulary check against `SKILLS`). Drives both rendering (Card vs Headline) and Card content. _Avoid_: score, rating.

**Match Tier**: The `tier` field of the **Match Verdict** — green / amber / red — assigned directly by the **Match Judge** LLM, not derived from a numeric formula. Controls **which file** the **Card** is appended to (`green.md` / `amber.md` / `red.md`), not the rendering shape — every kept Position renders as a full **Card**. _Avoid_: score, rating.

**Relevance Classifier**: The conceptual name for the call site that decides whether a **Position** is in the applicant's professional domain (AI / Data / Game Dev / SWE). Implemented as a call to `LLMExtractor.classify_relevance_batch` — there is no wrapper class. Batched and pipelined: the **Pipeline Orchestrator**'s main thread accumulates surviving Pre-Filter outputs into a single buffer (per ADR-0019, the pipeline is single-language so no per-language split) and, the moment the buffer reaches `claude_classify_batch_size` items (default 100, per ADR-0016), hands that batch off to a dedicated classify worker thread (per ADR-0017). Classify runs in parallel with the parsers — there is no "wait for all parsers" barrier. End-of-run, any non-empty residual buffer flushes as a final undersized batch. The classify worker streams in-domain survivors onto a judge queue; a dedicated judge worker thread renders, appends+fsyncs, and marks seen one-by-one. On `ClaudeUsageLimitError` the workers degrade to no-ops for the rest of the run (parsers, dedup, and prefilter continue filling `.seen.json`) and the run ends successfully with `degraded_reason="usage_limit"` recorded on the **Run Divider**. Off-domain items are marked seen immediately. A malformed batch response (bad JSON, length/id mismatch) fails the whole batch — none of its items are marked seen, and the failure is appended to `data/logs/llm_classify_relevance.events.jsonl` with the full prompt + response captured in `data/logs/llm_classify_relevance.transcripts.jsonl` for analysis. Per ADR-0025, `llm_classify_relevance` and `llm_judge_match` are each their own log component (one `.events.jsonl` + one `.transcripts.jsonl` per call site); the implementation class `ClaudeExtractor` is not a log component and there is no `claude_extractor.*` stream. Per ADR-0028 every component identifier carries a layer prefix (`parser_`, `llm_`, `pipeline_`). Runs **only on Positions that survive the Domain Pre-Filter**, so its volume is bounded by the Pre-Filter's drop rate. **Off-domain Positions are discarded — never written to the Results File.** _Avoid_: filter (ambiguous), gate.

**Domain Pre-Filter**: A deterministic module that runs before the **Relevance Classifier**, dropping **Positions** whose **title** matches any **Negative Keyword**. Title-only, blacklist-only — there is no whitelist mechanism and no body-text matching (per ADR-0026, superseding ADR-0005). Substring match (case-insensitive, after `normalize()`) over `title` only. Pure, stateless, pre-LLM, single responsibility. The module enumerates *all* matching keywords (not first-hit short-circuit) on `PreFilterVerdict` so the orchestrator can record a per-position decision into `data/logs/pipeline_prefilter.transcripts.jsonl` and emit a per-keyword end-of-run summary into `pipeline_prefilter.events.jsonl`. Each transcript record carries `url, title, source, passes, reason ∈ {passed, blacklist_drop}, blacklist_matches, title_len` — every prefilter decision is logged, not only drops. Aggregate stats count *positions matched* (not raw occurrences) and surface a `NEGATIVE_KEYWORDS_dead` list of zero-match keywords for config pruning. The Pre-Filter assumes its input has already been keyword-filtered upstream (parsers query their sources with `KEYWORDS`, or apply the keyword client-side per the **Parser** invariant) — the prefilter does not redo that work. _Avoid_: filter (ambiguous), gate, classifier.

**Deduplication**: The process of skipping **Positions** seen in a previous run or already yielded earlier in the current run. Three-tier: an in-run ephemeral URL set owned by a per-run dedup scope (`RunScopedDedup`, obtained from the **Deduplication Store** via its `run_scope()` context manager — scoped to one `run()` invocation, populated synchronously when the underlying store's `is_seen` returns `miss`, sitting in front of the persistent store to absorb the Cartesian `KEYWORDS × LOCATIONS × SOURCES` overlap without paying duplicate `enrich()` cost — counted as `dedup_run_hits` on the **Run Divider**), plus the persistent **Deduplication Store** with two tiers: exact-match on URL, plus exact-match on the `(company, title, location)` tuple lowercased. The tuple tier only fires when **all three fields are non-`None`** on both sides — if any is `None` (the **Parser** found no signal), dedup falls back to URL-only. This trades a few extra duplicates from missing-field listings against the alternative of wrongly aliasing unrelated listings under a weak `(None, "Senior Engineer", None)` tuple. When the tuple matches under a new URL (a syndicated copy), the **Deduplication Store** records an alias entry under the new URL — copying the original record's `status` and `first_seen` — so subsequent runs hit the cheap URL tier without re-paying the tuple lookup. `first_seen` therefore answers "when did this *role* first appear, under which URL", not "when did this URL first appear". `is_seen` returns a 4-variant `SeenResult` (`url_hit | tuple_hit | judge_pending | miss`); `url_hit` and `tuple_hit` drive the orchestrator to push `SKIP` (per ADR-0020, there is no dedup-driven discover early-stop — the orchestrator walks every stub up to the source's `max_results` cap), and the split exists so the **Run Divider** can report `dedup_url_hits` and `dedup_tuple_hits` separately — the operator-visible proof that the alias-write logic (ADR-0004) is doing real work (per ADR-0008). `judge_pending` is returned when the URL's status is `classified_in_domain` (per ADR-0021): the orchestrator enriches the stub and routes it directly to the judge queue, skipping the classify stage — counted as `judge_resumed` on the **Run Divider**. The alias write is performed inside `is_seen`. (A genuinely fuzzy match — Levenshtein, token-set ratio — may replace the second tier later if exact-on-lowercased proves too brittle; v1 keeps it simple.)

**Error semantics for `mark_seen`** (orchestrator behavior, single-writer Pi):
- If **Relevance Classifier** raises a non-quota error (Claude CLI gone, timeout, etc.): **do not mark seen** — next run retries. Optimizes for transient failures over silent loss.
- If **Relevance Classifier** raises `ClaudeUsageLimitError`: the classify and judge workers degrade to no-ops for the rest of the run; nothing in the affected batch is marked seen (per ADR-0017, superseding ADR-0016's abort-the-run policy). Parser-side marks (dedup hits, enrich failures, external redirects, off-domain marks for already-completed batches) continue. The run ends successfully; the **Run Divider** records `degraded_reason="usage_limit"`.
- If **Match Judge** raises a non-quota error after relevance accepted: same as the classifier non-quota case — **do not mark seen**, retry next run.
- For accepted in-domain Positions, the judge worker thread **appends** the rendered block to the **Results File**, **fsyncs** the file, then calls `mark_kept(stub)`. A crash between fsync and mark will replay the listing on the next run (one bounded duplicate in `current.md`). Off-domain Positions get `mark_off_domain(stub)` from the classify worker after the classifier returns false.
There is no automatic retry cap in v1 — a persistently-poisoned listing will re-pay enrich+classify cost on every run until manually skipped. Revisit if it actually happens. State lives in `.seen.json` (synced to the laptop via Syncthing alongside the **Results File** for durability, per ADR-0002; not tracked in git), shaped as `{url: {company_lc, title_lc, location_lc, status, first_seen}}` (any of `company_lc, title_lc, location_lc` may be `None`); the fuzzy index is built in-memory at load, not persisted. Off-domain **Positions** discarded by the **Relevance Classifier** are also marked seen — the gate is "already evaluated", not "already kept" — so re-runs don't re-pay classifier cost on the same noise. **Stubs whose `enrich()` raises `ParserError`** get `mark_enrich_failed(stub)` — the parser already burned its retry budget internally, so re-paying enrich next run would be wasted work. The classify worker calls `mark_classified_in_domain(stub)` for each in-domain verdict *before* enqueuing the judge job (per ADR-0021), so a transient judge-stage failure on the next run resumes from the classify→judge boundary instead of re-paying classify tokens; the judge worker's `mark_kept(stub)` on success overwrites `classified_in_domain` (last-writer-wins). `DeduplicationStore` exposes five narrow methods (`mark_off_domain`, `mark_kept`, `mark_enrich_failed`, `mark_external_redirect`, `mark_classified_in_domain`) rather than a single `mark_seen(status: str)` so each call site is greppable and unambiguous; the on-disk `status` values (`"off_domain" | "kept" | "enrich_failed" | "external_redirect" | "classified_in_domain"`) are unchanged-or-extended for debugging. The pipeline is single-writer at the process level (per ADR-0002) but multi-thread at the file level (per ADR-0017): `.seen.json` is mutated by the main thread, the classify worker, and the judge worker — guarded by a `threading.Lock` inside `DeduplicationStore`. `data/results/current.md` is single-writer per phase — main thread for the file header at startup and the **Run Divider** at end-of-run, judge worker for rendered blocks during the run (phase separation enforced by `join()` barriers, no lock needed). Parser threads remain pure producers — they yield **Position Stubs** into a queue and never touch dedup or the Results File. _Avoid_: duplicate filtering, URL filtering.

### Display

**Card**: The expanded view rendered for **every** kept **Position** regardless of **Match Tier**. Structure: an H1 headline `# {company} · {title} · {location_segment}` (no number, no tier emoji, no color span — the file the Card lives in already encodes the tier); `location_segment` is the literal `location` for on-site or location-only listings (`Hamburg`), `{location} (Hybrid)` / `{location} (Remote)` when a non-on-site `work_model` is known, `Unknown Location` when `location` is `None`, and `Unknown Location (Hybrid|Remote)` for the `None`-location + non-on-site combination. Below the H1: a meta line of `posted_date · contract_type · employment_type` (segments with `None` values omitted; whole line omitted if all three are `None`), a `**Salary:** {salary}` line (omitted when `salary` is `None`), then an `## AI Assessment` section carrying the **Match Verdict**'s 2–3 sentence `summary` followed by `**Matched:**` / `**Missing:**` bulleted lists (each list and its label omitted when empty), then a `## Job Description` section carrying the verbatim `raw_description` (whole section omitted when `raw_description == ""`), then a horizontal rule and a bare-autolink URL footer. Headings are English. The `matched` / `missing` lists come directly from the **Match Judge**; no second LLM call is needed. The previous Headline/Card distinction (single-line for amber/red, full block for green) is gone — splitting the **Results File** by tier solved green-burial without needing a compact form for amber/red.

**Renderer**: A pure module exposing `render(position, verdict, number, layout) -> str`. Always uses the **Layout**'s `CARD_TEMPLATE` (no per-tier template dispatch — tier shows up via the tier-derived `emoji`/`color`/`tier` placeholders, not via template choice). Builds a placeholder dict (raw fields + named placeholder groups + tier-derived `emoji` / `color` / `tier`), substitutes via `str.format_map`, returns the rendered block. No file I/O, deterministic on inputs. _Avoid_: formatter, presenter.

**Results File Manager**: The only module that reads or writes the **Results File** trio (`green.md` / `amber.md` / `red.md` under `data/results/`). One manager instance per tier file; the **Pipeline Orchestrator** holds all three and dispatches by **Match Tier**. Per-instance surface: `ensure_initialized()` (writes the hardcoded package-level `FILE_HEADER` — schema-version marker plus reset hint, no rendered H1 — if file is missing or 0 bytes; `mkdir(parents=True)` for missing parent dirs; idempotent on a non-empty file), `append(rendered_block)` (verbatim write + `flush` + `os.fsync`, propagates `OSError` unwrapped). The **Run Divider** is appended via `append()` to all three managers at end-of-run. Constructed with the path; the **Pipeline Orchestrator** derives paths as `data_dir / "results" / "{tier}.md"` where `data_dir` is the parent of the loaded `config.py` (per ADR-0022) — not Config fields, reset is "move/delete the tier file you want to clear", not "change the path". _Avoid_: writer, output manager.

### Observability

**Log Artifacts**: The set of files written to `data/logs/` during a run, laid out **by reader** rather than by component (per ADR-0024). Four file types: `<comp>.events.jsonl` (one structured row per step emitted by component `<comp>` — envelope `{ts, event, ...fields}` with free-form additional keys, no `component` field since the filename carries it; read by `jq` and analysis scripts; one file per component including parsers, orchestrator, dedup, prefilter, and LLM call sites); `lifecycle.jsonl` (single shared file carrying status-display `registered` / `phase_changed` / `removed` events from every component, each row including a `component` field since the file is shared — for post-mortem timeline reconstruction of a run that hung or skipped a phase); `run.log` (single shared file carrying tracebacks and end-of-run `SUMMARY OF SESSION` blocks from every component, chronologically interleaved, each block preceded by a `=== <component>  <timestamp>  <kind> ===` header where `kind ∈ {summary, traceback}`; read by human eyeballs after a run); `<comp>.transcripts.jsonl` (LLM components only — full prompt/response payloads, kept distinct from `<comp>.events.jsonl` because the payloads are large strings with a different reader). Lifecycle events do not duplicate into the per-component events file; tracebacks and summaries do not duplicate into events either. Per ADR-0028, every component identifier carries a layer prefix — `parser_` (e.g. `parser_bundesagentur_api`, `parser_stellen_hamburg_api`, `parser_jobs_beim_staat_html`, `parser_http`), `llm_` (e.g. `llm_classify_relevance`, `llm_judge_match`), or `pipeline_` (e.g. `pipeline_orchestrator`, `pipeline_dedup`, `pipeline_prefilter`) — so a sorted `ls data/logs/` groups by layer. The prefix is part of the identifier itself: it appears in filenames, the `component` field of `lifecycle.jsonl`, the `=== <component> ===` header in `run.log`, and the **Status Display** row labels. The shared aggregate files `lifecycle.jsonl` and `run.log` remain unprefixed (they are not component files). For LLM code, **the component is the call site** (`llm_classify_relevance`, `llm_judge_match`) — not the implementation class (`ClaudeExtractor`) — so each call site owns its own `.events.jsonl` + `.transcripts.jsonl` pair regardless of which module writes the row, per ADR-0025. The previous convention (one mixed `.log` per component) is superseded. _Avoid_: log file (without qualifier — there are four kinds), debug log.

**Status Display**: A live, in-process view of pipeline progress, sitting alongside the **Run Divider** (end-of-run summary written to the **Results File**), the **Log Artifacts** under `data/logs/`, and the **Failure Report** (crash artifact). Where those three are durable disk artifacts read after the fact, the Status Display is ephemeral — it exists only while `run()` is executing and exists to answer "what is the pipeline doing right now". A `StatusDisplay` Protocol has two implementations: `RichStatusDisplay` (a `rich.Live` table that repaints in place at ~4 Hz, selected when `sys.stdout.isatty()` — laptop dev, parser smoke-tests) and `PlainStatusDisplay` (line-by-line `print` of register / phase-transition / remove events, selected when stdout is not a TTY — Pi cron, captured-to-log invocations). The **Pipeline Orchestrator** picks the implementation in its entrypoint and passes the instance into `run()` as a kwarg. Per ADR-0017 the display is thread-safe (each implementation guards its internal mutations with a `threading.Lock`); the main thread, the classify worker, and the judge worker each update their own rows directly. Parser threads remain pure producers — they never touch the display; the main thread updates parser rows on their behalf. The display surfaces one row per pipeline stage: an overall `pipeline` phase row (top), a transient `startup` phase row (removed once parser threads start), one **Agent Row** per configured **Parser** (keyed by `parser_<parser_type>`), and **Agent Rows** for `pipeline_dedup`, `pipeline_prefilter`, `llm_classify_relevance`, and `llm_judge_match` (per ADR-0028). Row body strings are formatted by **RunMetrics** (the display module is project-agnostic) and follow a `key=value` shape mirroring **Run Divider** vocabulary where applicable; bodies update at event frequency, the 4 Hz Live refresh reads the latest values. `register()` and `update_phase()` also emit a row into the shared `data/logs/lifecycle.jsonl` (per ADR-0024) so the state-transition timeline is captured on disk regardless of display mode — these events do not land in any per-component log. _Avoid_: progress bar (implies known total), TUI (overloaded), dashboard.

### Sources & extraction

**Source**: A configured job board or API in the **Config**'s `SOURCES`, declared as a `SourceEntry` carrying a **Parser Type** and a per-source `max_results` cap. _Avoid_: site, board.

**Parser**: A Python module in `src/application_pipeline/parsers/` that fetches **Positions** from one **Source**. Two-phase API: `discover(query)` returns cheap **Position Stubs** from list-pages; `enrich(stub)` fetches the detail page and returns a full **Position** conforming to the **Position Schema**. The orchestrator runs **Deduplication** between the phases so detail-fetches are paid only on unseen stubs. Each **Parser** is a context manager owning a single `httpx.Client`; no shared mutable module-level state — instances are self-contained so the orchestrator can run one parser per source on its own thread. The **Pipeline Orchestrator** owns the Cartesian expansion of `KEYWORDS × LOCATIONS [+ remote]` and emits one `discover(ParserQuery(keyword, location, max_results))` call per combination per source; `location` is a sealed **Location** (`City(name) | Remote`). Each parser declares its **Location Coverage** as module-level symbols and dispatches via `parsers.location.resolve()` (per ADR-0014). **Source ↔ Parser is one-to-one**: the **Config Loader** rejects duplicate `parser_type` entries in `SOURCES`. **Parsers own all per-host pacing**; the orchestrator never sleeps between queries. Parsers may emit `ExternalRedirect` in addition to `Position` from `enrich` when the wrapper page is an outbound redirect to a third-party board. **Keyword-match invariant** (per ADR-0026): every stub yielded from `discover()` is expected to match its `ParserQuery.keyword` in the title — the **Domain Pre-Filter** assumes this and does no inclusion check of its own. Parsers whose source supports keyword queries on the wire enforce it via the wire query (the v1 case for all three parsers); parsers whose source does not must apply the keyword client-side as a title-substring filter before yielding. This is parser-local authority, analogous to **Location Coverage** (ADR-0014). _Avoid_: scraper, fetcher.

**Position Stub**: The cheap result of `Parser.discover()` — `url, title, company, location, source`. Required: `url, title, source`. Optional (`str | None`): `company, location`. Sources that don't expose one of the optional fields leave it `None`; consumers handle missing data rather than the parser fabricating values. Carries exactly the fields **Deduplication** needs to decide whether to skip. _Avoid_: preview, summary.

**External Redirect**: A payload emitted by a **Parser**'s `enrich()` instead of a `Position` when the wrapper page is an outbound redirect to a third-party board (no iframe, no `raw-url` input, but a `<a href>` pointing outside `jobs-beim-staat.de`). Carries the original `PositionStub` and the `outbound_url` string. Listings whose wrapper page is an outbound redirect to a third-party board are not enriched in v1; the outbound URL is recorded to the per-component debug log for later analysis. The **Pipeline Orchestrator** marks the stub `status: "external_redirect"` in the **Deduplication** store and increments the run-summary counter. _Avoid_: redirect stub, forwarded listing.

**Parser Type**: The string on a `SourceEntry` identifying which **Parser** to use — maps directly to a filename in `parsers/`. _Avoid_: adapter.

**Location**: The geographic slot the **Pipeline Orchestrator** passes to a **Parser** via `ParserQuery.location`. A sealed sum type with two arms: `City(name: str)` for a named place from `Config.LOCATIONS`, and `Remote` for the orchestrator's remote-only slot emitted when `INCLUDE_REMOTE=True`. Replaces the prior `str | None` field. Defined in `src/application_pipeline/parsers/types.py` alongside `ParserQuery`. _Avoid_: place, geo (when the field is meant).

**City**: The `City(name)` arm of **Location**. `name` is the user-typed string from `Config.LOCATIONS`, normalized via `normalize()` before lookup. No central catalog — the set of resolvable city names is the dynamic union of configured **Sources**' `serves()` predicates. _Avoid_: town, place.

**Remote**: The `Remote` arm of **Location**. Emitted by the orchestrator's remote slot. Each **Parser** decides what `Remote` means for its **Source** via `remote_wire()` (e.g. Bundesagentur adds `arbeitszeit=ho`; jobs-beim-staat uses `place=homeoffice`; stellen.hamburg returns empty). _Avoid_: homeoffice, work-from-home (when the type variant is meant).

**Location Coverage**: The module-level **Protocol** every **Parser** module satisfies — `serves(name: str) -> bool` (does the source serve this city?), `to_wire(name: str) -> str` (how to write it on the wire), `serves_remote: bool` (does the source serve remote?), `remote_wire() -> Any` (what to send for `Remote`). Each parser is the local authority on its coverage; there is no global city catalog. The **Config Loader** validates `LOCATIONS` against the union of configured sources' `serves` predicates at load time and rejects unresolvable entries with `ConfigError`. Per ADR-0014. _Avoid_: location map, slug table.

**LLM Extractor**: A Python interface (Protocol) with two methods — `classify_relevance_batch(items: list[ClassifyItem]) -> list[RelevanceVerdict]` and `judge_match(raw_description) -> MatchVerdict`. `ClassifyItem` carries an opaque string `id`, `title`, and `raw_description`; the response is the same id-keyed shape (`[{"id": "...", "in_domain": bool}, ...]`) so dropped/reordered items become loud rather than silent (per ADR-0016). `RelevanceVerdict` is `{in_domain: bool}`. Per ADR-0019, the pipeline is single-language (German); there is no `language` argument and no per-language prompt selection. The hardcoded prompt for each call site lives inside the package (`src/application_pipeline/templates/prompts/`) and owns the agent protocol — task framing, output-tag instructions, JSON shape, example block. The **Triage Profile** is injected at runtime via the prompt's `{USER_INFO}` slot from concatenated `data/user-info/` files (per ADR-0019), keeping the user-tunable content separated from the protocol contract. **Skills** are bound at `ClaudeExtractor` construction (not method-arg) and rendered into the `judge_match` prompt's `{skills}` slot. The v1 production implementation is `ClaudeExtractor`, which shells out to the **Claude Code CLI** (`claude -p - --output-format json --model <alias>` plus `--effort <level>` where applicable) as a fresh subprocess per call (no `--session-id` reuse — system-prompt savings come from Anthropic's prompt-caching TTL, per ADR-0016). Per ADR-0018 (model+effort pinned per call site in `ClaudeExtractor` module-level constants) and ADR-0027 (judge pin updated to `haiku --effort medium` from the prior `sonnet --effort medium`), the call shape is: `haiku` (no `--effort`) for the **Relevance Classifier**, `haiku` with `--effort medium` for the **Match Judge** — and deliberately *not* surfaced to the user-editable `Config`: model choice is a call-site reasoning-load decision, not a search-shape preference. Structured outputs are wrapped in semantic XML tags by the prompt (`<verdicts>` for the **Relevance Classifier**, `<verdict>` for the **Match Judge**) and parsed via the **Agent Output Protocol** rather than `json.loads`-ing the envelope's `result` field directly (per ADR-0018). Pi-only at runtime. _Avoid_: LLM, model (without qualifier).

**Agent Output Protocol**: A project-agnostic pure module under `application_pipeline/llm/` that extracts a structured JSON payload from a Claude Code CLI response whose `result` field carries free-form text with the payload wrapped in a semantic XML tag (per ADR-0018, mirroring the convention used by the unrelated RALPH-Loop plugin in this repo's `pycastle/` directory). Exposes `extract_json_block(text: str, tag: str) -> Any` plus `AgentOutputProtocolError(kind: "tag_missing" | "json_malformed")`. Internally: finds the rightmost `</tag>`, walks backward through preceding `<tag>` openings (so agent commentary, preamble, or stray tag-shaped substrings inside the JSON cannot derail extraction), strips an optional surrounding markdown fence inside the candidate body with a regex (handles ` ```json\n…\n``` `, ` ```\n…\n``` `, trailing whitespace), then `json.loads`. Returns the first candidate that parses; raises `AgentOutputProtocolError(kind="tag_missing")` if no tag is present at all, `kind="json_malformed"` if every candidate failed to parse. The **LLM Extractor** catches the error at its call-site boundary, writes a forensics line into the per-call transcript (`tag_missing` or `json_malformed` as `envelope_error_class`), and re-raises as the existing `ExtractorBatchMalformedError` / `ExtractorMalformedJSONError`. Reusable for any future Claude call site without adding call-site coupling to the module. _Avoid_: output parser (too generic), response handler (overloaded with HTTP).

**Match Judge**: The conceptual name for the call site that produces a **Match Verdict** for an in-domain **Position**. Implemented as a direct call to `LLMExtractor.judge_match` — there is no wrapper class. Replaces the formula-based Position Matcher. Runs only on **Positions** that passed the **Relevance Classifier**. Returns a **Match Verdict**. Before the `raw_description` is substituted into the judge prompt, German boilerplate paragraphs (benefits sections, company-marketing prose, application-instructions) are stripped at the call site via a sentinel list maintained inside the **LLM Extractor**'s `ClaudeExtractor` implementation.

**Pagination**: Successive page fetches per **Source**, preferably ordered newest-first by posted_date, until either the **Source** signals no next page (empty page, no next-link, or API end-of-results) **or** the per-Source `max_results` cap is reached. Per ADR-0020 there is no dedup-driven discover early-stop: the orchestrator walks every stub a parser yields and lets **Deduplication** absorb the overlap; `SKIP_AND_END_QUERY` (per ADR-0009) is only emitted on `run_state.is_aborted` for fast drain on fatal abort. Newest-first ordering is a soft preference rather than a binding correctness invariant — it matters because the per-Source `max_results` cap should intersect the genuinely-new listings on first run / post-`.seen.json`-reset, but out-of-order yields no longer risk silent truncation on later runs. Each `SourceEntry` in the **Config** carries its own cap (default 1000). The cap bounds first-run cost; on subsequent runs **Deduplication** keeps `enrich()` cost small naturally. Sources that don't expose a posted_date sort key approximate via the source's default ordering (typically reverse-chronological).

### Invocation

The full pipeline runs as a single Python entry point on the Pi: `python -m application_pipeline` (package per `pyproject.toml`, entry via `src/application_pipeline/__main__.py`). On the Pi, cron invokes `~/application-pipeline/current/.venv/bin/python -m application_pipeline` wrapped by `flock -n /var/lock/application-pipeline.lock` (single-writer enforcement) and by the deploy wrapper that handles tag-pull + staging-clone + symlink flip (per ADR-0011). There is no `/search` Claude Code skill — that idea was retired. The laptop is used only during development to smoke-test individual **Parsers** in isolation (no LLM, no **Deduplication**, no **Results File**). Ollama, the **Relevance Classifier**, the **Match Judge**, **Deduplication**, and **Results File** rendering only ever run on the Pi. Note: the `pycastle/` directory in this repo is an unrelated RALPH Loop coding-agent plugin used to *build* this project; it is not this project's Python package.

## Relationships

- The **Results File** is a trio (`green.md`, `amber.md`, `red.md`), each containing zero or more **Positions** of its tier with per-file unique numbers. Files are rolling and reset per-tier; **Deduplication** state survives the reset of any subset.
- A **Position** belongs to exactly one **Source**, was fetched by exactly one **Parser**, and was enriched by the **LLM Extractor**.
- A **Position** is written only if it passes the **Domain Pre-Filter** *and* the **Relevance Classifier**. The **Match Tier** in its **Match Verdict** controls which of the three **Results File**s the **Card** is appended to, not inclusion or rendering shape.
- The **Domain Pre-Filter** runs before the **Relevance Classifier** and never bypasses the LLM gate; it only drops listings.
- The **Match Judge** runs after **Relevance Classifier** acceptance, takes `raw_description` plus construction-bound **Skills** and the injected **Triage Profile**, and returns a **Match Verdict**. There is no string-intersection or percentage formula.
- The **Triage Profile** lives in `data/user-info/` and reaches the LLM via the hardcoded prompt's `{USER_INFO}` slot (per ADR-0019); **Skills** are bound at `ClaudeExtractor` construction and reach the LLM via the `{skills}` slot; `NEGATIVE_KEYWORDS` lives in the **Config** and reaches the **Domain Pre-Filter** directly (per ADR-0026 the prefilter no longer consumes `SKILLS` or any inclusion list).

## Flagged ambiguities

- **"Filter"** — used to mean three different things: the **Domain Pre-Filter** (deterministic drop), the **Relevance Classifier** (LLM-based discard), and the **Match Threshold** (collapses to **Headline**). These are distinct stages with distinct effects; never use the bare word "filter".
- **"Match Score"** — formerly a percentage formula. Removed. The **Match Judge** LLM emits **Match Tier** directly. See [ADR-0003](docs/adr/0003-match-tier-judged-by-llm.md).
- **"Profile"** — disambiguate. The v1 **Triage Profile** (prose, lives in `data/user-info/`, injected into the prompts' `{USER_INFO}` slot, drives LLM judgment) is distinct from the Phase-2 **CV Profile** (structured, used to generate application material). Never use "profile" without a qualifier.
- **"Summary" vs "Raw Description"** — the 2–3 sentence summary in the **Card** is part of the **Match Verdict**, written by the **Match Judge**. `raw_description` is the full original text. Never use "description" when the full text is meant.
- **"Source" vs "Parser"** — a **Source** is a config entry (what to search); a **Parser** is the module (how). One **Parser** can serve multiple **Sources** if their structure matches.
