# Template tree splits into routing buckets; `.claude/skills/` is package-shipped and seeded by `init`

The `src/application_pipeline/templates/` tree reorganises into per-destination *buckets*. Each top-level subdirectory under `templates/` declares the host root it seeds into:

- `templates/application-pipeline/...` → seeds to `<cwd>/application-pipeline/` (the existing settings dir per ADR-0029).
- `templates/claude/skills/...` → seeds to `<cwd>/.claude/skills/`.

Everything currently under `templates/` (except the non-seeded `prompts/`) moves under `templates/application-pipeline/`. The agent-skills tree at repo-root `.claude/skills/` moves into `templates/claude/skills/`, becoming a package-shipped, refreshable artefact rather than a hand-maintained repo-root directory.

`init --refresh` semantics for shipped subtrees: known files overwrite unconditionally; the helper never deletes files it doesn't own, so user-added skills in their own sibling directories survive. Renamed or retired files become zombies the operator removes by hand — the same contract `setup/` already has.

Ride-along: `cv_skeleton.tex` moves out of `<settings-dir>/skills/cv_skeleton.tex` to `<settings-dir>/cv-template/cv_skeleton.tex`, eliminating the conceptual collision between the LaTeX-skeleton `skills/` dir and the agent-skills `.claude/skills/` dir now landing on the host. `/write-cv` reads the new path; `init --refresh` deletes the obsolete `<settings-dir>/skills/` directory if it still exists.

## Why

- **`.claude/skills/` belongs to the package, not the host.** The skills (`analyse-listing`, `iterate-cv`, `write-cv`, `_shared/*`) encode pipeline-specific workflows that evolve with the package — file layouts referenced (per ADR-0043: `domain-fit.md` retiring), prompts consumed, slot-maps assembled. Treating them as hand-maintained host content guarantees drift the moment package shape changes; treating them as package-shipped + refreshable matches the cron model that already keeps `setup/*.sh` current.
- **One init invocation, two roots, declared by structure.** Adding a second destination root (the host's `.claude/`) outside `application-pipeline/` could have been encoded as a side flag, a second subcommand, or a hardcoded list. Encoding it as the *first path segment under `templates/`* makes the routing self-documenting: anyone adding a new template file picks the destination by choosing the bucket dir. No registry, no decision point in code.
- **Avoiding the duplicate-file failure mode.** The alternative ("keep `.claude/skills/` at repo root for dev convenience, copy into `templates/` at package time") would mean two sources of truth on disk. Skills only used at runtime (no test coverage) drift silently when one copy is edited. Single source eliminates the entire class of bug.
- **Dogfood path for the dev machine.** With templates as source of truth, this very repo's `.claude/skills/` becomes a generated artefact resurrected by `init --refresh`. Editing a skill means editing `templates/claude/skills/...`, then running `init --refresh` (or letting cron do it on the Pi). The dev/host distinction disappears for skill content.
- **Refresh contract stays minimal.** "Overwrite known files, never delete" is the same contract `setup/*.sh` has had since ADR-0011's refresh amendment. Extending it to `.claude/skills/` adds no new mental model. Cleanup of retired files is manual, parallel to `domain-fit.md` cleanup per ADR-0043 — the package signals what's current, the operator removes what's not.
- **`cv_skeleton.tex` rename is forced by the dual-`skills/` collision.** Before this ADR there was one `skills/` dir on the host (LaTeX skeleton). With `.claude/skills/` arriving on the same machine, two unrelated `skills/` dirs at different roots would coexist. Renaming the LaTeX one to `cv-template/` removes the ambiguity without affecting any consumer outside `/write-cv` and `init`.

## Considered alternatives

- **Keep `.claude/skills/` at repo root as the source of truth; build-time copy into `templates/`.** Rejected: two on-disk copies of the same files, divergence the moment one is edited, and a build step that runs only when someone remembers it. The "convenient dev workflow" claim doesn't survive contact with how often a forgotten copy step ships a stale skill.
- **Symlink repo-root `.claude/skills/` → `templates/claude/skills/`.** Rejected: dev box is Windows 11 (per `Get-CimInstance Win32_OperatingSystem`); NTFS symlinks require elevated permissions or developer-mode toggles, and `git` handling of symlinks across Windows/Linux contributors is brittle.
- **Add a `--skills` flag to `init` that opts into seeding `.claude/skills/`.** Rejected: a flag with one sensible value (always-on under cron) is friction without payoff. The bucket layout achieves the same routing declaratively.
- **Ship `.claude/` wholesale (including `settings.json`, hooks, agents).** Rejected for this change: those are host-local Claude-Code configuration the operator owns. Issue #488 scopes to skills specifically; expanding the surface here mixes pipeline-owned content with host-owned configuration. Future ADRs can extend the bucket if a concrete need emerges.
- **`init --refresh` deletes-then-rewrites the package-owned `.claude/skills/` subtree** (so renamed/retired files vanish automatically). Rejected during grilling: clean for files inside dirs the package fully owns, but the contract becomes "any file inside a package-named dir is owned by the package", which silently nukes user notes or work-in-progress files dropped inside e.g. `iterate-cv/`. The accumulated-zombies cost is lower than the surprise-delete cost.
- **Ship a manifest (`.application-pipeline-manifest`) listing every file the package owns**, so refresh can delete files removed between versions. Rejected: machinery overhead for a single-author personal pipeline. Manual cleanup of a handful of retired files per ADR cycle is cheap.
- **Defer the `cv_skeleton.tex` rename to a separate change.** Rejected: the collision between LaTeX-`skills/` and agent-`.claude/skills/` is introduced *by* this change, and `/write-cv` is being edited anyway for the ADR-0043 file-layout migration. Bundling avoids a second migration window where users see two `skills/` dirs and have to guess which is which.
- **Auto-delete legacy `domain-fit.md` from `<settings-dir>/user-info/triage-profile/` on `init --refresh`.** Rejected: that file contains user-authored bullets the operator must merge into `match-criteria.md` by hand (per ADR-0043). Auto-delete would destroy work. The pipeline's `PromptError` at run time is the single migration signal; `init` does not validate user-content schemas.

## Consequences

- **`templates/` layout reorganises** into `templates/application-pipeline/...` (carrying everything previously under `templates/` except `prompts/`) and `templates/claude/skills/...` (carrying the four skills `_shared/`, `analyse-listing/`, `iterate-cv/`, `write-cv/`). `templates/prompts/` stays in place — non-seeded, package-internal.
- **`init_cmd.py` seeds per-bucket.** The seeder walks each top-level subdir under `templates/` and dispatches to the matching host root (`application-pipeline/` → `<cwd>/application-pipeline/`, `claude/skills/` → `<cwd>/.claude/skills/`). `home_dir()` becomes one of several roots, not the sole target. The `_EXCLUDE_DIRS = {"prompts"}` guard moves up a level (skip top-level subdirs not in the bucket map).
- **Repo-root `.claude/skills/` is deleted from version control.** The dev machine re-acquires it by running `application-pipeline init --refresh` on the project itself (or by hand, copying from `templates/claude/skills/`). `.claude/settings.local.json` and any non-shipped `.claude/` content stays untouched.
- **`init --refresh` on the host:** overwrites every file under `<cwd>/.claude/skills/_shared/`, `analyse-listing/`, `iterate-cv/`, `write-cv/` with the templates-bundled version. Files in other `<cwd>/.claude/skills/<custom-skill>/` dirs are left alone. Files inside package-owned skill dirs that aren't in the templates (operator-added notes, drafts) are also left alone (overwrite-known-only); the operator removes them by hand if a skill rename leaves them orphaned.
- **`cv_skeleton.tex` host path changes** from `<cwd>/application-pipeline/skills/cv_skeleton.tex` to `<cwd>/application-pipeline/cv-template/cv_skeleton.tex`. `/write-cv` reads the new path. `init --refresh` deletes the obsolete `<cwd>/application-pipeline/skills/` directory if present (it only ever held `cv_skeleton.tex`).
- **Skill content migrates for ADR-0043 simultaneously.** `_shared/STARTUP-TRIAGE.md` (existence check), `_shared/TRIAGE-ROUTING.md` (routing table), `analyse-listing/SKILL.md` (file list), `iterate-cv/SKILL.md` (file list + routing references) all drop references to `domain-fit.md` and route to `match-criteria.md` instead. Mechanical edits, no behavioural change beyond what ADR-0043 already mandates.
- **CONTEXT.md updates**: **CV Skeleton** entry (new path, new template-source path), **Invocation** section (bucket layout, two-root `init` seeding, `.claude/skills/` as package-owned).
- **Tracking**: issue #488 (the seeding/refresh mechanics) and issue #535's skill-side fan-out (the ADR-0043 file references).

## Supersedes / amends

- **Amends ADR-0011** (`init` semantics): the refresh contract ("overwrite `setup/*.sh`; preserve user-authored content") extends to a second package-owned subtree (`.claude/skills/` under each shipped skill dir). The "single home dir" assumption of ADR-0029 holds for `application-pipeline/` content; `.claude/skills/` is a second root with its own write rules but the same seed-if-missing / refresh-overwrite-known cadence.
- **Amends ADR-0029** (home dir is `<cwd>/application-pipeline/`): `init` now writes to two roots under `<cwd>/`. The `application-pipeline/` root retains its hardcoded behaviour and remains the home of `config.py` and `user-info/`; `.claude/skills/` is the second, narrower root with no user-authored content inside the directories the package owns.
- **Amends ADR-0030** (`cv_skeleton.tex` lives at `<settings-dir>/skills/cv_skeleton.tex`): destination relocates to `<settings-dir>/cv-template/cv_skeleton.tex`. The skeleton's role (format-by-example consumed by `/write-cv`, refreshable by `init --refresh`) is unchanged.
- **Amends ADR-0042** in passing: `init --refresh`'s "delete-on-refresh" list grows from `{layout.py}` to `{layout.py, skills/cv_skeleton.tex}` (the latter as part of the directory cleanup after rename, not because the skeleton itself is retired).
